| 维度 | 内容 |
|------|------|
| **功能描述** | 在每个工作流阶段的关键决策点，预设 AI 常见的"偷懒借口"和对每个借口的反驳表。防止 agent 跳过关键步骤 |

| **示例** | Analyst 的澄清能力：Rationalization "需求已经够清楚了" → Reality "你觉得清楚是因为你在脑补，用户没说的部分才是最危险的" |

## 六、agent-skills 独有的、值得特别关注的设计模式

> 以下不是单个功能点，而是设计模式层面的创新，对我们框架有整体性影响。

### 6.1 "Process, not prose" 设计哲学

**来源：** agent-skills / docs/skill-anatomy.md
**地址：** https://github.com/addyosmani/agent-skills/blob/main/docs/skill-anatomy.md

**核心思想：** 角色定义不是"参考文档"而是"可执行流程"。每个能力点必须有步骤、检查点、退出条件。不能写"确保代码质量好"，要写"运行 npm test 并验证所有测试通过"。

**对我们的影响：** 我们的角色定义文件（analyst.md、architect.md 等）应该从描述性文档转向过程性文档。每个能力点都写成步骤 + 验证清单。

### 6.2 Verification is non-negotiable

**来源：** agent-skills / README.md
**地址：** https://github.com/addyosmani/agent-skills

**核心思想：** 每个角色/能力的末尾必须有验证清单。"看起来对"永远不够。验证需要证据：测试输出、构建结果、运行时数据。

**对我们的影响：** 在每个角色能力定义中加入 Verification 部分，列出必须提供的证据。

### 6.3 Parallel Fan-out + Merge 模式

**来源：** agent-skills / references/orchestration-patterns.md
**地址：** https://github.com/addyosmani/agent-skills/blob/main/references/orchestration-patterns.md

**核心思想：** Review 阶段可以并行启动多个专项审查者（code quality + security + test），然后在一个合并步骤中综合结果。前提：子任务真正独立、产出不同类型的发现。

**对我们的影响：** Full 流程的 Review 阶段可以引入这个模式。对应 agent-skills 的 /ship 命令设计。

### 6.4 "用户/斜杠命令是编排者，Persona 不调用 Persona"

**来源：** agent-skills / AGENTS.md + references/orchestration-patterns.md
**地址：** https://github.com/addyosmani/agent-skills

**核心思想：** 角色之间不能互相调用。如果 code-reviewer 发现安全问题，它在报告中推荐启动 security-auditor，而不是自己调用。编排权属于用户或工作流定义。

**对我们的影响：** 这与我们的"对话式工作流"设计高度一致——人是对话的中心节点，角色之间不直接交互。

## 八、对我们 v0.3 框架的即时改进建议

基于 agent-skills 的分析，以下是对现有 v0.3.1 框架可以**立即吸收**的改进：

### 8.1 角色定义文件应从"描述性"改为"过程性"

当前角色定义可能偏向"这个角色是谁、有什么能力"的描述。应该改为"触发条件 → 步骤流程 → 验证清单"的过程性结构。

**来源模式：** agent-skills 的 SKILL.md anatomy（Overview → When to Use → Process → Rationalizations → Red Flags → Verification）

### 8.2 每个角色能力加入 Anti-Rationalization 表格

不是泛泛地说"要严格"，而是**枚举 AI 在这个能力点上最可能的偷懒路径，预设反驳**。

### 8.3 每个角色能力加入 Verification 清单

能力定义的末尾必须是可验证的证据要求。"看起来对"不算通过。

### 8.4 Reviewer 角色加入 Doubt-Driven 子能力

Full 流程的 Review 阶段默认使用对抗性审查，而不是温和的"检查一下"。

### 8.5 Developer 角色加入 Source-Driven 子能力

编码前获取官方文档、标注来源、不确定的标记 UNVERIFIED。
