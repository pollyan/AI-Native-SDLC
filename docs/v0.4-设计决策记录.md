## 一、产品定位

### 1.1 我们是谁

一家 B2B AI Coding 工具公司，产品包括开发 Agent、测试 Agent、需求 Agent、运维 Agent 等。本工作流框架是为开发 Agent 设计的差异化方法论。

### 1.2 框架定位

| 维度 | 定义 |
|------|------|
| **短期** | 作为方法论资产，由 FDE 团队在客户现场实施赋能时使用 |
| **中期** | 集成到 AI Coding 工具中，成为内置的差异化功能 |
| **长期** | 扩展到完整产研全流程（战略→投注→需求→设计→开发→测试→上线→运维） |

### 1.3 用户定义

- **Developer**：客户企业的一线开发人员，框架的主要使用者。在所有阶段中是决策者和操作主体。
- **FDE**：现场开发工程师，短期内作为框架实施的替身用户，负责客户赋能培训。

### 1.4 交互界面

以 IDE 插件为主（也支持自研 IDE 和 CLI）。工作流阶段的切换在同一个对话框中完成，通过加载不同的角色定义切换 AI 行为。

## 二、核心设计原则

### 原则 1：人主控，AI 辅助

- 人控制所有阶段切换
- AI 可以在门控条件明确满足时主动建议进入下一阶段，但人最终确认
- 所有 AI 产出（修复建议、方案推荐等）由人决定是否采纳
- **这是与竞品（如 Devin "AI 自动干、人旁观"模式）的核心差异化**

### 原则 2：Spec 文件作为统一传递载体

- 每个阶段的产出都是一份 Spec 文件
- Spec 既是下一阶段的输入（上下文传递），也是本阶段的质量约束
- Spec 模板 = 方法论的具象化，内置最佳实践
- 通过模板约束内容结构，保证稳定的输出质量

### 原则 3：统一语言作为基础设施

- 项目的领域词汇表（Context）被所有阶段的 AI 行为依赖
- 术语在需求评审阶段开始建立，后续阶段持续维护
- 术语冲突时实时拦截，发现新术语时立即写入

## 三、工作流阶段定义（MVP）

### 3.1 全景 vs MVP

```
全景：战略 → 投注 → 需求 → 技术设计 → 开发 → 测试 → 上线 → 运维
                          │                              │
                          └────── MVP 覆盖这段 ──────────┘
```

战略和投注有计划覆盖，但不在 MVP 范围。框架设计需预留扩展位置。

### 3.2 四个阶段

#### 阶段 1：Requirement Review（需求评审）

| 维度 | 定义 |
|------|------|
| **输入** | 需求文档（PRD/Issue/口头描述，形态不限） |
| **预处理** | 根据输入成熟度走不同路径：完整 PRD → 直接评审；Issue → 轻量补充；口头描述 → 完整结构化 |
| **核心动作** | 结合业务上下文和代码上下文，发现模糊地带、提出精确问题、澄清边界场景 |
| **关键产出** | 验收标准（作为后续 Review 的校验基准） |
| **输出** | 需求 Spec（包含澄清后的需求描述 + 验收标准 + 边界定义 + 术语定义） |
| **门控条件** | 问题定义清晰 ✓ 范围界定 ✓ 验收标准已列出 ✓ 关键术语已定义 ✓ 用户确认 ✓ |

**模糊地带的四种类型：**

| 类型 | 说明 | 需要的信息源 |
|------|------|------------|
| 信息缺失 | 需求压根没提某个边界场景 | 需求文档 + Context |
| 术语歧义 | 用了模糊的词或一词多义 | 需求文档 + Context |
| 逻辑矛盾 | 需求跟现有代码逻辑冲突 | 需求文档 + 代码库 |
| 隐含假设 | 没写但默认了某个前提 | 需求文档 + 业务上下文 |

> ⏳ 待确认：Requirement Review 阶段 AI 能否访问代码库？（问题 15 未回答）

#### 阶段 2：Plan（方案设计）

| 维度 | 定义 |
|------|------|
| **输入** | 需求 Spec |
| **核心动作** | 技术方案设计 + 任务拆分 |
| **任务拆分粒度** | 可独立测试的垂直切片（端到端，贯穿所有集成层） |
| **输出** | 方案 Spec（包含技术方案 + 垂直切片列表） |

#### 阶段 3：Execute（执行）

| 维度 | 定义 |
|------|------|
| **输入** | 方案 Spec（垂直切片列表） |
| **执行模式** | 逐切片串行 |
| **推荐方式** | TDD（测试→实现→重构），始终为推荐实践，从不强制 |
| **输出** | 可运行的代码 |

#### 阶段 4：Review（评审）

| 维度 | 定义 |
|------|------|
| **审查维度** | ① 代码质量 ② 需求一致性 ③ 验收标准逐条确认 |
| **产出形式** | 问题清单 + 自动修复建议 |
| **决策权** | 开发者决定是否采纳修复建议 |

## 四、流程分层

### 4.1 三种流程

| 流程 | 阶段组合 | 适用场景 | 预估时长 |
|------|---------|---------|---------|
| **Quick** | Execute | 改文案、修小 bug、配置调整 | < 1 小时 |
| **Standard** | Plan → Execute | 常规功能开发，需求清晰 | 半天 ~ 2 天 |
| **Full** | Requirement Review → Plan → Execute → Review | 新功能模块、跨团队协作、架构变更 | 跨天 |

### 4.2 流程推荐机制

- **推荐时机**：需求进入系统时
- **判断依据**：需求文档特征 + 项目代码上下文 + 开发者确认
- **默认策略**：信息不足时宁重勿轻（推荐 Standard 或 Full）
- **用户始终可以**：在任何阶段改变路径、跳过阶段、回退阶段
- **长期演进方向**：积累历史使用数据后优化推荐准确度

## 六、方案预研决策（Plan 阶段）

### 6.1 决策：方案预研从 Requirement Review 移至 Plan

**理由：** Requirement Review 的定位是搞清楚"要什么"，不是决定"怎么做"。方案对比是 Plan 阶段的核心工作。

### 6.2 设计原则

1. **不做成必填项** — 标注为"当方案间存在重大 tradeoff 时填写"，避免流程负担
2. **重点不是列优劣** — 而是回答三个核心问题：
   - 各方案分别接受了什么代价？
   - 选择 A 意味着放弃了 B 的什么？
   - 什么条件下这个选择需要被推翻？
3. **不是独立阶段** — 是一种应该随时可以触发的工程实践，当复杂度、不确定性或影响范围达到阈值时显性化

### 6.3 在 Plan Spec 中的位置

```markdown
## 方案预研（可选：当方案间存在重大 tradeoff 时填写）
### Option A: [方案名]
- 接受的代价: ...
- 放弃了什么: ...
- 推翻条件: ...
### Option B: [方案名]
- 接受的代价: ...
- 放弃了什么: ...
- 推翻条件: ...
> 选择理由: [为什么选 A，基于什么假设]
> 推翻信号: [什么情况下需要重新考虑]
```

### 6.4 Plan 阶段门控条件

**两层门控机制（竞品共性：先自动检查，再人工确认）：**

**第一层：Spec 自审核（AI 自动检查）**

| # | 检查项 | 来源竞品 | 理由 |
|---|--------|---------|------|
| A1 | 技术方案与 Requirement Spec 一致 | Superpowers cross-ref | 防止方案跑偏 |
| A2 | 每个切片是端到端垂直切片，可独立测试 | Matt Pocock tracer bullet | 核心设计原则 |
| A3 | 切片依赖关系正确，无循环依赖 | GSD dependencies | 串行执行的前提 |
| A4 | 接口合约已定义 | GSD Interface-First | 防止切片间接口模糊 |
| A5 | 无 placeholder/TBD 残留 | Superpowers No Placeholders | 防止“看起来完成实际没完成” |
| A6 | 每个 Requirement Spec 的功能需求都有切片覆盖 | SpecKit Spec Coverage | 防止漏需求 |

**第二层：人工确认**

Developer 检查自审核结果，对方案做最终判断，确认后进入 Execute。

**风险识别：** Plan Spec 模板里保留风险识别字段（可选）。Standard 流程不强制作门控条件，但 Developer 可以根据实际情况决定是否填写。

## 6.5 Execute 阶段切片完成标准

**设计原则：** TDD 是推荐实践，从不强制。客户没有测试基础就是没有，不强求。

**切片最小完成标准：**

| # | 条件 | 说明 |
|---|------|------|
| D1 | 切片涉及的功能可运行 | 不是“写完了”，是“跑起来了” |
| D2 | 相关测试通过（如果有的话） | 有测试就必须通过；没有测试不阻断，在 Review 时会标记建议补充 |
| D3 | 无 placeholder/TBD 残留 | 同 Plan 门控，防止假完成 |

**推荐实践（不强制）：**

| # | 条件 | 说明 |
|---|------|------|
| R1 | TDD 循环完成（测试→实现→重构） | 推荐但非门控 |
| R2 | 代码已提交（有意义的 commit） | 可追溯 |
| R3 | 切片未引入已知回归 | 至少跑一下相关测试套件 |

## 6.6 Review 阶段报告结构与问题分级

### 三轴审查（借鉴 Matt Pocock 两轴 + 我们的差异化）

| 轴 | 检查什么 | 来源 |
|---|---------|------|
| **A. 代码质量** | 命名、结构、错误处理、安全、性能 | Matt Pocock Standards 轴 |
| **B. 需求一致性** | 实现是否与 Requirement Spec 一致 | Matt Pocock Spec 轴 + Superpowers |
| **C. 验收标准确认** | 逐条检查 Requirement Review 阶段定义的验收标准 | 我们的差异化，竞品都没有 |

### 三级问题分级（行业共识）

| 级别 | 定义 | 处理要求 |
|------|------|---------|
| **Critical** | 功能缺陷、数据安全、上线阻断 | 当前切片必须修复后才能继续 |
| **Important** | 质量隐患、设计不合理、测试不足 | 建议修复，Developer 决定是否采纳 |
| **Minor** | 风格、命名、微小改进 | 记录即可，不阻断 |

### 问题标准格式

每个问题必须包含：
- 级别（Critical/Important/Minor）
- 问题标题
- 位置：file:line
- 为什么重要
- 建议修复（自动生成，Developer 决定是否采纳）

### 不做的事

- **不做反馈处理协议**——"人主控"模式下，Developer 自己读报告自己判断，不需要"怎么处理 review 反馈"的协议

## 6.7 Context 文件管理机制

### 谁维护

**AI 写入，人确认。** AI 负责发现术语冲突、建议新增术语、维护文件格式；人负责确认定义是否正确。

### 何时更新：四个触发点

| # | 触发点 | 动作 | 阶段 |
|---|--------|------|------|
| **T1** | Requirement Review 中首次确定领域术语 | 创建/追加术语定义 | Requirement Review |
| **T2** | 后续阶段发现术语冲突或歧义 | 拦截并提示澄清，更新定义 | 任意阶段 |
| **T3** | 新概念出现（之前没涉及的领域） | 追加术语，标记来源阶段 | 任意阶段 |
| **T4** | 术语含义发生变更（需求演化） | 更新定义，标记变更历史 | 任意阶段 |

### 核心原则（借鉴 Matt Pocock）

- **懒创建**：不需要预先手动建表，在第一次确定术语时自动生成
- **实时维护**：术语被确定时立即写入，不批量处理
- **单文件存储**：，便于 AI 一次性加载

### 不做的事

- **MVP 阶段不引入 ADR 机制**——决策记录已在设计文档中维护，框架成熟后再考虑

## 6.8 Spec 模板结构

### 设计原则

每个阶段的 Spec 都是下一阶段的输入 + 本阶段的质量约束。模板内置最佳实践，通过结构约束保证稳定输出质量。

### Requirement Review Spec 模板

```markdown
# Requirement Review Spec: [功能名]

## 需求来源
- 输入类型: [完整 PRD / Issue / 口头描述]
- 来源链接: ...

## 需求描述
[澄清后的完整需求描述]

## 术语定义
[本需求涉及的新术语，同步写入 CONTEXT.md]

## 验收标准
- [ ] AC-001: [可验证的验收条件]
- [ ] AC-002: ...
> 注：验收标准将作为 Review 阶段 C 轴的校验基准

## 边界定义
### 包含
- ...
### 不包含
- ...

## 澄清记录
| # | 问题 | 回答 | 来源 |
|---|------|------|------|
| Q1 | ... | ... | AI 提问 / Developer 补充 |

## 自审核检查
- [ ] 问题定义清晰
- [ ] 范围已界定
- [ ] 验收标准已列出
- [ ] 关键术语已定义
```

## 6.9 角色定义（AI 在不同阶段的行为规范）

| 阶段 | AI 角色 | 核心行为 | 不做的事 |
|------|---------|---------|---------|
| **Requirement Review** | 需求评审助手 | 提出精确问题、发现模糊地带、建议验收标准 | 不做方案设计、不做技术选型 |
| **Plan** | 方案设计助手 | 生成技术方案、拆分垂直切片、识别风险、方案预研对比 | 不做需求质疑（已在上一阶段完成） |
| **Execute** | 编码助手 | 逐切片实现、推荐 TDD 实践、运行验证 | 不擅自修改切片范围、不跳过切片 |
| **Review** | 评审助手 | 三轴审查、问题分级、生成修复建议 | 不自动修复（Developer 决定） |

## 6.10 门控机制汇总

**统一原则：每个阶段的门控 = 自审核（AI 检查）+ 人工确认（Developer 拍板）**

| 阶段 | 自审核内容 | 人工确认 |
|------|-----------|---------|
| **Requirement Review** | 问题定义清晰 ✓ 范围界定 ✓ 验收标准 ✓ 术语定义 ✓ 无 TBD ✓ | Developer 确认 |
| **Plan** | 方案一致 ✓ 切片端到端 ✓ 依赖正确 ✓ 接口定义 ✓ 无 TBD ✓ 需求覆盖 ✓ | Developer 确认 |
| **Execute** | 切片可运行 ✓ 测试通过（如有）✓ 无 TBD ✓ | 无显式阶段门控，切片逐个完成 |
| **Review** | 三轴报告完整 ✓ 问题已分级 ✓ 验收标准逐条确认 ✓ | Developer 确认处理方式 |

**设计决策：Execute 阶段没有阶段级门控。** 切片是逐个完成的，每个切片有自己的完成标准（D1-D3）。Review 本身就是 Execute 之后的质量检查。
