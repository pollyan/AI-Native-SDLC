# AI-Native SDLC v0.4 — 规范驱动开发框架设计

> **版本：** v0.4
> **日期：** 2026-05-17
> **状态：** 设计完成，待评审
> **变更说明：** 从 v0.3 的"多智能体专家团"演化为"规范驱动开发框架"，以 Spec 文件为统一传递载体、以统一语言为基础设施、以垂直切片驱动执行。

---

## 一、概述

### 1.1 什么是 AI-Native SDLC

AI-Native SDLC（AI 原生软件开发生命周期）是一套面向企业开发团队的**规范驱动开发方法论**。它以结构化的 Spec 文件作为阶段间信息传递的统一载体，通过四阶段工作流（Requirement Review → Plan → Execute → Review）驱动 AI 辅助开发全过程，同时确保人始终是决策主体。

核心理念可以概括为三句话：

- **Spec 文件**是阶段间信息传递的唯一载体——结构化、可审查、可版本控制
- **统一语言**是跨阶段信息无损传递的基础设施——术语一致，沟通才有意义
- **垂直切片**是执行阶段的基本单元——端到端、可独立验证、逐步交付

### 1.2 核心理念

| 理念 | 含义 |
|------|------|
| **人主控，AI 辅助** | 人控制所有阶段切换和关键决策；AI 可以在门控条件满足时主动建议，但人最终确认 |
| **Spec 文件作为统一传递载体** | 每个阶段的产出都是一份结构化的 Spec 文件，既是下一阶段的输入，也是本阶段的质量约束 |
| **统一语言作为基础设施** | 项目的领域词汇表（Context）被所有阶段的 AI 行为依赖，术语冲突时实时拦截 |
| **垂直切片驱动执行** | Plan 阶段将方案拆分为端到端的垂直切片，Execute 阶段逐切片串行执行 |

## 二、产品定位

### 2.1 目标用户

| 角色 | 定义 | 说明 |
|------|------|------|
| **Developer** | 客户企业的一线开发人员 | 框架的主要使用者。在所有阶段中是决策者和操作主体 |
| **FDE** | 现场开发工程师 | 短期内作为框架实施的替身用户，负责客户赋能培训 |

### 2.2 使用场景

以 IDE 插件为主要交互界面（也支持自研 IDE 和 CLI）。工作流阶段的切换在同一个对话框中完成，通过加载不同的角色定义切换 AI 行为。

典型场景：Developer 在 IDE 中收到一个新功能需求，按照工作流从 Requirement Review 开始，逐步经过 Plan → Execute → Review，每个阶段的产出是一份 Spec 文件，最终交付可验证的代码。

### 2.3 交付形态

| 阶段 | 形态 | 说明 |
|------|------|------|
| **短期** | 方法论资产 | 由 FDE 团队在客户现场实施赋能时使用 |
| **中期** | 产品集成 | 集成到 AI Coding 工具中，成为内置的差异化功能 |
| **长期** | 全流程覆盖 | 扩展到完整产研全流程（战略→投注→需求→设计→开发→测试→上线→运维） |

## 三、核心设计原则

### 原则 1：人主控，AI 辅助

- 人控制所有阶段切换
- AI 可以在门控条件明确满足时主动建议进入下一阶段，但人最终确认
- 所有 AI 产出（修复建议、方案推荐等）由人决定是否采纳
- **这是与竞品（如 Devin "AI 自动干、人旁观"模式）的核心差异化**

### 原则 2：Spec 文件作为统一传递载体

- 每个阶段的产出都是一份 Spec 文件
- Spec 既是下一阶段的输入（上下文传递），也是本阶段的质量约束
- Spec 模板 = 方法论的具象化，内置最佳实践
- 通过模板约束内容结构，保证稳定的输出质量

### 原则 2.5：流程优于文档（借鉴 agent-skills）

- Spec 模板应是可执行流程，而非参考文档
- 每个能力点必须有步骤、检查点、退出条件
- 模板的价值在于"填完就知道下一步做什么"，而非"写完再想这是什么意思"

### 原则 3：统一语言作为基础设施

- 项目的领域词汇表（Context）被所有阶段的 AI 行为依赖
- 术语在 Requirement Review 阶段开始建立，后续阶段持续维护
- 术语冲突时实时拦截，发现新术语时立即写入
- **对人：统一语言是锦上添花；对 AI：统一语言是信息无损传递的前提条件**

### 原则 4：垂直切片驱动执行

- Plan 阶段将方案拆分为端到端的垂直切片（不是单个层的水平切片）
- 每个切片贯穿所有集成层（schema + API + UI + tests），可独立测试
- Execute 阶段逐切片串行执行，推荐 TDD 模式（测试→实现→重构）
- **禁止"先写完所有测试再写所有实现"的水平切片反模式**

## 四、工作流设计

### 4.1 流程分层

全景视角下，框架覆盖的范围：

```
全景：战略 → 投注 → 需求 → 技术设计 → 开发 → 测试 → 上线 → 运维
                          │                              │
                          └────── MVP 覆盖这段 ──────────┘
```

战略和投注有计划覆盖，但不在 MVP 范围。框架设计已预留扩展位置。

两种流程模式适配不同复杂度的任务：

| 流程 | 阶段组合 | 适用场景 |
|------|---------|---------|
| **Quick** | Execute | 小修复、配置调整、需求极其明确且简单的改动 |
| **Standard** | Requirement Review → Plan → Execute → Review | 所有正常开发任务，完整流程不省略 |

**Quick 模式进入条件（借鉴 Superpowers Scope Check）：**

- 需求只涉及 1 个垂直切片
- 不涉及跨模块改动
- Developer 明确确认走 Quick

**设计理由：** 去除中间档——三档分层过度设计。企业场景要求严格，Standard 不省略任何环节。

**流程推荐机制：**

- **推荐时机**：需求进入系统时
- **判断依据**：需求文档特征 + 项目代码上下文 + Developer 确认
- **默认策略**：信息不足时走 Standard
- **Developer 始终可以**：在任何阶段改变路径
- **回退机制**：允许从后续阶段回退到前序阶段（如 Execute 回退到 Plan 修正切片拆分，Plan 回退到 Requirement Review 补充需求）。回退时保留已完成的产出，不丢弃已完成的工作

### 4.2 四个阶段详解

#### 阶段 1：Requirement Review（需求评审）

| 维度 | 定义 |
|------|------|
| **输入** | 需求文档（PRD/Issue/口头描述，形态不限） |
| **预处理** | 根据输入成熟度走不同路径：完整 PRD → 直接评审；Issue → 轻量补充；口头描述 → 完整结构化 |
| **核心动作** | 结合业务上下文和代码上下文，发现模糊地带、提出精确问题、澄清边界场景。AI 可访问代码库，用于交叉验证需求与现有代码的逻辑一致性 |
| **关键产出** | 验收标准（作为后续 Review 阶段的校验基准） |
| **输出** | 需求 Spec（包含澄清后的需求描述 + 验收标准 + 边界定义 + 术语定义） |
| **AI 行为** | 角色定位：**需求评审助手**。提出精确问题、发现模糊地带、建议验收标准、问题重构——不按用户说的原样理解，而是重新理解“真正要解决什么”（借鉴 gstack） |
| **AI 不做的事** | 不做方案设计、不做技术选型（方案设计在 Plan 阶段进行） |

**模糊地带的四种类型：**

> **追问方法（借鉴 gstack Six Forcing Questions）：** 从六个维度对需求进行追问——Who/What/When/Where/Why/How，确保每个模糊地带都被充分暴露。

| 类型 | 说明 | 需要的信息源 |
|------|------|------------|
| 信息缺失 | 需求压根没提某个边界场景 | 需求文档 + Context |
| 术语歧义 | 用了模糊的词或一词多义 | 需求文档 + Context |
| 逻辑矛盾 | 需求跟现有代码逻辑冲突 | 需求文档 + 代码库 |
| 隐含假设 | 没写但默认了某个前提 | 需求文档 + 业务上下文 |

#### 阶段 2：Plan（方案设计）

| 维度 | 定义 |
|------|------|
| **输入** | 需求 Spec |
| **核心动作** | 技术方案设计 + 垂直切片拆分。当方案涉及不熟悉的技术栈时，先做领域研究（借鉴 GSD RESEARCH.md 机制），研究结果被后续阶段引用 |
| **任务拆分粒度** | 可独立测试的垂直切片（端到端，贯穿所有集成层） |
| **输出** | 方案 Spec（包含技术方案 + 垂直切片列表） |
| **AI 行为** | 角色定位：**方案设计助手**。生成技术方案、拆分垂直切片、识别风险、方案预研对比 |
| **AI 不做的事** | 不做需求质疑（已在上一阶段完成） |

**方案预研**：当方案间存在重大 tradeoff 时，Plan Spec 中包含方案对比，重点回答三个问题：
1. 各方案分别接受了什么代价？
2. 选择 A 意味着放弃了 B 的什么？
3. 什么条件下这个选择需要被推翻？

#### 阶段 3：Execute（执行）

| 维度 | 定义 |
|------|------|
| **输入** | 方案 Spec（垂直切片列表） |
| **执行模式** | 逐切片串行（默认）。根据切片复杂度可选：Subagent-Driven（逐切片分发到独立上下文）vs Inline Execution（批量执行），复杂切片适合前者（借鉴 Superpowers） |
| **推荐方式** | TDD（测试→实现→重构），始终为推荐实践，从不强制。先建信号再做分析——没有可重复的 pass/fail 信号，任何分析都是猜测（借鉴 Matt Pocock 反馈循环优先理念） |
| **输出** | 可运行的代码 |
| **AI 行为** | 角色定位：**编码助手**。逐切片实现、推荐 TDD 实践、运行验证 |
| **AI 不做的事** | 不擅自修改切片范围、不跳过切片。调试时生成 3-5 个可证伪假设，而非锚定在第一个猜测上（借鉴 Matt Pocock） |

**Execute 阶段不产出独立 Spec 文件**，产出是代码，追踪通过 Plan Spec 中的切片列表 checkbox 勾选实现。

#### 阶段 4：Review（评审）

| 维度 | 定义 |
|------|------|
| **审查维度** | ① 代码质量 ② 需求一致性 ③ 验收标准逐条确认 |
| **产出形式** | 问题清单 + 自动修复建议 |
| **决策权** | Developer 决定是否采纳修复建议 |
| **AI 行为** | 角色定位：**评审助手**。三轴审查、问题分级、生成修复建议 |
| **AI 不做的事** | 不自动修复（Developer 决定）、不定义反馈处理协议——人主控模式下，Developer 自己读报告自己判断 |

**三轴审查（借鉴 Matt Pocock 两轴 + 我们的差异化）：**

> **反锚定设计（借鉴 Matt Pocock）：** 三轴独立并行审查，防止互相污染——代码质量审查不受需求一致性判断影响，反之亦然。

| 轴 | 检查什么 | 来源 |
|---|---------|------|
| **A. 代码质量** | 命名、结构、错误处理、安全、性能 | Matt Pocock Standards 轴 |
| **B. 需求一致性** | 实现是否与 Requirement Spec 一致 | Matt Pocock Spec 轴 + Superpowers |
| **C. 验收标准确认** | 逐条检查 Requirement Review 阶段定义的验收标准 | **我们的差异化，竞品都没有** |

### 4.3 阶段转换与门控机制

**统一原则：每个阶段的门控 = 自审核（AI 检查）+ 人工确认（Developer 拍板）**

#### Requirement Review 门控

| # | 自审核检查项 | 来源 |
|---|------------|------|
| 1 | 问题定义清晰 | 设计决策 |
| 2 | 范围界定 | 设计决策 |
| 3 | 验收标准已列出 | 设计决策 |
| 4 | 关键术语已定义 | 设计决策 |
| 5 | 无 placeholder/TBD 残留 | Superpowers |

人工确认：Developer 确认以上内容。

#### Plan 门控

| # | 自审核检查项 | 来源 |
|---|------------|------|
| A1 | 技术方案与 Requirement Spec 一致 | Superpowers cross-ref |
| A2 | 每个切片是端到端垂直切片，可独立测试 | Matt Pocock tracer bullet |
| A3 | 切片依赖关系正确，无循环依赖 | GSD dependencies |
| A4 | 接口合约已定义 | GSD Interface-First |
| A5 | 无 placeholder/TBD 残留 | Superpowers |
| A6 | 每个 Requirement Spec 的功能需求都有切片覆盖 | SpecKit Spec Coverage |

人工确认：Developer 检查自审核结果，对方案做最终判断。

#### Execute 门控

Execute 阶段没有阶段级门控。切片是逐个完成的，每个切片有自己的完成标准：

**切片最小完成标准：**

| # | 条件 | 说明 |
|---|------|------|
| D1 | 切片涉及的功能可运行 | 不是"写完了"，是"跑起来了" |
| D2 | 相关测试通过（如果有的话） | 有测试就必须通过；没有测试不阻断，Review 时标记建议补充 |
| D3 | 无 placeholder/TBD 残留 | 同 Plan 门控，防止假完成 |

**推荐实践（不强制）：**

| # | 条件 | 说明 |
|---|------|------|
| R1 | TDD 循环完成（测试→实现→重构） | 推荐但非门控 |
| R2 | 代码已提交（有意义的 commit） | 可追溯 |
| R3 | 切片未引入已知回归 | 至少跑一下相关测试套件 |

#### Review 门控

| # | 检查项 |
|---|--------|
| 1 | 三轴报告完整 |
| 2 | 所有问题已分级（Critical/Important/Minor） |
| 3 | 验收标准逐条确认 |
| 4 | Developer 确认处理方式 |

## 五、Spec 模板

### 5.1 Requirement Review Spec

```markdown
# Requirement Review Spec: [功能名]

## 需求来源
- 输入类型: [完整 PRD / Issue / 口头描述]
- 来源链接: ...

## 需求描述
[澄清后的完整需求描述]

## 术语定义
[本需求涉及的新术语，同步写入 CONTEXT.md]

## 验收标准
- [ ] AC-001: [可验证的验收条件]
- [ ] AC-002: ...
> 注：验收标准将作为 Review 阶段 C 轴的校验基准

## 边界定义
### 包含
- ...
### 不包含
- ...

## 澄清记录
| # | 问题 | 回答 | 来源 |
|---|------|------|------|
| Q1 | ... | ... | AI 提问 / Developer 补充 |

## 自审核检查
- [ ] 问题定义清晰
- [ ] 范围已界定
- [ ] 验收标准已列出
- [ ] 关键术语已定义
- [ ] 无 placeholder/TBD 残留

## Developer 确认
- [ ] 已阅读并确认以上内容
```

**字段说明：**

| 字段 | 必填 | 说明 |
|------|------|------|
| 需求来源 | 是 | 记录输入类型和链接，便于回溯 |
| 需求描述 | 是 | 澄清后的完整描述，不是原始需求的复制粘贴 |
| 术语定义 | Full 必填 | 新发现的领域术语，同步写入 Context 文件 |
| 验收标准 | 是 | 可验证的验收条件，作为 Review 阶段的校验基准 |
| 边界定义 | 是 | 明确包含和不包含的范围 |
| 澄清记录 | 是 | 记录评审过程中提出的问题和回答，便于审计 |
| 自审核检查 | 是 | AI 自动检查，确保 Spec 完整性 |
| Developer 确认 | 是 | 人工确认，人主控原则的体现 |


**字段说明：**

| 字段 | 必填 | 说明 |
|------|------|------|
| 技术方案 | 是 | 核心方案描述，清晰说明"怎么做" |
| 技术上下文 | 是 | 技术栈和环境信息，便于 Execute 阶段准确编码 |

| 自审核检查 | 是 | 6 项自动检查，确保方案质量和完整性 |
| Developer 确认 | 是 | 人工确认 |

**垂直切片的设计原则（详见 §三 原则 4 和 §4.2 阶段 3）：**

- 每个切片是端到端的垂直切片（schema + API + UI + tests），不是单个层的水平切片
- 切片可独立测试、可独立验证
- 依赖关系显式声明，无循环依赖
- **禁止"先写所有 schema → 再写所有 API → 再写所有 UI"的水平切片**

### 5.3 Review Report

```markdown
# Review Report: [功能名]

> 关联 Requirement Spec: [链接]
> 关联 Plan Spec: [链接]

## 审查总结
- 审查切片数: ...
- 发现问题数: Critical [N] / Important [N] / Minor [N]
- 综合评估: [通过 / 需修复后通过 / 需重大返工]

## A 轴：代码质量
### Strengths
- ...
### Issues
[按 Critical / Important / Minor 分组]

## B 轴：需求一致性
### 与 Requirement Spec 对照
- ...

## C 轴：验收标准逐条确认
| # | 验收标准 | 状态 | 备注 |
|---|---------|------|------|
| AC-001 | ... | ✅ 通过 / ❌ 未通过 | ... |

## 推荐修复项（自动生成，Developer 决定是否采纳）
- [ ] [修复项 1]
- [ ] [修复项 2]

## Developer 确认
- [ ] 已阅读审查报告，决定如下：...
```

**字段说明：**

| 字段 | 必填 | 说明 |
|------|------|------|
| 审查总结 | 是 | 审查范围和问题统计，给出综合评估 |
| A 轴：代码质量 | 是 | 命名、结构、错误处理、安全、性能方面的检查 |
| B 轴：需求一致性 | 是 | 实现与 Requirement Spec 的一致性对照 |
| C 轴：验收标准确认 | 是 | 逐条确认 Requirement Review 阶段定义的验收标准 |
| 推荐修复项 | 是 | 自动生成的修复建议，Developer 决定是否采纳 |
| Developer 确认 | 是 | 人工确认处理方式 |

**问题分级标准：**

| 级别 | 定义 | 处理要求 |
|------|------|---------|
| **Critical** | 功能缺陷、数据安全、上线阻断 | 当前切片必须修复后才能继续 |
| **Important** | 质量隐患、设计不合理、测试不足 | 建议修复，Developer 决定是否采纳 |
| **Minor** | 风格、命名、微小改进 | 记录即可，不阻断 |

**问题标准格式**（每个问题必须包含）：

```
- 级别（Critical/Important/Minor）
- 问题标题
- 位置：file:line
- 为什么重要
- 建议修复（自动生成，Developer 决定是否采纳）
```

## 六、Context 管理

### 6.1 术语表机制

项目的领域词汇表（Context 文件）是跨阶段的**基础设施层**，所有阶段的 AI 行为必须遵守 Context 中的术语定义。

**为什么统一语言对 AI 是基础设施：**

人可以模糊沟通——说"把这个订单处理一下"，对方能推断是 Order 还是 Fulfillment。但 AI 不具备这种隐性推理能力。如果 Requirement Review 阶段说"购买请求"，Plan 阶段说 "Purchase"，Execute 阶段用 `Transaction`，Review 阶段说"订单"——四个阶段之间的信息传递会逐步积累偏差。

### 6.2 维护触发点

**维护原则：AI 写入，人确认。**

| # | 触发点 | 动作 | 阶段 |
|---|--------|------|------|
| **T1** | Requirement Review 中首次确定领域术语 | 创建/追加术语定义 | Requirement Review |
| **T2** | 后续阶段发现术语冲突或歧义 | 拦截并提示澄清，更新定义 | 任意阶段 |
| **T3** | 新概念出现（之前没涉及的领域） | 追加术语，标记来源阶段 | 任意阶段 |
| **T4** | 术语含义发生变更（需求演化） | 更新定义，标记变更历史 | 任意阶段 |

**核心原则（借鉴 Matt Pocock）：**

- **懒创建**：不需要预先手动建表，在第一次确定术语时自动生成
- **实时维护**：术语被确定时立即写入，不批量处理
- **单文件存储**：便于 AI 一次性加载

### 6.3 存储形式

Context 文件存储在项目根目录的 `CONTEXT.md`，格式如下：

```markdown
# [项目名] Context

[一两句话说明这是什么项目]

## Language

**Order**:
客户提交的购买请求，包含一个或多个 LineItem
_Avoid_: Purchase, transaction, 购买请求

**Invoice**:
发货后发送给客户的付款请求
_Avoid_: Bill, payment request, 账单

## Relationships

- 一个 **Order** 产生一个或多个 **Invoice**
- 一个 **Invoice** 属于恰好一个 **Customer**

## Flagged ambiguities

- "account" 被同时用来指 Customer 和 User — 已解决：这是两个不同概念
```

**格式规则：**

- 每个术语定义**一句话**，定义它"是什么"，不定义它"做什么"
- 同义词标记为 `_Avoid_`，明确"应该避免的别名"
- `Relationships` 用加粗术语名 + 基数表达
- `Flagged ambiguities` 记录历史上出现过的术语冲突及解决结果
- 只包含项目特有的领域术语，不含通用编程概念

## 七、AI 行为边界

**所有角色的共同约束：**

1. 使用 Context 文件中的统一术语
2. 每个阶段产出后执行自审核检查
3. 不跳过门控条件
4. 遇到不确定的内容使用 `[NEEDS CLARIFICATION]` 标记，不猜测
5. 不问"是否继续"——Developer 要求执行就执行到底

**物理实现：** 所有角色都是 `.md` 文件定义的身份，在同一个 IDE 对话进程中被先后加载。不存在独立进程、独立上下文或"换人"的问题——从头到尾就是同一个对话，只是在不同阶段加载不同的 `.md` 来切换行为。

## 八、与竞品的对比

### 8.0 差异化定位（原 §一.3）

当前 AI 编程工具/框架市场存在两类极端：

- **Devin / Qoder 模式**：AI 自动干、人旁观——AI 自主执行，人类只在必要时介入
- **传统编码工具**：人全干、AI 打下手——AI 只是代码补全和问答

AI-Native SDLC 选择的是**中间路线**：

> 人是决策主体，AI 是专业助手。每个阶段由人控制和确认，AI 提供专业能力支持，但最终方案选择、质量判断、阶段切换全部由人决定。

这个定位与 Superpowers（obra/superpowers，183K stars）的理念最为接近，但我们在以下方面做了关键升级：

| 维度 | Superpowers | AI-Native SDLC v0.4 |
|------|-------------|---------------------|
| 验收标准 | 无显式追踪 | Requirement Review 阶段定义验收标准，Review 阶段逐条确认 |
| 统一语言 | 无内建机制 | Context 文件作为基础设施，四阶段持续维护 |
| 执行粒度 | Task（可水平切片） | 垂直切片强制——端到端、可独立验证 |
| Review 维度 | 两轴（代码质量 + Spec 合规） | 三轴（代码质量 + 需求一致性 + 验收标准逐条确认） |
