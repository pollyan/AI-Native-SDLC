## 一、背景与问题

### 1.1 当前 AI 工具的共同瓶颈

无论是 AI 编码工具（Cursor、Claude Code）、AI 协作平台（CodeBanana）、还是提示词框架（BMAD），在复杂任务面前都面临同一个问题：

**单 Agent 承担过多职责 → 上下文膨胀 → 可靠性下降 → 人工兜底成本高。**

具体表现：
- 长任务中 Agent "失忆"——做到后面忘了前面
- 一个人盯 Agent 比自己写还累
- 角色混淆——该追问的时候在执行，该执行的时候在发散
- 知识无法沉淀——换个人来，Agent 又从零开始

### 1.2 行业趋势

AI 编程工具正在从单 Agent 模式转向多智能体协作模式。阿里 Qoder 的专家团（Experts Mode）、开源框架 BMAD 的多角色体系，都在验证同一个方向：**让多个专精的 Agent 协作，而不是让一个全能 Agent 硬扛。**

### 1.3 我们的机会

现有方案各有缺陷：
- **BMAD**：设计理念好，但学习成本高、单 LLM 模拟（非真正多 Agent）、无持久记忆
- **Qoder**：真正多 Agent，但"Agent 执行、人旁观"的定位有风险，且绑定在特定 IDE 内
- **CodeBanana**：三位一体架构有野心，但 Agent 过于复杂、无法并发

我们需要一个**吸收各方优点、规避已知缺陷**的方案。

## 二、核心理念

**一套与项目/工作区稳定绑定的多角色专家智能体系统，以需求流转为主线，让 AI 作为辅助角色参与全流程，人始终是执行主体和决策者。**

三个关键定位：
1. **多角色分工**，而不是一个 Agent 全栈
2. **辅助定位**，而不是替代人
3. **需求流转驱动**，而不是工具堆砌

### 2.1 核心场景：对话式工作流

本系统的核心使用场景是**人在 IDE 对话框中、以连贯上下文驱动 AI 专家协作**。

这意味着：
- 人是中心节点，不是旁观者。对话本身承担了反馈、澄清、纠偏的功能
- 每个 AI 专家角色在人的对话框中"出场"，协助完成当前阶段的任务后"退场"
- 阶段性产出通过文件持久化，但**信息流转不依赖文件**——对话上下文是第一信息源，文件是确认快照
- 人随时可以在对话框中指出问题、跳过环节、修改方向，不需要通过特殊的接口或机制来"通知"系统

> 这与传统 pipeline 模式的根本区别：pipeline 是"agent 之间通过文件自动流转"，而我们是"人主导、agent 辅助的对话式协作"。文件的角色是持久化阶段性产出，不是 agent 之间的通信协议。

## 三、架构设计

### 3.1 角色体系（v0.2 精简版）

v0.2 采用**最小角色集**策略：先用最少的角色验证协作机制是否跑得通，再扩展。

系统由 4 个核心角色组成：

| 角色 | 核心职责 | 典型产出 |
|------|---------|---------|
| **Facilitator** | 入口路由：理解意图、追问澄清、推荐角色和流转路径 | 流转建议、角色推荐 |
| **Architect** | 方案设计：技术选型、架构决策、接口定义 | 架构方案、技术选型文档 |
| **Developer** | 代码实现：编码、Bug 修复、技术优化 | 代码、变更记录 |
| **Reviewer** | 质量把关：代码评审 + 测试验证 + 安全审查 | Review 意见、测试报告 |

> **设计理由**：v0.1 列了 8 个专家角色 + Facilitator + Orchestrator = 10 个角色。对第一版验证来说太多了。先验证"多角色协作比单 Agent 好"这个核心假设，再验证"角色分工是否足够细"。需求分析、产品经理、运维、技术文档等角色，先由 Facilitator 兼任或作为后续扩展。

#### 角色扩展机制

角色集是**可插拔**的。在 `.agents/config.yaml` 中定义激活哪些角色：

```yaml
roles:
  active:
    - facilitator
    - architect
    - developer
    - reviewer
  # 后续可扩展：
  # - product-manager
  # - ops-engineer
  # - tech-writer
```

#### 关于 Facilitator 的定位

v0.2 中 Facilitator 的定位是**轻量入口**，不是独立的智能调度系统：

- **主要功能**：帮助用户理清"该做什么、按什么顺序"，尤其在用户不确定时提供指引
- **不是必须经过的关卡**：用户已知目标角色时可直接调用
- **兼任简化角色**：在 4 角色模式下，需求澄清等轻量工作由 Facilitator 承担

> **与 v0.1 的关键变化**：v0.1 把 Facilitator 定义为"零学习成本"的入口，但这在实践中有个风险——路由质量取决于 Facilitator 对项目上下文的理解。v0.2 的策略是降低对 Facilitator 的期望：它不是"智能路由器"，而是"引导助手"。真正复杂的需求澄清和路由判断，通过对话中人的参与来补全。

### 3.2 信息流转方式

#### 对话上下文 + 文件快照的双层模型

本系统的信息流转基于两个层次：

**第一层：对话上下文（实时、连续、零成本）**
- 人和当前出场的 AI 角色在同一对话窗口中交互
- 所有澄清、追问、纠偏都在对话中自然发生
- 人发现问题不需要"回到文件里加 issue"，直接在对话框里说就行
- 对话上下文天然承担了反馈回路的功能

**第二层：文件快照（持久化、可审查、可版本控制）**
- 每个阶段的产出以文件形式持久化，作为"确认快照"
- 文件的角色不是 agent 之间的"API 接口"，而是**人对 AI 产出的确认记录**
- 下一阶段的 agent 从对话上下文（主要）+ 前序文件（补充）两方面获取信息
- 文件放进代码库，Git 追踪所有变更

```
对话框中：
  人 ←→ Facilitator（澄清需求）
  人 ←→ Architect（设计方案）
  人 ←→ Developer（实现代码）
  人 ←→ Reviewer（质量验证）

每个阶段产出持久化：
  Architect 阶段结束 → 架构方案.md（确认快照）
  Developer 阶段结束 → 变更记录.md（确认快照）
  Reviewer 阶段结束 → Review 意见.md（确认快照）
```

#### 产出格式设计原则

文件不是给机器解析用的，是给人审核用的：

1. **清晰可审**：每个角色的产出必须足够清晰，让人能判断"这步 OK，继续下一步"
2. **低成本可改**：人想修改时，直接在对话框里说，或者直接编辑文件，不需要特殊的修改机制
3. **信息边界明确**：每个角色从前序环节获取什么信息？从对话上下文能拿到什么，必须从文件里拿什么？这需要在角色定义文件中明确说明

> **与 v0.1 的关键变化**：v0.1 把文件传递描述为"角色间信息传递"的核心机制，容易误导为"agent 之间通过文件自动流转的 pipeline"。v0.2 明确：**对话上下文是第一信息源，文件是持久化快照**。

### 3.3 配置分层

```
项目层（随代码库版本控制）
├── .agents/
│   ├── config.yaml          # 项目级角色配置（激活哪些角色、流转路径）
│   ├── roles/               # 角色定义（每个角色的职责、边界、产出要求）
│   │   ├── architect.md
│   │   ├── developer.md
│   │   └── reviewer.md
│   ├── workflows/           # 工作流定义（可选，复杂项目用）
│   │   └── feature-flow.md
│   └── templates/           # 产出模板（可选，标准化产出格式）
│       ├── arch-template.md
│       └── review-template.md

个人层（本地，不进入版本控制）
├── ~/.agents/
│   ├── config.yaml          # 个人偏好（模型选择、沟通风格等）
│   ├── preferences/         # 编码习惯、沟通风格偏好
│   └── evolution/           # 角色自演进数据（v0.3+）
```

**项目层定义"团队共识"：**
- 我们的技术栈是什么
- 代码规范是什么
- 需求流转的环节有哪些
- 每个角色的职责边界

**个人层定义"个体偏好"：**
- 我喜欢什么样的沟通风格
- 我习惯用什么模型
- 我的历史偏好和上下文

**核心价值：团队共识和个体偏好解耦。** 项目配置可迁移（换人接手自然继承），个人配置有主权（不随项目传播）。

## 四、需求流转工作流

以一个完整的功能需求为例，展示角色如何在对话框中协作：

### 4.1 典型流转

```
阶段零：入口
  人 → Facilitator："我想给系统加一个微信登录功能"
  Facilitator → 人：澄清追问 + 推荐流转路径
  人确认后进入下一阶段

阶段一：设计
  Architect 出场，在对话框中与人讨论技术方案
  对话中：方案讨论、选型权衡、接口定义
  阶段产出 → 架构方案.md（确认快照）

阶段二：实现
  Developer 出场，基于对话上下文 + 架构方案文件开始编码
  对话中：实现细节、边界处理、遇到的问题
  阶段产出 → 代码 + 变更记录.md

阶段三：验证
  Reviewer 出场，做代码评审和测试验证
  对话中：Review 意见、测试覆盖、改进建议
  阶段产出 → Review 意见.md
```

### 4.2 关键设计

- **人可以随时介入**：任何阶段发现问题，直接在对话框里提出，当前角色当场处理
- **可以跳步**：简单 Bug 修复可以跳过 Architect，直接 Developer + Reviewer
- **可以回退**：人在 Reviewer 阶段发现架构有问题，可以让 Architect 重新出场
- **对话是连贯的**：整个过程中，人始终在同一个对话框中，上下文连贯，不需要在多个工具之间切换

### 4.3 人介入的节点

虽然理论上人可以在任何时刻介入，但某些节点天然是"检查点"——建议系统在这些节点提示人确认后再继续：

| 检查点 | 说明 | 默认行为 |
|--------|------|---------|
| 方案确认 | Architect 产出后，人确认方案再进入开发 | 暂停等待确认 |
| 实现完成 | Developer 完成编码后，人确认可以进入评审 | 可自动流转（由配置决定） |
| 评审通过 | Reviewer 产出后，人确认可以交付 | 暂停等待确认 |

> 这些检查点不是强制的，可以在 `.agents/config.yaml` 中配置。

## 五、自演进机制（v0.2 简化版）

### 5.1 v0.2 范围：记录偏好

v0.2 的自演进仅限于**个人偏好的记录和应用**：

- 系统记录用户在协作过程中的偏好信号（喜欢简洁还是详细的回复、常用技术选型倾向等）
- 偏好存储在个人层 `~/.agents/preferences/`
- 角色定义中包含"读取用户偏好"的指令
- 不做自动调优，不做团队学习

### 5.2 后续演进路线

| 版本 | 进化能力 | 说明 |
|------|---------|------|
| v0.2 | 偏好记录 | 个人偏好文件，角色定义中读取 |
| v0.3 | 偏好应用 | 自动将偏好注入角色 prompt，减少重复说明 |
| v0.4 | 团队学习 | 记录最优阵容、常见问题、效率数据 |
| v0.5 | 自动调优 | 基于历史数据自动调整角色定义和流转路径 |

> **与 v0.1 的关键变化**：v0.1 在第一版就设计了完整的个体进化 + 团队进化机制。v0.2 的策略是先证明"多角色协作比单 Agent 好"，再谈"协作越来越聪明"。分散注意力的功能晚做。

## 六、可观测性

多 agent 协作最大的调试坑不是"角色定义不好"，而是**出了问题不知道问题出在哪个环节**。

### 6.1 最轻量日志

v0.2 就加一个 `.agents/log.jsonl`，每次角色处理追加一条记录：

```jsonl
{"ts":"2026-05-09T14:30:00+08:00","role":"architect","input":"需求文档.md","output":"架构方案.md","status":"done","duration_s":120}
{"ts":"2026-05-09T14:45:00+08:00","role":"developer","input":"架构方案.md","output":"变更记录.md","status":"done","duration_s":300}
{"ts":"2026-05-09T15:20:00+08:00","role":"reviewer","input":"变更记录.md","output":"review-意见.md","status":"issues_found","duration_s":180}
```

字段定义：
- `ts`：时间戳
- `role`：当前出场的角色
- `input`：消费的文件（如有）
- `output`：产出的文件（如有）
- `status`：`done` / `issues_found` / `blocked` / `skipped`
- `duration_s`：耗时（秒）

**不需要可视化 dashboard，但至少能 `tail -f` 看流转状态。** 调试时会省大量时间。

### 6.2 后续扩展

- v0.3：聚合统计（每个角色的平均耗时、常见阻塞点）
- v0.4：集成到 IDE 内的状态面板

## 七、模型路由

不同角色可以运行在不同的 AI 模型上：

| 角色类型 | 模型倾向 | 原因 |
|---------|---------|------|
| Architect | 强推理模型（Opus 级） | 需要深度思考和多维度权衡 |
| Developer | 快速编码模型（Sonnet/GLM 级） | 吞吐量优先，上下文干净 |
| Reviewer | 专项模型 | 覆盖率和准确性优先 |
| Facilitator | 轻量快速模型 | 响应速度优先，不执行复杂任务 |

**实现依赖：** 需要底层工具支持 subagent + 模型指定能力。当前 OpenClaw 的 `sessions_spawn` 已支持 `model` 参数，基本可行。

## 八、关键设计决策及理由

| 决策 | 选择 | 理由 |
|------|------|------|
| Agent 定位 | 辅助角色，非执行主体 | 当前 AI 可靠性不足以承担执行主体责任；人做最终判断才能保证质量 |
| 核心场景 | 人在 IDE 中的对话式工作流 | 人是中心节点，对话本身就是反馈回路，不需要额外的通知/状态机制 |
| 信息流转 | 对话上下文为主 + 文件快照为辅 | 对话是第一信息源（实时、零成本），文件是持久化确认（可追溯、可审查） |
| 文件角色 | 人的确认快照，不是 agent 间的 API | 文件是给人审核的，不是给机器解析的 |
| 工作流驱动方式 | 需求流转（而非任务列表） | 以交付物为主线更符合实际工作节奏 |
| 入口设计 | Facilitator 作为轻量引导 | 不是智能路由器，是引导助手；复杂判断由人通过对话补全 |
| v0.2 角色数量 | 精简到 4 个核心角色 | 先验证协作机制，再验证角色分工 |
| 自演进范围 | v0.2 只做偏好记录 | 先证明多角色协作有效，再做智能进化 |
| 可观测性 | v0.1 就有 log.jsonl | 成本极低，调试价值极高 |
| 配置管理 | 项目层 vs 个人层分离 | 团队共识和个体偏好解耦，项目可迁移 |
| 是否强制流程 | 否，可跳步可回退 | 简单需求不需要重型流程，灵活性优先 |

## 九、与现有方案的对比

| 维度 | BMAD | Qoder 专家团 | 本方案 |
|------|------|-------------|-------|
| Agent 定位 | 执行主体 | 执行主体 | **辅助角色，人为主** |

| 可观测性 | 无 | 无 | **v0.1 就有 log.jsonl** |

| 工具绑定 | 无（纯提示词） | 绑定 Qoder IDE | **无（对话+文件驱动）** |

## 十、可行性评估

### 10.1 技术可行性

| 能力 | 当前状态 | 说明 |
|------|---------|------|
| 多 Agent 调度 | ✅ 可行 | OpenClaw sessions_spawn 支持 subagent + model 指定 |
| 对话式交互 | ✅ 可行 | 当前 IDE 内的 AI 对话窗口天然支持 |
| 文件持久化 | ✅ 可行 | 纯文件读写，无技术障碍 |
| 配置分层 | ✅ 可行 | 项目目录 + 用户目录，标准文件管理 |
| 可观测性（log.jsonl） | ✅ 可行 | 纯文本追加，零依赖 |
| 模型路由 | ⚠️ 基本可行 | sessions_spawn 支持 model 参数，需验证不同模型对角色定义的兼容性 |
| 偏好记录 | ✅ 可行 | 本地文件存储，角色 prompt 中读取 |

### 10.2 主要风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 角色定义不够精确导致输出质量不稳定 | 高 | 先用 4 个核心角色验证，逐步扩展；角色定义中明确信息边界 |
| 对话上下文过长导致后期角色"遗忘"早期信息 | 中 | 文件快照承担关键信息持久化；适时用文件补充上下文 |
| Facilitator 路由质量依赖项目上下文理解 | 中 | 降低对 Facilitator 的期望，复杂场景由人通过对话补全 |
| 团队接受度——新工作方式的采用成本 | 中 | 对话式交互降低入门门槛；先在非关键项目试点 |

## 十一、落地路径

### Phase 1：骨架搭建（Week 1）

定义 4 个核心角色的 `.md` 定义文件 + 配置结构 + log.jsonl 格式。

产出：
- `.agents/config.yaml`
- `.agents/roles/facilitator.md`
- `.agents/roles/architect.md`
- `.agents/roles/developer.md`
- `.agents/roles/reviewer.md`
- `.agents/templates/`（可选，初步产出模板）

### Phase 2：跑通一个真实需求（Week 2）

用一个真实的小需求（比如给现有项目加一个 API endpoint），跑一遍完整流转。

验证重点：
- 角色之间信息传递是否流畅
- 人介入的体验是否自然
- 对话上下文是否足够支撑下游角色工作
- 文件快照是否有价值（还是纯对话就够了）

### Phase 3：调优（Week 3）

基于跑通的结果：
- 调整角色定义（职责边界是否清晰、产出要求是否合理）
- 调整信息边界定义（哪些信息必须从文件拿、哪些从对话拿）
- 收集痛点

### Phase 4：决策点（Week 4）

基于三周验证结果决定：
- 是否扩展角色集（引入产品经理、运维等）
- 是否启动偏好记录机制（v0.2 自演进）
- 是否需要更结构化的产出模板
- 是否需要工作流定义文件（复杂项目）

## 附录 A：术语表

| 术语 | 定义 |
|------|------|
| **专家角色（Specialist）** | 有明确职责边界的专项 Agent，只负责特定领域的工作 |
| **Facilitator** | 轻量引导角色，帮助用户理清"该做什么、按什么顺序" |
| **对话式工作流** | 人在 IDE 对话框中、以连贯上下文驱动 AI 专家协作的工作模式 |
| **文件快照** | 阶段性产出的持久化记录，用于确认和版本追踪，不是 agent 间的通信协议 |
| **检查点** | 建议人确认后再继续的节点，非强制 |
| **项目层配置** | 随代码库版本控制的团队共识配置，所有人共享 |
| **个人层配置** | 保留在用户本地的个体偏好配置，不进入版本控制 |
| **需求流转** | 一个需求从提出到交付的完整生命周期，每个环节由对应的角色负责 |
| **模型路由** | 不同角色使用不同 AI 模型的能力 |
