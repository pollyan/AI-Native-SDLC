## Q1：交互流程

### 决策
**混合自适应模式** — 根据输入质量自动选择交互模式。

### 理由
企业场景中输入质量差异很大，不能用一种模式覆盖所有情况。

### 竞品参考
- superpowers：一次一问（每次只问一个问题）
- spec-kit：批量 + [NEEDS CLARIFICATION] 标记
- gstack：混合（根据输入质量选择模式）
- bmad：Fast path vs Coaching path

### 具体实现
- 输入质量差（一句话/口头描述）→ 一次一问模式
- 输入质量好（完整 PRD）→ 批量处理 + NEEDS CLARIFICATION
- 中间状态 → 混合模式

## Q2：Spec 模板字段

### 决策

**必填字段（7 项）：**
1. 需求来源（输入类型 + 链接）
2. 需求描述（澄清后的完整描述）
3. 功能需求（每个需求：现状 → 目标 → 验收标准，参考 gsd 三元组）
4. 验收标准（可证伪的 Pass/Fail 检查项）
5. 边界定义（In Scope / Out of Scope）
6. 澄清记录（Q&A 过程）
7. 自审核检查（6 项）

**可选字段（4 项）：**
1. 术语定义（新发现的领域术语）
2. 约束条件（时间/技术/兼容性）
3. 依赖关系
4. 风险识别

**特色机制：**
- [NEEDS CLARIFICATION] 标记 — **不设数量上限**，有多少标多少
- 每个 NEEDS CLARIFICATION 必须说明**为什么需要澄清**（影响什么决策）

### 竞品参考
- gsd：Current/Target/Acceptance 三元组
- spec-kit：[NEEDS CLARIFICATION] ≤3 个（我们改为不设上限）
- bmad：Decision Log（我们合入澄清记录，不独立文件）

### 安辉的关键调整
- **[NEEDS CLARIFICATION] 不设上限**：需求评审的核心目的就是发现问题，有多少列多少
- 每个标记必须说明原因（影响什么决策），不能标完就放着

## Q3：门控机制

### 决策
**6 项定性 checklist + Developer 人工确认。不做量化评分。**

### 6 项自审核检查
1. 问题定义清晰
2. 范围已界定
3. 验收标准已列出
4. 关键术语已定义
5. 无 placeholder/TBD 残留
6. **内部一致性**（新增）— Spec 内部各章节之间是否有矛盾

### 理由
- 量化评分（gsd 的 Ambiguity Score）实现复杂，AI 自评打分准确度存疑
- 定性检查更直观，Developer 一眼能看懂
- 6 项已够用，未来需要时再加量化

### 竞品参考
- gsd：4 维度量化评分（Goal 35% / Boundary 25% / Constraint 20% / Acceptance 20%）
- superpowers：4 项定性检查（Placeholder / Consistency / Scope / Ambiguity）
- 我们：取 superpowers 的定性思路，加一项内部一致性检查

## Q4：AI 提示词规范

### 决策
**bmad 的 Coach 风格 + gstack 的反谄媚规则，两者结合。**

### 态度：Coach Don't Quiz（bmad）
- 对话风格是"协作探索"，不是"考试"
- 不让用户感到被审问

### 内容：反谄媚规则（gstack）
- 禁止说"这个需求看起来很合理"、"不错的方案"、"可以接受"
- 必须对每个需求点给出四种状态之一：

| 状态 | 含义 | AI 动作 |
|------|------|---------|
| ✅ 清晰 | 理解到位 | 直接写入 Spec |
| ⚠️ 模糊 | 理解不完全 | 标记 [NEEDS CLARIFICATION] + 具体缺什么 |
| ❌ 矛盾 | 和已有信息冲突 | 指出冲突点 + 要求澄清 |
| 🔍 缺失 | 完全没有信息 | 提出问题 + 建议默认值 |

## Q5：边界条件处理

### 决策
**AI 自动判断输入质量 + 给出推荐路径和理由 + Developer 确认。**

### 输入质量分级

| 质量等级 | 示例 | 推荐路径 |
|---------|------|---------|
| 极差 | "帮我加个导出功能" | 一次一问，逐个澄清 |
| 较差 | Issue 有基本描述但缺细节 | 先读代码库理解上下文，再针对性提问 |
| 中等 | 标准 PRD | 批量处理 + NEEDS CLARIFICATION |
| 极好 | 完整的技术设计文档 | 直接生成 Spec，Self-Review 后确认 |

### 额外场景

| 场景 | 处理方式 |
|------|---------|
| Developer 不熟悉业务 | AI 多提供背景解释，问题更具体（给选项而不是开放式提问） |
| Developer 很熟悉业务 | 跳过基础问题，直接聚焦边界条件和异常场景 |
| 跨团队需求 | 标记跨团队依赖，建议先和关联团队对齐 |

### 理由
- 减少 Developer 操作步骤（不需要手动选模式）
- 但保留人工干预的能力（Developer 可以覆盖 AI 的判断）
- AI 必须给出充分的理由，不能只说"我判断输入质量为中等"

## Q6：输出质量保证

### 决策
**四层防护机制。**

| 层级 | 机制 | 说明 |
|------|------|------|
| 1 | 禁止猜测规则 | AI 不确定的内容必须标 [NEEDS CLARIFICATION]，不允许编造 |
| 2 | Placeholder 扫描 | 产出后自动扫描 TBD/TODO/待定/?? 等占位符，不通过门控 |
| 3 | 自审核 6 项 | 门控机制本身 |
| 4 | Developer 最终确认 | 人审阅通过才算完成 |

### MVP 不做的事
- 不做自动修复循环（spec-kit 的做法）—— 修复由 Developer 决定
- 不做跨模型第二意见（gstack 的做法）—— 记入功能积压
