### 1.6 各框架独特亮点

## 三、各框架的独特亮点

### 3.1 只在该框架中出现的优秀设计

| 框架 | 独特亮点 | 描述 |
|------|---------|------|
| **Superpowers** | Spec Self-Review 循环 | 写完 spec 后自动做 placeholder scan、一致性检查、scope 检查，发现问题直接修复，不需要再跑一遍。这是唯一内建自审核的框架 |
| **Superpowers** | 方案对比强制 | 要求提供 2-3 个方案含 trade-off + 推荐标记，而不是只给一个方案 |
| **SpecKit** | `[NEEDS CLARIFICATION]` 标记 | 不确定的需求显式标记，不跳过也不猜，强制留下审计痕迹。这个设计极其轻量但极其有效 |
| **SpecKit** | User Story 独立可测试性声明 | 每个 User Story 必须声明"如何独立测试 + 交付什么价值"，防止不可验证的故事混入 |
| **SpecKit** | Constitution Check | 项目级"宪法"文件定义质量标准，所有产出必须通过宪法检查 |
| **SpecKit** | Complexity Tracking | 当设计违反了 Constitution 中的规则，必须填写复杂度追踪表（违反了什么 + 为什么需要 + 更简方案为什么不行） |
| **gstack** | Six Forcing Questions | YC 风格的六个追问，把模糊的需求变成经过压力测试的问题定义 |
| **gstack** | Problem Reframing | AI 不按用户说的原样理解，而是重新理解"你真正要解决的是什么" |
| **gstack** | 多维度评审 | CEO/Design/Eng/DX 四个独立视角评审，每个视角有独立的 skill 和输出 |
| **gstack** | Reflection on Thinking | AI 对用户思考方式的观察记录在 design doc 里，下次读到会重新遇到 |
| **GSD** | STATE.md 跨会话状态 | 精确记录当前项目位置和决策，新会话加载后立刻知道状态 |
| **GSD** | YAML Frontmatter 元数据 | PLAN.md 使用结构化 frontmatter（phase/plan/title/dependencies/estimated_tasks），可被工具解析 |
| **GSD** | RESEARCH.md 领域研究 | 每个 Phase 开始前做领域研究，研究结果被后续 Plan 和 Execute 引用 |
| **BMAD** | PRFAQ（Press Release + FAQ） | 亚马逊风格的文档——先写新闻稿和 FAQ，倒逼需求清晰化 |
| **BMAD** | 12 步 PRD 引导流程 | 最完整的 PRD 引导，从 Discovery 到 Polish，每步有独立的 .md 文件 |
| **BMAD** | FR/NFR 严格分离 | 功能需求和非功能需求分开管理，NFR 有明确的 Category + Measurement |
| **BMAD** | Implementation Readiness Check | 实现前的就绪检查，验证所有规划文档的一致性 |
| **OpenSpec** | Delta Spec（增量规范） | 只描述变化（ADDED/MODIFIED/REMOVED），不全量重写，归档时自动 merge |
| **OpenSpec** | 源事实 vs 变更分离 | `specs/` 是当前系统的真实行为，`changes/` 是提案，两者严格分离 |
| **OpenSpec** | RFC 2119 关键词 | 使用标准化的 SHALL/MUST/SHOULD 表达需求强度 |
| **OpenSpec** | 非线性 Action-based | 不是阶段式而是 action-based——随时可以 propose/apply/archive |
| **specs.md** | 自适应检查点 | 根据任务复杂度自动调整检查点数量（0-2），不强制一刀切 |
| **specs.md** | Intent 概念 | 用"意图"而非"需求"作为核心组织单元，更贴近业务思维 |
| **specs.md** | 三种流程可选 | Simple/FIRE/AI-DLC 三种流程适配不同复杂度，用户自选 |

#### Superpowers — executing-plans / subagent-driven-development

**两种执行模式：**

**模式 A：executing-plans（单 Agent 内联执行）**
1. Load and review plan — 加载计划文件，审查关键缺失
2. Execute tasks — 逐 Task 执行，每个 Task 包含：加载 → 理解 → 实现 → 运行验证 → 标记完成
3. Complete development — 全部完成后总结

**模式 B：subagent-driven-development（多 Agent 分发执行）**
1. 读取 plan 文件一次，提取所有 task 全文
2. 创建 TodoWrite 跟踪列表
3. 对每个 Task：
   - 分发独立 implementer subagent（带完整 task text + context）
   - 实现完成后，分发 spec compliance reviewer subagent
   - spec 合规后，分发 code quality reviewer subagent
   - 两阶段 review 都通过后，标记 Task 完成
4. 不停止问"是否继续"——用户要求执行就执行到底

**关键规则：**
- 每个 subagent 不继承 session context，主 agent 构造精确指令
- 只有三种停止原因：BLOCKED（无法解决）、真正的歧义、全部完成
- "Should I continue?" 提示是反模式，浪费时间

**执行状态追踪：**
- 使用 TodoWrite 工具维护任务状态列表
- 每个 task checkbox 实时更新

### 1.2 五大核心设计理念

#### 理念一：门控驱动的阶段推进（Phase Gates）

几乎每个工程 skill 都有明确的 **"不要跳过"** 边界：

- `diagnose`："Do not proceed to Phase 2 until you have a loop you believe in"、"Do not proceed until you reproduce the bug"
- `tdd`："Never refactor while RED. Get to GREEN first"
- `review`："Don't proceed until you have it"（指 fixed point）

**设计模式**：每个阶段有明确的 **进入条件** 和 **退出条件**，不满足条件不往下走。这不是"建议"，而是 hard gate。

**对我们的启示**：我们的框架 v0.3.1 有检查点概念但偏软（"建议系统提示人确认"）。Matt Pocock 的做法更极端——即使在单 agent 内部，也用硬门控强制阶段推进的质量。

#### 理念二：反馈循环是核心能力（Feedback Loops First）

`diagnose` skill 的 Phase 1 标题是 **"Build a feedback loop — This IS the skill"**。全文用了最大篇幅（占整个 skill 的 ~40%）讲如何构建反馈循环，提供了 10 种从高到低排列的循环构建方式，以及优化循环本身的 4 个维度。

这不是"调试技巧"，而是一个**方法论立场**：没有可重复的 pass/fail 信号，任何分析都是猜测。

**核心金句**：
> "If you have a fast, deterministic, agent-runnable pass/fail signal for the bug, you will find the cause. If you don't have one, no amount of staring at code will save you."

**对我们的启示**：我们的 Developer 角色有"调试能力"，但没有给调试一个系统化的方法论。`diagnose` 的反馈循环方法论可以直接嵌入。

#### 理念三：反锚定——强制多假设（Anti-Anchoring）

两个 skill 都强调了**反锚定**：

- `diagnose` Phase 3："Generate 3–5 ranked hypotheses before testing any of them. Single-hypothesis generation anchors on the first plausible idea."
- `review`：用两个**并行 sub-agent** 分别审查 Standards 和 Spec，"so they don't pollute each other's context"

**设计模式**：AI 天然有锚定倾向（抓住第一个想法不放），通过**结构化约束**（必须生成多个假设、必须并行独立审查）来对抗。

**对我们的启示**：我们的 Reviewer 角色做质量验证时，应该引入多维度独立审查而不是一次"综合检查"。

#### 理念四：领域语言一致性（Domain Language Consistency）

**几乎所有 engineering skill 都要求**：

- "use the project's domain glossary"
- "respect ADRs in the area you're touching"
- "use CONTEXT.md vocabulary for the domain"

这不是某个 skill 的特殊要求，而是**跨 skill 的基础设施约定**。`setup-matt-pocock-skills` 就是专门配置这个基础设施的 skill——配置 issue tracker、triage labels、domain doc layout。

**配套机制**：
- `CONTEXT.md`：领域词汇表（纯术语定义，不含实现细节）
- `docs/adr/`：架构决策记录
- `CONTEXT-MAP.md`：多上下文仓库的入口索引

**对我们的启示**：我们的框架有 `config.yaml` 中的 `tech_stack` 和 `code_style`，但**没有领域词汇一致性机制**。这是 `grill-with-docs` 和 `ubiquitous-language`（已废弃但理念保留在 `grill-with-docs` 中）的核心贡献。

#### 理念五：垂直切片优于水平切片（Vertical Slices Over Horizontal）

`tdd` skill 明确反对"水平切片"——不要先写完所有测试再写所有实现：

> "DO NOT write all tests first, then all implementation. This produces CRAP TESTS."

正确做法是**垂直的 tracer bullet**：一个测试 → 一个实现 → 重复。

`to-issues` 同样强调垂直切片：每个 issue 是一个 "thin vertical slice that cuts through ALL integration layers end-to-end"。

**对我们的启示**：我们的 Developer 角色在 Execute 阶段按 Task List 逐步实现，但没有强调**切片策略**。应加入垂直切片的强制约束。

### 1.3 设计模式总结

| 模式 | 体现 | 核心机制 |
|------|------|---------|
| 阶段门控 | diagnose、tdd、review | 每阶段有明确的进入/退出条件 |
| 反馈循环优先 | diagnose Phase 1 | 先建信号再做分析 |
| 反锚定 | diagnose（多假设）、review（并行 sub-agent） | 结构化对抗锚定倾向 |
| 领域语言一致 | 所有 engineering skill | CONTEXT.md + ADRs 基础设施 |
| 垂直切片 | tdd、to-issues | Tracer bullet，不水平切片 |
| 内联文档更新 | grill-with-docs、improve-codebase-architecture | 决策产生时立即更新 CONTEXT.md/ADR |
| 可证伪假设 | diagnose Phase 3 | 每个假设必须有可证伪的预测 |
| 状态机驱动 | triage | 5 个状态角色，明确的转换规则 |
| 一次性原型 | prototype | 明确标记 throwaway，用完即删 |
| 两轴评审 | review | Standards 轴 + Spec 轴独立报告 |

### 2.3 可增强（我们有但 Matt 做得更深）

以下是我们现有设计可以**从 Matt Pocock 具体方法论中吸收增强**的领域：

#### 增强 1：阶段内的硬门控细化

**现状**：我们的检查点在阶段之间（Plan 完成、Execute 完成、Review 完成），但**阶段内部**没有门控。

**Matt 的做法**：`diagnose` 在单个 skill 内部就有 6 个 phase，每个 phase 都有明确的 "Do not proceed until..." 条件。例如：
- Phase 1（建反馈循环）→ 必须有可靠的 pass/fail 信号才能进入 Phase 2
- Phase 2（复现）→ 必须复现 bug 才能进入 Phase 3
- Phase 3（假设）→ 必须列出 3-5 个可证伪假设才能进入 Phase 4

#### 增强 2：Developer 角色的调试方法论

**现状**：Developer 有"调试能力"，但没有系统化方法。

**Matt 的做法**：`diagnose` 提供了完整的 6 阶段调试方法论，从"建反馈循环"到"清理+事后复盘"，每个阶段有具体步骤和决策树。

#### 增强 3：Reviewer 角色的结构化评审

**现状**：Reviewer 有"质量验证"能力，但没有定义评审维度和报告结构。

**Matt 的做法**：`review` 用两轴独立审查（Standards vs Spec），并行 sub-agent 各自输出不超过 400 字的报告。两轴分开报告，防止一个维度掩盖另一个维度的问题。

#### 增强 4：Architect 角色的拷问式方案设计

**现状**：Architect 有"方案设计"能力，但缺少与用户交互的结构化方法。

**Matt 的做法**：`grill-with-docs` 提供了完整的拷问式审查流程——逐个问题拷问、内联更新 CONTEXT.md 和 ADR、精确术语检查。`improve-codebase-architecture` 在此基础上加了"深度模块"概念和 grilling loop。

#### 增强 5：领域词汇管理基础设施

**现状**：我们有 `config.yaml` 中的 `tech_stack` 和 `code_style`，但没有领域词汇一致性机制。

**Matt 的做法**：`setup-matt-pocock-skills` 配置 `CONTEXT.md`（领域词汇表）+ `docs/adr/`（架构决策记录）。所有 engineering skill 启动时都读取这些文件，确保产出使用一致的领域语言。

## 四、不建议借鉴的部分（及理由）

### 4.1 不建议借鉴：caveman 压缩沟通模式

**理由**：caveman 模式通过砍掉冠词、填充词、寒暄来降低 token 消耗约 75%。但在我们的**对话式工作流**中，人与 AI 的对话本身就是核心交互模式。过度压缩会降低可读性和交互体验。我们的框架定位是"辅助角色、人为主"，沟通质量比 token 效率更重要。

**例外**：如果用户主动要求简洁模式，可以作为 Analyst/Developer 角色的可选开关。

### 4.2 不建议借鉴：git-guardrails（Claude Code 专属）

**理由**：这是 Claude Code 的 PreToolUse hook 机制，依赖特定的工具链。我们的框架是工具无关的（"对话 + 文件驱动"），安全防护应该放在 config.yaml 的 Constitution 机制中，而不是工具链级别的 hook。

### 4.3 不建议借鉴：migrate-to-shoehorn

**理由**：这是 TypeScript 测试中 `as` 类型断言的迁移工具，是特定技术栈的特定问题。我们的框架是语言/技术栈无关的。

### 4.4 不建议借鉴：scaffold-exercises

**理由**：这是课程练习的脚手架工具，属于教育培训场景，与我们的开发工作流框架无关。

### 4.5 不建议借鉴：obsidian-vault

**理由**：个人笔记管理工具，与框架设计无关。

### 4.6 不建议借鉴：设计即废的 skill（design-an-interface、request-refactor-plan）

**理由**：这些已 deprecated。`design-an-interface` 的"Design It Twice"理念（生成 3+ 个完全不同的接口设计然后比较）有启发性，但实现上依赖并行 sub-agent，与我们当前的单对话上下文设计不匹配。如果未来支持 sub-agent 可以重新评估。

### 4.7 不建议借鉴：writing-beats / writing-fragments / writing-shape

**理由**：这些是文章写作工具，属于内容创作领域，与软件开发工作流无关。但其中的**逐段构建、逐步确认**的交互模式值得注意——如果未来框架扩展到文档撰写工作流，可以参考。

### 4.8 不建议借鉴：单层 skill 调用模式

**理由**：Matt Pocock Skills 是**扁平的 skill 列表**，用户按需调用单个 skill。没有"工作流"的概念——没有 skill 之间的衔接、没有阶段推进、没有状态管理。我们的双层架构（工作流 × 角色能力）是刻意的设计选择，提供更好的用户引导和端到端体验。不应该退回到扁平的 skill 列表。

## 六、核心判断

Matt Pocock Skills 库最大的价值不在于单个 skill 的功能，而在于**它展示了一种"怎么把工程方法论编码成 AI 可执行的流程"的方式**：

1. **每个方法论都有步骤、有门控、有退出条件**——不是"确保代码质量好"这种空话，而是"运行测试并验证所有测试通过"
2. **反锚定机制贯穿始终**——无论是调试还是评审，都强制 AI 做多假设/多维度分析
3. **领域语言一致性是基础设施**——所有 skill 都依赖 CONTEXT.md 和 ADRs，确保术语一致
4. **反馈循环优先于分析**——先建可重复信号再做猜测

这些设计理念可以直接嵌入我们现有的角色定义文件中，不需要改变框架的架构（双层结构、对话式工作流、文件快照等核心设计保持不变）。增强发生在**能力定义的深度**层面，而不是架构层面。

## 四、统一语言基础设施建议

## 一、为什么统一语言在 AI-Native SDLC 里不是"好实践"，而是"基础设施"

### 1.1 人可以模糊沟通，AI 不行

人与人协作时，说"把这个订单处理一下"，对方能从上下文推断是 Order 还是 Fulfillment。但 AI 不具备这种隐性推理能力——它对术语的理解完全来自你给它的定义。

如果 `CONTEXT.md` 里定义了 "Order"，而：
- **Analyst** 在需求澄清时说"购买请求"
- **Architect** 在方案里写 "Purchase"
- **Developer** 在代码里用 `Transaction`
- **Reviewer** 在评审时说"订单"

这四个角色之间的信息传递就会**逐步积累偏差**，每个阶段都丢一点精度，到 Reviewer 那里可能已经面目全非。

在单 Agent 模式下，这个问题不严重——同一个 Agent 上下文连贯，术语漂移有限。但在**多角色协作**模式下（正是我们的框架设计），术语一致性是信息无损传递的前提条件。

### 1.2 Matt Pocock 的做法：不是"建议参考"，是"强制执行"

Matt Pocock Skills 库的做法比一般 DDD 实践更激进：

1. **每个 engineering skill 都硬编码了** "use the project's domain glossary"
2. **所有 skill 启动时必须读取** `CONTEXT.md` 和 `docs/adr/`
3. **grill-with-docs 在拷问过程中实时拦截术语冲突**——用户用的词和 `CONTEXT.md` 不一致时立即叫停
4. **发现新术语时立即写入**，不批量处理
5. **废弃了专门的 `ubiquitous-language` skill**，把术语维护融合到了日常工作中（边拷问方案边维护术语，比专门建词汇表更自然）

这个设计哲学很对：**统一语言不是一次性建表，是持续维护的活文档**。

## 二、Matt Pocock 的具体机制拆解

### 2.1 CONTEXT.md — 领域词汇表

格式规范（来源：`CONTEXT-FORMAT.md`）：

```markdown
# {上下文名称}

{一两句话说明这是什么上下文、为什么存在}

## Language

**Order**:
客户提交的购买请求，包含一个或多个 LineItem
_Avoid_: Purchase, transaction, 购买请求

**Invoice**:
发货后发送给客户的付款请求
_Avoid_: Bill, payment request, 账单

**Customer**:
下单的个人或组织
_Avoid_: Client, buyer, account, 客户

## Relationships

- 一个 **Order** 产生一个或多个 **Invoice**
- 一个 **Invoice** 属于恰好一个 **Customer**

## Example dialogue

> **Dev:** "当 **Customer** 下了一个 **Order** 时，我们立刻创建 **Invoice** 吗？"
> **Domain expert:** "不——**Invoice** 只在 **Fulfillment** 确认后才生成。"

## Flagged ambiguities

- "account" 被同时用来指 **Customer** 和 **User** — 已解决：这是两个不同概念。
```

**关键规则**：
- 每个术语定义**一句话**，定义它"是什么"，不定义它"做什么"
- 同义词标记为 `_Avoid_`，不是禁止使用，而是明确"应该避免的别名"
- `Relationships` 用加粗术语名 + 基数表达
- `Example dialogue` 用对话体展示术语在自然交流中的用法——这一条很关键，AI 从对话中学习的效率远高于从定义中学习
- `Flagged ambiguities` 记录历史上出现过的术语冲突及解决结果
- **只包含项目特有的领域术语**，不含通用编程概念

### 2.2 CONTEXT-MAP.md — 多上下文仓库

对于 monorepo 或多子域的项目：

```markdown
# Context Map

## Contexts

- [Ordering](./src/ordering/CONTEXT.md) — 接收和跟踪客户订单
- [Billing](./src/billing/CONTEXT.md) — 生成发票和处理付款
- [Fulfillment](./src/fulfillment/CONTEXT.md) — 管理仓储拣选和发货

## Relationships

- **Ordering → Fulfillment**: Ordering 发出 `OrderPlaced` 事件；Fulfillment 消费以启动拣选
- **Fulfillment → Billing**: Fulfillment 发出 `ShipmentDispatched` 事件；Billing 消费以生成发票
- **Ordering ↔ Billing**: 共享类型 `CustomerId` 和 `Money`
```

**推断规则**：
- 存在 `CONTEXT-MAP.md` → 多上下文模式
- 只存在根 `CONTEXT.md` → 单上下文模式
- 都不存在 → 懒创建，在第一个术语被确定时创建

### 2.3 ADR（Architecture Decision Record）— 轻量决策记录

Matt Pocock 的 ADR 格式**极度精简**：

```markdown
# {决策的简短标题}

{1-3 句话：上下文是什么、决定了什么、为什么。}
```

**只创建 ADR 的条件**（三个必须同时满足）：
1. **难回退** — 改变主意的成本有意义
2. **不看上下文会觉得奇怪** — 未来读者会问"为什么他们这样做"
3. **有真实权衡** — 存在真正的备选方案并且你选了其中一个

如果容易回退、不令人意外、没有真正替代方案——就不需要 ADR。

### 2.4 grill-with-docs 的术语维护流程

在拷问方案的过程中，grill-with-docs 执行以下术语相关动作：

| 时机 | 动作 |
|------|------|
| 用户使用的术语与 CONTEXT.md 冲突 | 立即指出："你的词汇表定义 'cancellation' 为 X，但你似乎在说 Y——到底是哪个？" |
| 用户使用模糊或重载的术语 | 提议精确术语："你说 'account'——你指的是 Customer 还是 User？这是两个不同的概念。" |
| 讨论域关系时 | 构造具体场景探测边界情况，逼迫用户精确化概念边界 |
| 用户陈述与代码不一致时 | 交叉验证："代码取消的是整个 Order，但你刚说可以部分取消——哪个是对的？" |
| 术语被确定时 | 立即更新 CONTEXT.md，不批量 |
| 满足 ADR 三个条件时 | 提出 ADR 建议 |

## 三、融入 AI-Native-SDLC 的具体方案

### 3.1 在框架中的定位

统一语言应该是一个**跨角色的基础设施层**，不是某个角色的专属能力：

```
┌─────────────────────────────────────────┐
│           工作流层（Brainstorm/Plan/     │
│           Execute/Review）              │
├─────────────────────────────────────────┤
│    角色能力层（Analyst/Architect/        │
│    Developer/Reviewer）                 │
├─────────────────────────────────────────┤
│  ★ 基础设施层（CONTEXT.md + ADRs）      │  ← 新增
│  所有角色启动时读取，产出时遵守           │
└─────────────────────────────────────────┘
```

### 3.2 文件结构

```
.agents/
├── context.md              # 领域词汇表（新建）
├── context-map.md          # 多上下文索引（可选，monorepo 时）
├── docs/
│   └── adr/                # 架构决策记录（新建）
│       └── 0001-xxx.md
├── config.yaml             # 现有配置
└── roles/
    ├── analyst.md          # 增强：术语发现 & 澄清
    ├── architect.md        # 增强：术语冲突拦截 & ADR
    ├── developer.md        # 增强：代码与术语一致性
    └── reviewer.md         # 增强：术语合规检查
```

### 3.3 各角色的增强内容

#### Analyst（Brainstorm 阶段）
- **新增能力：术语发现**
  - 在澄清需求时，主动识别领域术语
  - 发现一词多义/多词一义时，提议 canonical term
  - 每个术语被确定后，立即写入 `context.md`
- **触发词**：用户提及项目特有的业务概念时自动触发

#### Architect（Plan 阶段）
- **新增能力：术语冲突拦截**
  - 方案中使用术语前，先检查 `context.md`
  - 用户使用的术语与 `context.md` 冲突时立即指出
  - 讨论域关系时构造具体场景探测术语边界
  - 发现新的架构决策满足 ADR 三条件时提出记录
- **拷问式方案审查增强**：grill-with-docs 的完整流程可以融入 Architect 的方案设计能力

#### Developer（Execute 阶段）
- **新增能力：代码与术语一致性**
  - 编码时使用 `context.md` 中的术语命名（类名、方法名、变量名）
  - 发现代码中的命名与 `context.md` 不一致时指出
  - 不需要在代码里找新术语——那是 Analyst 和 Architect 的工作

#### Reviewer（Review 阶段）
- **新增能力：术语合规检查**
  - 作为评审的一个维度：产出是否使用了一致的领域语言
  - 发现术语漂移时标记为 issue

### 3.4 context.md 格式规范（适配我们的框架）

在 Matt Pocock 的基础上，我们做以下适配：

1. **支持中英双语**：中文项目可能有中英文混用的术语，定义时同时给出
2. **与 config.yaml 中的 tech_stack 对齐**：tech_stack 描述技术选型，context.md 描述业务领域，两者互补
3. **懒创建**：在 Brainstorm 阶段的第一次术语确定时自动创建，不需要预先手动建文件
4. **文件大小控制**：通过分组（按子域/生命周期/角色）保持可读性，超过 100 行考虑拆分到 context-map.md 模式

### 3.5 与现有设计的整合点

| 现有设计 | 整合方式 |
|---------|---------|
| 双层架构（工作流 × 角色能力） | 统一语言作为第三层"基础设施层"，被所有角色依赖 |
| 上下文预算管理 | `context.md` 作为**共享缓存**——不同角色加载时读同一个文件，避免各自维护独立的术语理解，减少上下文消耗 |
| 活跃/归档分离 | `context.md` 是活跃文件，长期保留。被废弃的术语移到 `context-archive.md` |
| 门控机制 | 在 Plan → Execute 的检查点中增加"关键术语是否已定义"的验证 |
| 文件快照 | `context.md` 的变更可以作为阶段产出的快照之一——哪些术语在这一轮被新增/修改了 |

## 四、为什么这比"在 config.yaml 里写 tech_stack"强

| 维度 | config.yaml 的 tech_stack | context.md 统一语言 |
|------|--------------------------|-------------------|
| 内容 | 技术选型（语言、框架、数据库） | 业务领域术语（Order、Customer、Fulfillment） |
| 维护时机 | 项目初始化时设一次 | **每次需求讨论时持续更新** |
| 读者 | Developer（知道用什么技术） | **所有角色**（知道说什么语言） |
| 冲突检测 | 无 | 术语冲突时实时拦截 |
| 关系表达 | 无 | 域关系 + 基数 + 示例对话 |
| 漂移检测 | 无 | Reviewer 阶段检查术语合规 |

两者互补，不替代。

## 六、核心判断

Matt Pocock 把 DDD 统一语言从"团队协作的好实践"重新定位成了"AI 系统的信息基础设施"。这个视角转换是关键洞察：

- **对人**：统一语言是锦上添花，人可以用模糊语言沟通
- **对 AI**：统一语言是**信息无损传递的前提条件**，没有它多角色协作就是在玩传话游戏

在我们的 AI-Native SDLC 框架里，这一点比在 Matt Pocock 的单层 skill 模式里更重要——因为我们有四个角色在不同阶段传递信息，每一轮传递都会放大术语漂移的影响。

统一语言不是框架的"功能"，是框架的**地基**。

| 1 | 自适应检查点 | 中 | Phase 3 |

## 八、对我们 v0.3 框架的即时改进建议

以下是对现有 v0.3.1 框架可以**立即吸收**的改进（来自行业最佳实践）：

### 8.1 角色定义文件应从"描述性"改为"过程性"

当前角色定义可能偏向"这个角色是谁、有什么能力"的描述。应该改为"触发条件 → 步骤流程 → 验证清单"的过程性结构。

**来源模式：** 行业 SKILL.md 设计规范（Overview → When to Use → Process → Rationalizations → Red Flags → Verification）

### 8.2 每个角色能力加入 Anti-Rationalization 表格

不是泛泛地说"要严格"，而是**枚举 AI 在这个能力点上最可能的偷懒路径，预设反驳**。

### 8.3 每个角色能力加入 Verification 清单

能力定义的末尾必须是可验证的证据要求。"看起来对"不算通过。

### 8.4 Reviewer 角色加入 Doubt-Driven 子能力

Full 流程的 Review 阶段默认使用对抗性审查，而不是温和的"检查一下"。

### 8.5 Developer 角色加入 Source-Driven 子能力

编码前获取官方文档、标注来源、不确定的标记 UNVERIFIED。

## 六、规范驱动开发框架研究

## 一、研究概览

研究了 7 个 GitHub 上最活跃的规范驱动开发/AI 编程框架。这些框架覆盖了从纯方法论（Superpowers）到完整工具链（SpecKit、OpenSpec）到企业级工作流系统（GSD、gstack）的完整光谱。

**整体趋势：**
1. **工作流阶段化已成共识**——所有框架都把"从想法到交付"拆成多个阶段
2. **Spec 先行是共同信条**——先写规范再写代码，没有例外
3. **"人把关 + AI 执行"是主流模式**——人负责决策和审核，AI 负责实现
4. **技能/Skill 文件化是核心实现方式**——通过 .md 文件定义 AI 的行为
5. **可观测性和状态管理越来越受重视**——GSD 的 `.planning/STATE.md`、OpenSpec 的 `state.yaml` 都是例证

### 1. Superpowers（⭐ 183K）

**仓库：** https://github.com/obra/superpowers
**作者：** Jesse Vincent (Prime Radiant)
**定位：** 一套基于可组合技能的开发方法论，适配所有主流 IDE

#### 核心理念

**技能（Skill）驱动 + 自动触发。** Superpowers 不要求用户手动选择流程——AI agent 在做任何事之前自动检查是否有适用的 skill，然后强制执行。

关键哲学：
- **TDD 强制**——先写测试，再写代码，不例外
- **系统化优于即兴**——用流程代替猜测
- **证据优于声明**——验证后才算完成
- **YAGNI 极简**——不提前做不需要的东西

#### 目录结构

```
superpowers/
├── skills/                    # 核心技能定义
│   ├── brainstorming/         # 阶段 1：设计头脑风暴
│   │   └── SKILL.md
│   ├── writing-plans/         # 阶段 2：写实现计划
│   │   └── SKILL.md
│   ├── executing-plans/       # 阶段 3a：执行计划（单 agent）
│   │   └── SKILL.md
│   ├── subagent-driven-development/  # 阶段 3b：执行计划（多 agent）
│   │   └── SKILL.md
│   ├── test-driven-development/  # 强制 TDD
│   │   └── SKILL.md
│   ├── requesting-code-review/   # 代码评审
│   │   └── SKILL.md
│   └── finishing-a-development-branch/  # 收尾
│       └── SKILL.md
├── .cursor-plugin/            # Cursor 插件
├── .claude-plugin/            # Claude Code 插件
├── .codex-plugin/             # Codex 插件
└── tests/                     # 技能测试
```

#### 工作流设计

```
Brainstorm → Plan → Execute → Review → Finish
```

**关键设计决策：**
1. **HARD-GATE 机制**——brainstorming skill 有一个硬门控：`Do NOT invoke any implementation skill, write any code... until you have presented a design and the user has approved it.`
2. **自审核循环**——写完 spec 后自动做 placeholder scan、内部一致性检查、scope 检查
3. **Subagent 模式**——每个 task 分给独立的 subagent 执行，主 agent 做两阶段 review（先检查 spec 合规性，再检查代码质量）
4. **计划粒度极细**——每个 step 2-5 分钟，包含完整代码和验证命令
5. **无缝衔接**——brainstorming 的终端状态是调用 writing-plans，writing-plans 的终端状态是调用 executing-plans

#### Skill 文件结构

每个 SKILL.md 包含：
```yaml
---
name: brainstorming
description: "..."
---
```
然后是完整的 Markdown 指令，包括：
- 过程流程图
- 具体步骤 checklist
- 检查清单
- 关键原则

#### 亮点

1. **自动触发机制**——AI 自动检测应该用哪个 skill，用户不需要手动选择
2. **极致的 TDD 强制**——代码评审中如果发现代码写在测试之前，会删掉代码
3. **Two-stage review**——先检查"是否按 spec 做"，再检查"代码质量好不好"
4. **计划即 prompt**——PLAN.md 本身就是给 subagent 的 prompt，不是文档
5. **"Anti-pattern: This Is Too Simple"**——即使是简单项目也强制过设计流程

#### 不足

- 没有"角色"概念——所有 skill 都由同一个 agent 执行
- 没有配置分层——没有项目层/个人层的区分
- 没有自演进机制

### 2. BMAD Method（⭐ 46.7K）

**仓库：** https://github.com/bmad-code-org/BMAD-METHOD
**定位：** 完整的 AI 驱动开发方法论，从需求到交付

#### 核心理念

**角色驱动 + 阶段流程。** BMAD 定义了一组有名字、有性格的 AI 角色（Mary the Analyst、Winston the Architect 等），用户通过对话与这些角色交互。

#### 目录结构

```
BMAD-METHOD/src/
├── core-skills/               # 核心技能（通用）
│   ├── bmad-brainstorming/
│   ├── bmad-advanced-elicitation/
│   ├── bmad-distillator/
│   └── ...
├── bmm-skills/                # BMAD 方法论模块
│   ├── 1-analysis/            # 阶段 1：分析
│   │   ├── bmad-agent-analyst/    # Mary - 需求分析师
│   │   ├── bmad-product-brief/
│   │   └── bmad-prfaq/
│   ├── 2-plan-workflows/      # 阶段 2：规划
│   │   ├── bmad-agent-pm/        # 产品经理
│   │   ├── bmad-create-prd/      # 创建 PRD（12 步引导流程）
│   │   ├── bmad-agent-ux-designer/

│   ├── 3-solutioning/         # 阶段 3：方案设计
│   │   ├── bmad-agent-architect/  # Winston - 架构师
│   │   ├── bmad-create-architecture/
│   │   ├── bmad-create-epics-and-stories/
│   │   └── bmad-generate-project-context/
│   └── 4-implementation/      # 阶段 4：实现
│       ├── bmad-agent-dev/        # 开发工程师
│       ├── bmad-dev-story/
│       ├── bmad-code-review/
│       ├── bmad-sprint-planning/
│       └── bmad-retrospective/
```

#### 角色设计

每个角色有独立的 SKILL.md + customize.toml：

**Mary（需求分析师）**：
- 图标：📊
- 身份："Strategic business analyst and requirements expert"
- 沟通风格：专业、证据驱动
- 激活流程：解析配置 → 采纳人设 → 加载持久事实 → 问候 → 展示菜单
- 菜单系统：每个角色有编号菜单，用户可以选择功能

**Winston（架构师）**：
- 图标：🏗️
- 身份："Channels Martin Fowler's pragmatism and Werner Vogels's cloud-scale realism"
- 原则："Rule of Three before abstraction"、"Boring technology for stability"

#### 定制化系统

BMAD 有三层配置合并机制：
```
base (customize.toml) → team (_bmad/custom/{skill}.toml) → user (_bmad/custom/{skill}.user.toml)
```
- 标量覆盖
- 表深度合并
- 带 code/id 的数组替换匹配项并追加新项

#### 亮点

1. **有性格的角色**——不只是"架构师"，而是"Winston，务实风格，Martin Fowler 的追随者"
2. **三层配置合并**——base → team → user，精细控制
3. **菜单系统**——每个角色有编号菜单，用户像点菜一样选择功能
4. **12 步 PRD 引导流程**——从 discovery 到 polish，每步有独立的 .md 文件
5. **persistent_facts**——角色记住的持久事实，支持文件引用和 glob 模式
6. **config.yaml 集中配置**——用户名、沟通语言、文档语言、产出目录等

#### 不足

- 角色太多太重——完整安装有 20+ skills
- 学习曲线陡峭——需要理解整个方法论
- 角色之间切换成本高——每个角色有独立的激活流程

### 3. SpecKit（⭐ 93.8K，GitHub 官方）

**仓库：** https://github.com/github/spec-kit
**定位：** GitHub 官方的 Spec 驱动开发工具链

#### 核心理念

**Spec 变成可执行的。** SpecKit 的核心创新是把 specification 从"废弃的脚手架"变成"直接生成工作实现的源文件"。

#### 目录结构

```
spec-kit/
├── templates/                 # 模板
│   ├── spec-template.md       # 功能规范模板（User Story + Requirements + Success Criteria）
│   ├── plan-template.md       # 实现计划模板（技术上下文 + 任务拆分）
│   ├── tasks-template.md      # 任务模板
│   └── constitution-template.md  # 项目宪法（质量门控）
├── workflows/                 # 工作流定义
│   └── speckit/
│       └── workflow.yml       # specify → plan → tasks → implement
├── presets/                   # 预设配置
├── extensions/                # 扩展
└── integrations/              # IDE 集成
```

#### 工作流设计（workflow.yml）

```yaml
steps:
  - id: specify          # 生成 spec
  - id: review-spec      # 🚪 门控：人工审核 spec
  - id: plan             # 生成计划
  - id: review-plan      # 🚪 门控：人工审核计划
  - id: tasks            # 生成任务
  - id: implement        # 执行实现
```

**关键设计：**
- 工作流是声明式的（YAML），不是代码
- 每个步骤之间有 `gate` 类型的审核节点
- 支持 `on_reject: abort` 和重新开始

#### Spec 模板结构

```markdown
# Feature Specification
## User Scenarios & Testing (mandatory)
  ### User Story 1 - [Title] (Priority: P1)
    - Why this priority
    - Independent Test
    - Acceptance Scenarios (Given/When/Then)
## Requirements (mandatory)
  ### Functional Requirements (FR-001, FR-002...)
  ### Key Entities
## Success Criteria (mandatory)
  ### Measurable Outcomes (SC-001, SC-002...)
## Assumptions
```

#### 亮点

1. **GitHub 官方背书**——与 GitHub Copilot 深度集成
2. **声明式工作流**——YAML 定义步骤和门控，清晰可改
3. **Constitution 机制**——项目级"宪法"定义质量标准
4. **模板驱动**——规范、计划、任务都有标准模板
5. **Need Clarification 标记**——不确定的需求显式标记 `[NEEDS CLARIFICATION: ...]`
6. **独立可测试的 User Story**——每个 story 必须能独立测试、独立交付价值

#### 不足

- 更偏"规范生成工具"而非"开发方法论"
- 没有角色概念
- 没有自演进

### 4. OpenSpec（⭐ 46.4K）

**仓库：** https://github.com/Fission-AI/OpenSpec
**定位：** 开放的 Spec 工作流框架，强调"fluid not rigid"

#### 核心理念

**非线性工作流。** OpenSpec 认为真实工作不是线性的——你会实现到一半发现设计有问题，需要回头改 spec。所以它提供的是"actions"（随时可以执行的动作），而不是"phases"（必须按顺序走的阶段）。

#### OPSX 工作流

```
/opsx:propose → specs/ + design.md + tasks.md
/opsx:apply   → 自动实现
/opsx:archive → 归档到历史
```

**关键创新：**
- **Action-based，不是 Phase-based**——create、implement、update、archive 随时可用
- **依赖是 enabler，不是约束**——依赖关系告诉你"什么可能"，不强制"下一步"
- **Schema.yaml + templates/*.md 可定制**——任何人可以改 prompt 模板，即时生效

#### 目录结构

```
OpenSpec/
├── src/
│   ├── commands/          # CLI 命令
│   ├── core/              # 核心逻辑
│   ├── prompts/           # Prompt 模板
│   └── ui/                # 仪表盘 UI
├── openspec/
│   └── specs/             # 自身用 OpenSpec 管理
├── schemas/               # JSON Schema
└── docs/
```

#### 亮点

1. **非线性工作流**——最贴近真实开发节奏的设计
2. **模板可定制**——改 prompt 模板立即生效，不需要发布新版本
3. **仪表盘 UI**——可视化追踪所有 spec 和变更
4. **`config.yaml` 项目配置**——注入技术栈上下文到所有产出
5. **"fluid not rigid"哲学**——整个设计围绕"工作不是线性的"这个认知

#### 不足

- TypeScript 实现，工具链较重
- 仍在早期，很多功能在重构中
- 不如 Superpowers 的 skill 生态成熟

### 5. Get Shit Done / GSD（⭐ 61K）

**仓库：** https://github.com/gsd-build/get-shit-done
**定位：** 轻量级上下文工程系统，解决"context rot"问题

#### 核心理念

**上下文质量管理。** GSD 的核心洞见是 AI 编程质量下降的根因是"上下文腐烂"——随着对话进行，AI 的 context window 被填满，质量从 PEAK → GOOD → DEGRADING → POOR 逐步下降。

**解决方案：** 把工作拆成小粒度的 PLAN.md（每个消耗 10-30% context），每个 PLAN 由独立 agent 执行。

#### 目录结构

```
get-shit-done/
├── agents/                    # 33 个专业 agent 定义
│   ├── gsd-planner.md         # 计划器（最复杂，~500 行）
│   ├── gsd-executor.md        # 执行器
│   ├── gsd-verifier.md        # 验证器
│   ├── gsd-code-reviewer.md   # 代码评审
│   ├── gsd-debugger.md        # 调试器
│   ├── gsd-research-synthesizer.md
│   └── ...                    # 还有 26 个
├── get-shit-done/             # 工作流和引用文件
├── hooks/                     # Git hooks
├── sdk/                       # GSD SDK
└── tests/                     # 测试
```

#### 工作流设计

```
/gsd-new-project  → 初始化（问题 → 研究 → 需求 → 路线图）
/gsd-discuss-phase → 讨论（用户确认决策）
/gsd-plan-phase   → 计划（生成 PLAN.md）
/gsd-execute-phase → 执行（subagent 执行每个 PLAN）
/gsd-verify-phase → 验证（自动验证 + 人工确认）
```

#### Agent 定义结构（以 gsd-planner 为例）

极其详细，包含：
- `<role>` — 角色定义和核心职责
- `<context_fidelity>` — 用户决策保真（锁定决策必须执行，延迟想法必须排除）
- `<scope_reduction_prohibition>` — 禁止简化用户决策
- `<planner_authority_limits>` — 规划者不能判断"太难"
- `<philosophy>` — 独行侠 + Claude 的工作模式
- `<task_breakdown>` — 任务拆分规则（每个 PLAN 2-3 个 task）
- `<dependency_graph>` — 依赖图构建
- `<goal_backward>` — 目标倒推方法论
- `<checkpoints>` — 检查点类型和格式
- `<tdd_integration>` — TDD 集成
- `<execution_flow>` — 完整执行流程（20+ 个步骤）

#### 上下文质量曲线

| Context Usage | Quality | 状态 |
|---------------|---------|------|
| 0-30% | PEAK | 全面、细致 |
| 30-50% | GOOD | 自信、扎实 |
| 50-70% | DEGRADING | 开始走捷径 |
| 70%+ | POOR | 敷衍了事 |

**核心规则：每个 PLAN 必须在 ~50% context 内完成。**

#### 亮点

1. **上下文质量管理是核心创新**——其他框架都没解决这个问题
2. **Goal-Backward 方法论**——从目标倒推必须满足的 truth、必须存在的 artifact、必须连接的 wiring
3. **Wave-based 并行**——按依赖图分组，无依赖的 PLAN 可以并行执行
4. **Interface-First**——先定义接口合约，再实现
5. **极度详细的 Agent 定义**——gsd-planner.md 有 500+ 行，覆盖几乎所有边界情况
6. **33 个专业 agent**——从研究员到安全审计员，每个有独立职责
7. **决策保真机制**——锁定决策不可违反，延迟想法不可混入
8. **CONTEXT.md 全局上下文**——所有决策和术语定义集中管理

#### 不足

- 复杂度极高——学习成本可能比 BMAD 还高
- 33 个 agent 有过度设计之嫌
- 依赖 Claude Code 的 subagent 能力

### 6. gstack（⭐ 91.8K）

**仓库：** https://github.com/garrytan/gstack
**作者：** Garry Tan (YC/Initialized Capital)
**定位：** AI 工程工作流，包含浏览器自动化、代码评审、设计系统

#### 核心理念

**技能集合而非方法论。** gstack 不定义一个"从 A 到 Z 的完整流程"，而是提供一组独立的技能，用户按需调用。

#### 技能分类

| 类别 | 技能 |
|------|------|
| 规划评审 | office-hours, plan-ceo-review, plan-eng-review, plan-design-review, plan-devex-review, autoplan |
| 实现评审 | review, codex, investigate, design-review, design-shotgun, qa |
| 发布部署 | ship, land-and-deploy, canary, document-release |
| 运维记忆 | context-save, context-restore, learn, retro, health |
| 安全 | cso (OWASP Top 10 + STRIDE) |

#### 亮点

1. **CEO 级别评审**——从产品视角审视需求，找"10-star product"
2. **多维度评审**——CEO + 设计 + 工程 + DX，每个维度独立评分
3. **Autoplan**——一条命令跑完 CEO → 设计 → 工程 → DX 评审
4. **跨会话记忆**——context-save/context-restore 支持跨会话恢复
5. **Learnings 系统**——跨会话学习，用 JSONL 存储经验
6. **浏览器自动化 QA**——用真实浏览器做 QA，比静态分析更有说服力
7. **安全审计（CSO）**——OWASP Top 10 + STRIDE，专业级安全检查

#### 不足

- 更像"工具箱"而非"方法论"——没有明确的端到端流程
- 依赖 Claude Code 的 skill 系统
- 复杂度高，单个 SKILL.md 就有数千行

### 7. specs.md（⭐ 142）

**仓库：** https://github.com/fabriqaai/specs.md
**定位：** 多流程可选的 Spec 框架

#### 核心理念

**不同场景用不同流程。** specs.md 提供三种流程：

| 流程 | 定位 | Agent 数 | 检查点 |
|------|------|----------|--------|
| Simple | 只生成 spec | 1 | 3（阶段门控） |
| FIRE | 自适应执行 | 3 | 0-2（按复杂度） |
| AI-DLC | 完整方法论 + DDD | 4 | 全覆盖 |

#### FIRE 流程设计

```
/fire-orchestrator   → 入口路由
/fire-planner        → 意图捕获 + 工作项拆分
/fire-builder        → 执行 + 自动生成 walkthrough
```

**关键创新：自适应检查点**——根据任务复杂度决定是否需要人工确认（0、1 或 2 个检查点），而不是固定检查点。

#### 亮点

1. **流程可选**——不像其他框架只提供一种工作流
2. **FIRE 的自适应检查点**——简单任务自动跑，复杂任务才暂停确认
3. **Walkthrough 自动生成**——每个变更自动文档化
4. **VS Code Extension**——可视化追踪进度
5. **Brownfield 友好**——自动检测已有项目的模式和约定

## 五、对我们框架的启示

### 5.1 高优先级借鉴（直接影响 v0.3 设计）

#### 1. 引入"门控"机制

**来源：** Superpowers (HARD-GATE) + SpecKit (review-spec gate)

**现状：** v0.3 有"检查点"概念但比较软——"建议系统提示人确认"。

**建议：** 在 workflow 文件中支持硬门控定义：

```yaml
# workflows/feature.md 中
checkpoints:
  plan:
    type: hard_gate  # 必须人工确认，不通过不能继续
    message: "请确认架构方案后再进入开发"
  review:
    type: hard_gate
    message: "请确认 Review 结果"
```

#### 2. 引入 Goal-Backward 方法论

**来源：** GSD

**现状：** v0.3 的 Plan 阶段只说"输出技术方案和可执行步骤"。

**建议：** 在 Architect 角色的"方案设计"能力中加入 Goal-Backward 方法：
1. 目标是什么（outcome，不是 task）
2. 目标成立必须满足哪些 truth
3. 每个 truth 需要哪些 artifact
4. 每个 artifact 需要哪些 wiring

这让 Plan 阶段的产出质量可验证、可追溯。

#### 3. 引入自审核循环

**来源：** Superpowers

**现状：** v0.3 没有产出质量自检机制。

**建议：** 在每个阶段产出后加一个轻量自检：
- Placeholder scan：有没有 TBD/TODO/待定？
- 一致性检查：方案前后有没有矛盾？
- Scope 检查：是不是太大了需要拆分？

#### 4. 引入 Need Clarification 标记

**来源：** SpecKit

**现状：** v0.3 没有处理不确定性的机制。

**建议：** 在 Analyst 的"澄清"能力和 Architect 的"方案设计"能力中，遇到不确定的点必须显式标记 `[待澄清: ...]`，不允许用模糊表述混过去。

### 5.2 中优先级借鉴（v0.4+ 考虑）

#### 5. 上下文质量管理

**来源：** GSD（核心创新）

**启示：** 我们框架目前没有考虑 context window 的管理。GSD 的洞见是：**每个阶段/任务的产出应该在 50% context 以内完成**。我们可以在 workflow 文件中为每个阶段设定建议的 context 预算。

#### 6. 声明式工作流定义

**来源：** SpecKit (workflow.yml) + OpenSpec (schema.yaml)

**现状：** v0.3 的 workflow 是 Markdown 文件。

**建议：** v0.4 可以考虑引入 YAML 格式的 workflow 定义，更结构化、更容易解析和验证。

#### 7. 三层配置合并

**来源：** BMAD

**启示：** v0.3 已有项目层/个人层分离，但缺少"团队层"和精细的合并规则。BMAD 的 base → team → user 三层合并值得参考。

### 5.3 长期借鉴（v0.5+）

#### 8. Action-based 工作流

**来源：** OpenSpec

**启示：** 当前的工作流还是线性的（Brainstorm → Plan → Execute → Review）。OpenSpec 的 Action-based 思路更贴合实际——未来可以支持"随时 update spec"、"随时回退到 Plan"等非线性操作。

#### 9. Interface-First 任务排序

**来源：** GSD

**启示：** Execute 阶段的任务排序应该遵循"先定义接口合约，再实现"的原则，避免"scavenger hunt"问题。

#### 10. 跨会话学习和记忆

**来源：** gstack (learnings) + GSD (learnings)

**启示：** 自演进机制可以参考 gstack 的 JSONL 存储方式，轻量且可追溯。

## 六、总结：我们框架的差异化定位

| 维度 | 主流框架 | 我们的框架 |
|------|---------|-----------|
| 核心创新点 | 各有侧重 | **工作流驱动 + 角色能力的双层解耦** |
| 工作流 vs 角色 | 要么工作流要么角色 | **两者结合** |
| 信息流转 | 文件传递（Superpowers/BMAD） | **对话上下文为主 + 文件快照为辅** |
| 用户心智模型 | "选角色"或"走流程" | **"在哪个阶段"——最直觉** |
| 上下文管理 | GSD 独有 | 需要补充 |
| 门控机制 | Superpowers/SpecKit 有 | 需要加强 |
| 自演进 | gstack/GSD 有（轻量） | 已规划（渐进式） |

**核心差异化：** 我们是唯一一个把"工作流阶段"和"角色能力"明确解耦的框架。Superpowers 纯阶段、BMAD 纯角色、GSD 重上下文——我们取三者之长，用一个干净的双层架构组合起来。
