## 二、Step 0：输入质量评估

### 2.1 概述

在进入正式评审流程前，先评估用户输入的质量等级，并推荐最合适的处理路径。Developer 可以接受推荐路径或自行选择。

### 2.2 输入

| 输入项 | 类型 | 说明 |
|--------|------|------|
| 用户原始输入 | 文本 / 文件 / URL | 任意形态的需求描述 |

### 2.3 处理逻辑

#### 2.3.1 质量等级判定

根据以下维度评估输入质量：

| 维度 | 评估标准 | 权重示意 |
|------|---------|---------|
| **完整性** | 是否包含目标、用户、场景、约束 | 高 |
| **具体性** | 是否有具体的用户角色、操作、预期结果 | 高 |
| **结构化程度** | 是否有标题、分节、列表等结构 | 中 |
| **可验证性** | 是否包含验收标准或可推断的验收条件 | 中 |
| **上下文丰富度** | 是否包含背景、关联系统、历史决策 | 低 |

#### 2.3.2 质量分级与推荐路径

引用设计决策 Q5 的分级标准：

| 质量等级 | 特征 | 推荐路径 | 交互模式 |
|---------|------|---------|---------|
| **极差** | 一句话，无任何细节（如"帮我加个导出功能"） | 一次一问，逐个澄清 | Coaching Mode |
| **较差** | 有基本描述但缺关键细节（如标准 Issue） | 先读代码库理解上下文，再针对性提问 | Mixed Mode |
| **中等** | 标准 PRD，有目标、用户、场景 | 批量处理 + NEEDS CLARIFICATION | Batch Mode |
| **极好** | 完整技术设计文档，有方案、边界、约束 | 直接生成 Spec，Self-Review 后确认 | Fast Path |

#### 2.3.3 处理流程

```
1. 读取用户输入
2. 按 5 个维度评分（定性：强/中/弱/无）
3. 综合判定质量等级
4. 给出推荐路径 + 理由
5. 等待 Developer 确认或选择其他路径
```

### 2.4 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 质量等级 | 文本 | 极差 / 较差 / 中等 / 极好 |
| 推荐路径 | 文本 + 理由 | 推荐的交互模式和理由 |
| 缺失项列表 | 列表 | 输入中缺少的关键信息 |

### 2.5 失败处理

| 场景 | 处理方式 |
|------|---------|
| 输入为空 | 提示 Developer 提供需求描述 |
| 输入完全不相关 | 指出输入与需求无关，请 Developer 确认意图 |
| 无法判定质量 | 默认按"较差"处理，走 Mixed Mode |

### 2.6 示例

**输入：** "帮我加个导出功能"

**输出：**
```
📊 输入质量评估

**质量等级：** 极差
**推荐路径：** Coaching Mode（一次一问，逐个澄清）

**理由：**
- 完整性：无 — 没有目标、用户、场景
- 具体性：无 — 没有具体的操作描述
- 结构化程度：无 — 一句话，无结构
- 可验证性：无 — 没有验收标准
- 上下文丰富度：无 — 没有背景信息

**缺失项：**
- 导出什么？（哪个模块、什么数据）
- 谁来导出？（用户角色）
- 导出成什么格式？
- 有没有数量限制？
- 有没有权限要求？

> 是否接受推荐路径？或者你可以先补充一些信息再开始。
```

## 三、Step 1：Context Gathering

### 3.1 概述

在进入需求理解之前，先收集项目上下文信息。这确保后续的提问和建议扎根于项目实际情况，而不是凭空猜测。引用 v0.4 原则 3（统一语言基础设施）和 gsd 的"先侦察代码库，再问问题"。

### 3.2 输入

| 输入项 | 类型 | 说明 |
|--------|------|------|
| 用户原始输入 | 文本 | 来自 Step 0 |
| 质量评估结果 | 结构化数据 | 来自 Step 0 |
| 项目根目录路径 | 路径 | 用于读取项目文件 |

### 3.3 处理逻辑

#### 3.3.1 读取顺序

按以下优先级收集上下文：

```
1. CONTEXT.md           — 领域词汇表（如果存在）
2. .agents/config.yaml  — 项目配置（如果存在）
3. README.md             — 项目概览（如果存在）
4. 相关源代码目录        — 理解现有实现（根据需求关键词定向读取）
5. 已有的 Spec 文件      — 避免重复工作（如果存在）
```

#### 3.3.2 上下文提取

从以上文件中提取：

| 信息 | 来源 | 用途 |
|------|------|------|
| 领域术语 | CONTEXT.md | 确保提问用词准确 |
| 技术栈 | config.yaml / README | 理解技术约束 |
| 已有功能 | 源代码 / 已有 Spec | 避免重复，识别关联 |
| 项目结构 | 目录结构 | 理解模块边界 |

#### 3.3.3 上下文预算

引用 v0.4 §6.5 上下文预算管理：

- Context Gathering 建议消耗 < 10% 上下文
- 如果项目过大，只读取与需求直接相关的模块
- 记录读取的文件列表，便于审计

### 3.4 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 项目上下文摘要 | 内部状态 | 不直接输出给用户，供后续步骤使用 |
| 术语列表 | 列表 | 从 CONTEXT.md 加载的已有术语 |
| 关联模块 | 列表 | 与需求相关的现有模块 |
| 已有决策 | 列表 | 与需求相关的已有设计决策 |

### 3.5 失败处理

| 场景 | 处理方式 |
|------|---------|
| 项目无 CONTEXT.md | 跳过术语加载，在 Step 2 中发现新术语时创建 |
| 无 README | 跳过，从目录结构推断 |
| 源代码过大 | 只读取入口文件和与需求关键词匹配的文件 |
| 完全无法获取上下文 | 继续 Step 2，在提问中额外收集项目信息 |

### 3.6 示例

**输入：** 需求"为订单模块新增批量导出功能"

**内部处理（不直接输出）：**

```
已加载上下文：
- CONTEXT.md: 找到术语 Order, LineItem, Customer, Invoice
- config.yaml: 技术栈 [typescript, react, node, mysql]
- 源代码: 找到 src/modules/orders/ 目录
- 关联模块: orders.controller.ts, orders.service.ts, orders.model.ts
- 已有 Spec: 无相关的活跃 Spec

术语预加载：
- Order: 客户提交的购买请求
- Customer: 下单的客户
- 注意：项目中 "account" 一词曾引发歧义（已解决）

关联代码结构：
- GET /api/orders — 列表查询
- GET /api/orders/:id — 单条查询
- 无导出相关代码
```

## 四、Step 2：Requirement Understanding

### 4.1 概述

核心交互步骤。根据输入质量和项目上下文，通过混合自适应的交互方式，逐个澄清需求中的模糊点。引用设计决策 Q1（混合自适应模式）和 Q4（Coach 风格 + 反谄媚规则）。

### 4.2 输入

| 输入项 | 类型 | 说明 |
|--------|------|------|
| 用户原始输入 | 文本 | 来自 Step 0 |
| 质量评估结果 | 结构化数据 | 来自 Step 0 的质量等级和推荐路径 |
| 项目上下文 | 内部状态 | 来自 Step 1 |

### 4.3 处理逻辑

#### 4.3.1 理解层 6 问

基于 v0.4 §4.2 阶段 1 的六维度追问（改编自 gstack Six Forcing Questions）：

| # | 维度 | 核心问题 | 要澄清什么 |
|---|------|---------|-----------|
| 1 | **Who** | 具体谁会用这个功能？ | 用户角色和使用场景 |
| 2 | **What** | 现在怎么解决的？替代方案的成本？ | 现状痛点和替代方案 |
| 3 | **What+** | 如果只能做一个功能，做什么？ | 最小可用版本 |
| 4 | **How** | 怎么判断做完了？ | 验收标准 |
| 5 | **Where** | 什么边界情况会破坏？ | 边界条件和异常场景 |
| 6 | **With** | 和已有的什么功能/需求有关联？ | 依赖和冲突 |

**不是每个需求都要问完 6 个问题。** 根据输入质量选择子集：

| 输入质量 | 提问策略 |
|---------|---------|
| 极差 | 完整走 6 问，一次一个问题 |
| 较差 | 跳过已明确的，聚焦缺失和模糊的 |
| 中等 | 批量提出观察，只对 NEEDS CLARIFICATION 点单独追问 |
| 极好 | 不提问，直接进入 Step 3，在 Spec 中标记观察 |

#### 4.3.2 提问策略（根据模糊点数量决定）

AI 先分析需求，找出所有需要澄清的模糊点，然后根据模糊点数量决定怎么提问：

| 模糊点数量 | 提问策略 | 做法 |
|-----------|---------|------|
| **0 个** | 不提问 | 所有需求点都 ✅ 清晰，直接进入 Step 3 |
| **1-3 个** | 一次性提问 | 把所有问题列出来，Developer 一次回答 |
| **4-6 个** | 分轮提问 | 分 2-3 轮，每轮 2-3 个问题 |
| **>6 个** | 逐个提问 | 一次问 1 个问题，逐个澄清，最多 6 轮 |

**一轮提问的流程：**

```
2a. AI 分析需求，找出所有模糊点
    ↓
2b. 展示分析结果 + 提问策略说明（用户可见）
    ↓
2c. 根据模糊点数量选择提问策略（上表）
    ↓
2d. 向 Developer 提问
    ↓
2e. Developer 回答（可以跳过某个问题，标记为 [NEEDS CLARIFICATION]）
    ↓
2f. AI 更新理解，检查是否有新的模糊点
    ├── 有新问题 → 回到 2b（展示新的分析结果）
    └── 没有新问题 → 进入 Step 3
```

**2b 分析结果展示格式（每次分支判断时都要给用户可见的交代）：**

```
📋 需求分析完成

我分析了你的需求，发现：
- ✅ N 个需求点理解清晰
- ⚠️ N 个需求点需要澄清
- ❌ N 个需求点和已有信息冲突
- 🔍 N 个方面信息完全缺失

[说明为什么选择这种提问策略，例如：]
问题不多，我一次性列出来。

Q1: ...
Q2: ...
```

或者模糊点多的时候：

```
📋 需求分析完成

我分析了你的需求，发现：
- ✅ 1 个需求点理解清晰
- ⚠️ 5 个需求点需要澄清
- ❌ 1 个需求点和现有代码的 export-service.ts 冲突

问题比较多，我们逐个来。先从最关键的开始：

Q1: ...
```

**提问规则：**
1. 每个问题必须具体（不给开放式"你觉得呢"）
2. 提问时提供上下文（"我看了 orders.controller.ts，发现..."）
3. Developer 可以要求跳过某个问题（标记为 [NEEDS CLARIFICATION]，不阻塞流程）
4. 对每个回答给出四种状态标注之一（✅⚠️❌🔍）
5. Developer 回答后，检查是否引出了新的模糊点
6. 如果有新的模糊点，根据新增数量决定下一轮策略

#### 4.3.3 状态标注规则

引用设计决策 Q4 的四种状态标注：

| 状态 | 含义 | AI 动作 |
|------|------|---------|
| ✅ **清晰** | 理解到位，可以直接写入 Spec | 记录到 Spec 草稿 |
| ⚠️ **模糊** | 理解不完全，有歧义 | 标记 [NEEDS CLARIFICATION] + 说明缺什么 + 为什么影响决策 |
| ❌ **矛盾** | 和已有信息冲突（代码、CONTEXT、其他需求） | 指出冲突点 + 两方的信息来源 + 要求 Developer 裁决 |
| 🔍 **缺失** | 完全没有信息 | 提出具体问题 + 建议默认值（Developer 可以接受默认值或补充） |

#### 4.3.4 Developer 熟悉度适配

引用设计决策 Q5 的额外场景：

| Developer 状态 | 适配策略 |
|---------------|---------|
| 不熟悉业务 | 多提供背景解释，问题更具体（给选项而非开放式提问） |
| 很熟悉业务 | 跳过基础问题，直接聚焦边界条件和异常场景 |
| 跨团队需求 | 标记跨团队依赖，建议先和关联团队对齐 |

### 4.4 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 需求理解记录 | 内部状态 | 包含所有已澄清和未澄清的需求点 |
| NEEDS CLARIFICATION 列表 | 列表 | 所有需要 Developer 进一步确认的点 |
| 澄清记录 | 表格 | Q&A 过程的审计轨迹 |
| 术语变更 | 列表 | 新发现的术语或与 CONTEXT.md 冲突的术语 |

### 4.5 失败处理

| 场景 | 处理方式 |
|------|---------|
| Developer 反复跳过问题 | 记录为 [NEEDS CLARIFICATION]，AI 建议一个保守默认值 |
| Developer 给出矛盾回答 | 指出矛盾，要求裁决，不自行选择 |
| 长时间无法澄清核心问题 | 建议 Developer 先和利益相关者对齐后再继续 |
| Developer 中途改变需求 | 重新评估输入质量，回到 Step 0 |

### 4.6 示例

**示例 1：Coaching Mode（极差输入）**

```
👤 Developer: 帮我加个导出功能

🤖 AI: 
我看了项目的订单模块代码。目前在 orders.controller.ts 中有列表查询和单条查询接口，没有导出相关功能。

**第一个问题：** 你说的"导出"，具体是导出什么数据？
- A. 订单列表（Order）
- B. 发票（Invoice）
- C. 其他（请说明）

> 我从 CONTEXT.md 中看到项目里 Order 和 Invoice 是两个不同的概念，想确认一下。
```

**示例 2：Batch Mode（中等输入）**

```
👤 Developer: [提交了一份标准 PRD，包含导出功能的目标、用户、场景]

🤖 AI:
我已经阅读了 PRD 文档。以下是我的理解：

**✅ 已明确的部分：**
- 目标用户：客服人员
- 核心场景：按条件筛选订单后导出 Excel
- 文件格式：.xlsx，固定 5 列
- 数量限制：10000 条

**⚠️ 需要澄清的部分：**
1. PRD 提到"权限控制"，但没有说明具体规则。**影响决策：** 决定是否需要新增权限接口。[NEEDS CLARIFICATION]
2. PRD 没有提到并发导出的处理方式。**影响决策：** 是否需要排队机制或并发限制。[NEEDS CLARIFICATION]

**🔍 缺失的信息：**
1. 导出失败时的重试策略（我建议：不重试，提示用户重新操作）

你可以批量回答，也可以逐个讨论。
```

**示例 3：Fast Path（极好输入）**

```
👤 Developer: [提交了完整的技术设计文档，包含方案、接口定义、边界条件]

🤖 AI:
输入质量极好，我直接生成 Spec，Self-Review 后给你看。如果有不确定的地方，我会在 Spec 中标记。
```

## 五、Step 3：Spec Generation

### 5.1 概述

基于 Step 2 的理解结果，按 Spec 模板生成 Requirement Review Spec，并自动执行 Self-Review。引用设计决策 Q2（必填 7 项 + 可选 4 项）和 Q6（禁止猜测 + Self-Review）。

### 5.2 输入

| 输入项 | 类型 | 说明 |
|--------|------|------|
| 需求理解记录 | 内部状态 | 来自 Step 2 |
| 澄清记录 | 表格 | 来自 Step 2 |
| 项目上下文 | 内部状态 | 来自 Step 1 |
| Spec 模板 | 文件 | Requirement Review Spec 模板 |

### 5.3 处理逻辑

#### 5.3.1 填充顺序

按以下顺序填充 Spec 模板（从确定性高到确定性低）：

```
1. 需求来源          — 直接从输入获取
2. 需求描述          — 基于 Step 2 的理解重组
3. 功能需求          — 拆分为 FR + 三元组
4. 验收标准          — 从功能需求的 Acceptance 提取
5. 边界定义          — 基于 Step 2 的理解划定
6. 澄清记录          — 从 Step 2 的 Q&A 记录
7. 术语定义（可选）   — 新发现的术语
8. 约束条件（可选）   — 从输入中提取的硬约束
9. 依赖关系（可选）   — 从 Step 1 上下文和 Step 2 澄清中提取
10. 风险识别（可选）  — 基于理解结果识别
11. 自审核检查        — 最后执行
```

#### 5.3.2 禁止猜测规则

引用设计决策 Q6 的第一层防护：

```
规则：
1. 不确定的内容 → 标记 [NEEDS CLARIFICATION: 影响什么决策]
2. 不编造验收标准
3. 不猜测技术约束
4. 不假设用户意图（除非给出默认值并标注为"AI 建议"）
5. 引用代码库信息时，标注来源文件
```

#### 5.3.3 Self-Review 流程

引用设计决策 Q6 的第三层防护（Q3 的 6 项自审核）：

```
Spec 生成后，自动执行以下检查：

1. 问题定义清晰？
   - 需求描述是否明确回答了"做什么"？
   - 是否避免了模糊语言（"尽快"、"比较好"、"灵活"）？

2. 范围已界定？
   - In Scope 和 Out of Scope 是否都有内容？
   - Out of Scope 是否明确列出了"可能被误以为要做的"？

3. 验收标准已列出？
   - 每个 FR 是否都有对应的 AC？
   - 每个 AC 是否可证伪（Pass/Fail）？
   - AC 是否避免了实现细节？

4. 关键术语已定义？
   - 新引入的术语是否写入术语定义？
   - 是否与 CONTEXT.md 中的已有术语冲突？

5. 无 placeholder/TBD 残留？
   - 全文扫描 TBD / TODO / 待定 / ?? / 待确认 / TBC
   - 如果有 → 移除并替换为 [NEEDS CLARIFICATION] 或实际内容

6. 内部一致性？
   - 功能需求中的 AC 是否与验收标准章节一一对应？
   - 需求描述是否与功能需求一致？
   - 边界定义是否与功能需求覆盖范围一致？
   - 澄清记录中的回答是否体现在 Spec 正文中？
```

#### 5.3.4 Self-Review 结果处理

| 结果 | 处理方式 |
|------|---------|
| 全部通过 | 进入 Step 4 |
| 有不通过项（可自动修复） | 自动修复后重新检查 |
| 有不通过项（需 Developer 输入） | 标记问题，进入 Step 4 时一并展示 |

### 5.4 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| Requirement Spec | Markdown 文件 | `.agents/active/{task-name}/requirement-spec.md` |
| Self-Review 报告 | 内嵌于 Spec | 自审核检查章节的勾选结果 |
| NEEDS CLARIFICATION 汇总 | 列表 | 所有未解决的标记（如果有） |

### 5.5 失败处理

| 场景 | 处理方式 |
|------|---------|
| 无法生成某个必填字段 | 标记 [NEEDS CLARIFICATION]，在自审核中标注 |
| Self-Review 发现大量不一致 | 回到 Step 2 补充澄清 |
| Spec 过长（超过合理范围） | 建议拆分为多个 Spec |

## 六、Step 4：User Review Gate

### 6.1 概述

最终门控。AI 完成 Self-Review 后，将完整 Spec 呈现给 Developer，由 Developer 做最终确认。引用设计决策 Q3（6 项定性 checklist + 人工确认）和 Q6（第四层防护：Developer 最终确认）。

### 6.2 输入

| 输入项 | 类型 | 说明 |
|--------|------|------|
| Requirement Spec | Markdown 文件 | 来自 Step 3 |
| Self-Review 结果 | 内嵌于 Spec | 来自 Step 3 |
| NEEDS CLARIFICATION 汇总 | 列表 | 来自 Step 3 |

### 6.3 处理逻辑

#### 6.3.1 展示策略

向 Developer 展示以下信息：

```
1. Spec 概要（关键内容摘要，不是全文）
   - 需求一句话描述
   - 功能需求数量和列表
   - 验收标准数量
   - In Scope / Out of Scope 摘要

2. 自审核结果
   - 6 项检查的通过/不通过状态
   - 不通过项的具体说明

3. NEEDS CLARIFICATION 列表（如果有）
   - 每个标记的问题 + 影响的决策 + AI 建议默认值

4. 术语变更（如果有）
   - 新增的术语
   - 与 CONTEXT.md 冲突的术语
```

#### 6.3.2 门控检查

**必须通过的条件：**

```
1. 自审核 6 项全部通过
   - 如果有不通过项 → 返回 Step 3 修复
   - Developer 不能跳过自审核不通过项

2. Developer 确认
   - Developer 阅读完整 Spec 后勾选确认
   - 这是人主控原则的最终体现

3. NEEDS CLARIFICATION 处理（软性）
   - Developer 可以选择：
     a. 补充信息解决标记
     b. 接受 AI 建议的默认值
     c. 标记为"后续解决"并进入 Plan（但有风险提示）
```

#### 6.3.3 Placeholder 扫描

引用设计决策 Q6 的第二层防护：

```
在 Developer 确认前，执行最终的 Placeholder 扫描：
- 扫描关键词：TBD, TODO, 待定, ??, 待确认, TBC, PLACEHOLDER, FIXME
- 如果发现 → 阻止通过，要求处理
- [NEEDS CLARIFICATION] 不是 placeholder，不阻止通过
```

#### 6.3.4 内部一致性检查

引用设计决策 Q3 的第 6 项（内部一致性）：

```
检查项目：
1. 每个 FR 是否都有对应的 AC？（FR-AC 一致性）
2. 验收标准章节的 AC 是否与 FR 中的 AC 一致？（AC-AC 一致性）
3. 需求描述是否涵盖了所有 FR？（描述-FR 一致性）
4. 边界定义的 In Scope 是否与 FR 覆盖范围一致？（边界-FR 一致性）
5. 澄清记录中的回答是否体现在 Spec 正文中？（澄清-正文一致性）
6. 术语定义是否在正文中被正确使用？（术语-正文一致性）
```

### 6.4 输出

| 输出项 | 格式 | 说明 |
|--------|------|------|
| 最终 Requirement Spec | Markdown 文件 | Developer 确认后的版本 |
| 门控通过记录 | 结构化数据 | 记录到 status.yaml |
| 术语更新 | 写入 CONTEXT.md | 新发现的术语同步到领域词汇表 |
| 结构化日志 | JSONL | 记录到 log.jsonl |

### 6.5 失败处理

| 场景 | 处理方式 |
|------|---------|
| Developer 拒绝 | 收集拒绝原因，返回 Step 2 或 Step 3 |
| 自审核不通过 | 返回 Step 3 修复 |
| Placeholder 扫描发现残留 | 返回 Step 3 修复 |
| Developer 要求修改 | 根据修改范围返回 Step 2（大改）或 Step 3（小改） |
| Developer 要求增加新需求 | 评估影响范围，必要时回到 Step 0 |

### 6.6 示例

**门控通过：**

```
📋 Requirement Spec 门控检查

**自审核结果：**
✅ 问题定义清晰
✅ 范围已界定
✅ 验收标准已列出
✅ 关键术语已定义
✅ 无 placeholder/TBD 残留
✅ 内部一致性

**NEEDS CLARIFICATION（1 项待决定）：**

  影响决策：是否需要新增权限接口
  AI 建议默认值：MVP 阶段不增加权限控制，所有登录用户均可导出

**Placeholder 扫描：** 通过（无残留）

**请确认：**
1. 阅读完整 Spec（已展示在下方）
```
