# Spec 模板详细设计

> 版本：1.0

> 状态：详细设计
> 关联决策：Q2（必填 7 项 + 可选 4 项 + NEEDS CLARIFICATION 不设上限）

---

## 一、设计原则

### 1.1 字段分类依据（引用设计决策 Q2）

| 分类 | 数量 | 说明 |
|------|------|------|
| **必填字段** | 7 项 | 缺少任何一项则 Spec 不通过门控 |
| **可选字段** | 4 项 | 有信息时填写，无信息时省略（不写 TBD） |
| **机制字段** | NEEDS CLARIFICATION | 不设数量上限，有多少标多少 |

### 1.2 核心设计理念

1. **模板是可执行流程**（引用 v0.4 原则 2.5）：填完就知道下一步做什么
2. **可证伪的验收标准**（引用 gsd 三元组）：每个需求有现状 → 目标 → 验收标准
3. **禁止占位符**（引用设计决策 Q6）：不写 TBD/TODO/待定，不确定的标 NEEDS CLARIFICATION
4. **需求来源可追溯**（引用 bmad Decision Log）：记录每个信息点的来源

## 二、必填字段详细设计

### 字段 1：需求来源（Requirement Source）

| 属性 | 内容 |
|------|------|
| **字段名称** | 需求来源 / Requirement Source |
| **说明** | 记录需求的输入类型和来源链接，便于回溯到原始需求。这个字段回答"这个需求从哪来的"，确保后续阶段可以追溯信息源头。 |
| **格式要求** | 包含两个子项：输入类型（枚举值）和来源链接/引用。用列表格式。 |
| **示例** | 见下方 |
| **注意事项** | 输入类型必须从枚举中选择，不要自创。如果来源是口头沟通，记录沟通对象和时间。 |

**示例：**

```markdown
## 需求来源
- **输入类型**: Issue
- **来源链接**: https://github.com/org/repo/issues/42
- **补充材料**: 2026-05-16 产品经理张伟口头补充的边界条件
```

**输入类型枚举：**

| 值 | 说明 |
|----|------|
| `完整 PRD` | 产品需求文档，包含背景、目标、用户故事、验收标准 |
| `PRD 草稿` | 未完成的 PRD，需要补充 |
| `Issue` | Bug 报告或功能请求 |
| `口头描述` | 面对面/即时通讯中的一句话需求 |
| `邮件/消息` | 通过邮件或即时通讯发送的需求描述 |
| `竞品分析` | 基于竞品功能的对标需求 |
| `用户反馈` | 来自用户/客户的反馈 |
| `技术债` | 从代码审查或技术评估中发现的需求 |

### 字段 2：需求描述（Requirement Description）

| 属性 | 内容 |
|------|------|
| **字段名称** | 需求描述 / Requirement Description |
| **说明** | 经过澄清后的完整需求描述。不是原始需求的复制粘贴，而是评审后重新组织的清晰陈述。这个字段回答"到底要做什么"。 |
| **格式要求** | 1-3 段 Markdown 文本。先写一句话总结（粗体），再展开细节。如果涉及多个子需求，用编号列表。 |
| **示例** | 见下方 |
| **注意事项** | 不要复制粘贴原始需求原文。必须是经过 AI 理解和重组后的清晰描述。如果原始需求有多处模糊已澄清，在澄清记录中体现，这里写澄清后的版本。 |

**示例：**

```markdown
## 需求描述

**为订单管理模块新增批量导出功能，支持按时间范围和订单状态筛选，导出为 Excel 格式。**

用户（客服人员）需要在订单列表页选中或按条件筛选订单后，点击"导出"按钮下载 Excel 文件。Excel 文件包含订单编号、客户名称、下单时间、订单金额、订单状态五列。导出上限为 10000 条，超出时提示用户缩小筛选范围。

不允许导出已删除状态的订单。
```

### 字段 3：功能需求（Functional Requirements）

| 属性 | 内容 |
|------|------|
| **字段名称** | 功能需求 / Functional Requirements |
| **说明** | 将需求拆解为可独立验证的功能点，每个功能点用三元组描述：现状（Current State）→ 目标（Target State）→ 验收标准（Acceptance Criteria）。这是 Spec 中最核心的字段，直接驱动后续 Plan 和 Review。引用 gsd 的可证伪需求模式。 |
| **格式要求** | 编号列表。每个功能需求用三级结构：FR 编号 + 名称 + 三元组表格。 |
| **示例** | 见下方 |
| **注意事项** | 每个 FR 必须独立可验证——不能出现"和 FR-002 一起看"的情况。验收标准必须是可证伪的（Pass/Fail），不能用"性能好"、"体验流畅"等主观描述。如果某个 FR 有 NEEDS CLARIFICATION，在验收标准中标出。 |

**三元组格式说明：**

| 维度 | 含义 | 示例 |
|------|------|------|
| **现状** | 现在系统是什么样 | "用户只能单条查看订单，无法批量操作" |
| **目标** | 做完后应该是什么样 | "用户可以选中多条订单并导出为 Excel" |
| **验收标准** | 怎么判断做完了 | "选择 3 条订单点击导出，下载的 Excel 包含这 3 条记录" |

**示例：**

```markdown
## 功能需求

### FR-001: 批量选择订单
| 维度 | 描述 |
|------|------|
| **现状** | 订单列表页只能单条查看，没有多选能力 |
| **目标** | 订单列表页支持全选、Shift 范围选、Ctrl/Cmd 多选 |
| **验收标准** | AC-001: 用户可以勾选多条订单，选中的订单高亮显示 |

### FR-002: 条件筛选导出
| 维度 | 描述 |
|------|------|
| **现状** | 没有导出功能 |
| **目标** | 用户可按时间范围（开始日期-结束日期）和订单状态（多选）筛选后导出 |
| **验收标准** | AC-002: 选择"2026年5月"+"已完成"状态，导出结果只包含该条件的订单 |

### FR-003: Excel 文件下载
| 维度 | 描述 |
|------|------|
| **现状** | 没有文件下载功能 |
| **目标** | 点击导出后生成 .xlsx 文件并触发浏览器下载 |
| **验收标准** | AC-003: 导出的 Excel 包含 5 列（订单编号、客户名称、下单时间、订单金额、订单状态），内容与筛选条件一致 |

### FR-004: 导出数量限制
| 维度 | 描述 |
|------|------|
| **现状** | 无限制 |
| **目标** | 单次导出上限 10000 条，超出时阻止导出并提示用户缩小范围 |
| **验收标准** | AC-004: 筛选结果为 12000 条时点击导出，显示"请缩小筛选范围，最多导出 10000 条"提示 |
```

### 字段 4：验收标准（Acceptance Criteria）

| 属性 | 内容 |
|------|------|
| **字段名称** | 验收标准 / Acceptance Criteria |
| **说明** | 从功能需求中提取的、独立的、可证伪的 Pass/Fail 检查项。这些标准将作为 Review 阶段 C 轴（验收标准逐条确认）的校验基准。每条验收标准必须能用"通过/未通过"来判定。 |
| **格式要求** | 编号列表（AC-001, AC-002, ...），每条包含：编号 + 描述 + 可选的 Pass/Fail 说明。 |
| **示例** | 见下方 |
| **注意事项** | 每条验收标准只验证一件事。避免"并且"——如果出现"并且"，拆成两条。验收标准不能引用实现细节（如"使用 Redis 缓存"），只描述行为。 |

**示例：**

```markdown
## 验收标准

- [ ] **AC-001**: 用户可以在订单列表页勾选多条订单，选中行高亮显示
- [ ] **AC-002**: 选择时间范围"2026-05-01 至 2026-05-17"和状态"已完成"后导出，结果仅包含该范围内已完成状态的订单
- [ ] **AC-003**: 导出的 Excel 文件包含 5 列（订单编号、客户名称、下单时间、订单金额、订单状态），列名与顺序正确
- [ ] **AC-004**: 筛选结果超过 10000 条时，点击导出显示提示信息，不触发下载
- [ ] **AC-005**: 不允许导出已删除状态的订单（筛选条件中不包含"已删除"选项）

> **注：** 以上验收标准将作为 Review 阶段 C 轴的校验基准。
```

### 字段 5：边界定义（Boundary Definition）

| 属性 | 内容 |
|------|------|
| **字段名称** | 边界定义 / Boundary Definition |
| **说明** | 明确划定"做什么"和"不做什么"的边界。这是需求评审中最容易遗漏的字段——没有明确说"不做"的事情，往往在实施阶段被默认为"要做"。引用 gsd 的 Boundary Clarity 维度。 |
| **格式要求** | 两个子章节：`### 包含`（In Scope）和 `### 不包含`（Out of Scope），各用列表。 |
| **示例** | 见下方 |
| **注意事项** | "不包含"部分和"包含"一样重要——写清楚不做什么是防止范围蔓延的关键。如果某个边界点有争议，标记 NEEDS CLARIFICATION 并说明影响什么。 |

**示例：**

```markdown
## 边界定义

### 包含（In Scope）
- 订单列表页的多选和筛选交互
- 后端批量查询和 Excel 生成接口
- 浏览器端文件下载
- 导出数量限制的前端校验

### 不包含（Out of Scope）
- 导出为 PDF/CSV 等其他格式（本期只做 Excel）
- 定时自动导出/邮件发送
- 导出任务的进度条和取消功能
- 导出历史记录
- 移动端适配（本期只做 PC 端）
```

### 字段 6：澄清记录（Clarification Log）

| 属性 | 内容 |
|------|------|
| **字段名称** | 澄清记录 / Clarification Log |
| **说明** | 记录需求评审过程中提出的所有问题和回答。这是评审过程的审计轨迹——任何人回头看这份 Spec，都能理解"为什么这样决定"。引用 bmad 的 Decision Log 概念。 |
| **格式要求** | Markdown 表格，列：编号、问题、回答、来源（AI 提问 / Developer 补充 / 原始文档）。 |
| **示例** | 见下方 |
| **注意事项** | 每个澄清点都是独立的——不要把多个问题合并到一行。来源必须标注清楚，区分"AI 发现的问题"和"人主动补充的信息"。未解决的澄清项保留在表格中，回答栏标记 `[NEEDS CLARIFICATION]`。 |

**示例：**

```markdown
## 澄清记录

| # | 问题 | 回答 | 来源 |
|---|------|------|------|
| Q1 | 导出的 Excel 列是否需要自定义？ | 不需要，固定 5 列 | Developer 补充 |
| Q2 | 超过 10000 条时是否支持分批导出？ | 不支持，直接提示缩小范围 | Developer 补充 |
| Q3 | 已删除的订单能否导出？ | 不能，筛选条件中不包含"已删除"选项 | AI 提问 |
| Q4 | 导出操作是否需要权限控制？ | [NEEDS CLARIFICATION] 影响范围：是否需要后端权限校验 | AI 提问 |
| Q5 | 并发导出时是否需要排队？ | MVP 不考虑，单用户场景 | Developer 补充 |
```

## 四、NEEDS CLARIFICATION 机制

### 4.1 标记规则（引用设计决策 Q2）

- **不设数量上限**：有多少标多少
- 每个标记必须说明**为什么需要澄清**（影响什么决策）
- 标记可以出现在任何字段中

### 4.2 标记格式

```
[NEEDS CLARIFICATION: 影响什么决策]
```

**示例：**

```markdown
### FR-005: 导出权限控制
| 维度 | 描述 |
|------|------|
| **现状** | 没有权限控制 |
| **目标** | [NEEDS CLARIFICATION: 是否需要角色级别的权限控制？影响后端接口设计] |
| **验收标准** | AC-006: [取决于权限控制方案] |
```

### 4.3 NEEDS CLARIFICATION 在澄清记录中的追踪

每个 NEEDS CLARIFICATION 标记必须同步出现在澄清记录中：

```markdown
| Q4 | 导出操作是否需要权限控制？ | [NEEDS CLARIFICATION: 是否需要角色级别权限控制，影响后端接口设计] | AI 提问 |
```

当 Developer 回答后，更新澄清记录并移除 Spec 正文中的标记。

## 五、完整 Spec 模板

以下是将所有字段组合后的完整模板：

```markdown
# Requirement Review Spec: [功能名]

> 关联任务：.agents/active/{task-name}/status.yaml
> 生成时间：YYYY-MM-DD HH:mm

## 需求来源
```
