# 多智能体专家团框架 — 高阶方案

> 版本：v0.1（讨论稿）
> 日期：2026-05-08
> 状态：内部对齐，待评审

---

## 一、背景与问题

### 1.1 当前 AI 工具的共同瓶颈

无论是 AI 编码工具（Cursor、Claude Code）、AI 协作平台（CodeBanana）、还是提示词框架（BMAD），在复杂任务面前都面临同一个问题：

**单 Agent 承担过多职责 → 上下文膨胀 → 可靠性下降 → 人工兜底成本高。**

具体表现：
- 长任务中 Agent "失忆"——做到后面忘了前面
- 一个人盯 Agent 比自己写还累
- 角色混淆——该追问的时候在执行，该执行的时候在发散
- 知识无法沉淀——换个人来，Agent 又从零开始

### 1.2 行业趋势

AI 编程工具正在从单 Agent 模式转向多智能体协作模式。阿里 Qoder 的专家团（Experts Mode）、开源框架 BMAD 的多角色体系，都在验证同一个方向：**让多个专精的 Agent 协作，而不是让一个全能 Agent 硬扛。**

### 1.3 我们的机会

现有方案各有缺陷：
- **BMAD**：设计理念好，但学习成本高、单 LLM 模拟（非真正多 Agent）、无持久记忆
- **Qoder**：真正多 Agent，但"Agent 执行、人旁观"的定位有风险，且绑定在特定 IDE 内
- **CodeBanana**：三位一体架构有野心，但 Agent 过于复杂、无法并发

我们需要一个**吸收各方优点、规避已知缺陷**的方案。

---

## 二、核心理念

**一套与项目/工作区稳定绑定的多角色专家智能体系统，以需求流转为主线，让 AI 作为辅助角色参与全流程，人始终是执行主体和决策者。**

三个关键定位：
1. **多角色分工**，而不是一个 Agent 全栈
2. **辅助定位**，而不是替代人
3. **需求流转驱动**，而不是工具堆砌

---

## 三、架构设计

### 3.1 角色体系

系统由三类角色组成：

#### A. 专家角色（Specialists）

按职能定义，每个角色有明确的职责边界和产出标准。

| 角色 | 核心职责 | 典型产出 |
|------|---------|---------|
| **需求分析师** | 需求澄清、可行性评估、竞品调研 | 需求文档、调研报告 |
| **产品经理** | PRD 撰写、优先级排序、用户故事拆分 | PRD、Story List |
| **架构师** | 技术方案设计、架构决策、技术选型 | 架构方案、技术选型文档 |
| **开发工程师** | 代码实现、Bug 修复、技术优化 | 代码、变更记录 |
| **测试工程师** | 测试用例设计、质量验证、边界覆盖 | 测试报告、Bug List |
| **代码评审员** | 代码规范检查、安全审查、可维护性评估 | Review 意见、改进建议 |
| **运维工程师** | 部署方案、监控策略、环境管理 | 部署文档、监控配置 |
| **技术文档** | 文档撰写、知识整理、API 文档 | 技术文档、使用手册 |

> 角色可根据项目类型增减，不限于上述列表。

#### B. 调度角色（Facilitator）

**零学习成本的用户入口。** 不执行具体工作，只做三件事：

1. **理解意图**：用户用自然语言描述需求，Facilitator 翻译为角色推荐
2. **追问澄清**：需求模糊时主动追问，不猜测
3. **路由推荐**：建议该找谁、按什么顺序协作

**设计约束：**
- Facilitator 本身不引入新的概念或命令体系
- 用户已知目标角色时可直接调用，不强制经过 Facilitator
- Facilitator 的推荐是建议，不是指令——用户有最终决定权

**交互示例：**

```
用户："我想给系统加一个微信登录功能。"

Facilitator：
> 这是一个涉及前后端的功能需求，建议按以下步骤推进：
> 
> 1. 【需求分析师】先梳理微信 OAuth 接入的技术约束和用户场景
> 2. 【产品经理】输出 PRD，定义登录流程和异常处理
> 3. 【架构师】设计接入方案（SDK 选型、Token 管理）
> 4. 【开发工程师】按方案实现
> 5. 【测试工程师】覆盖正常流程 + 异常场景
> 
> 要从第 1 步开始，还是你已经有了明确需求想直接进入开发？
```

#### C. 协调角色（Orchestrator，可选）

在复杂项目中，可引入一个协调者角色负责：
- 追踪需求流转状态（哪个环节、谁在负责、产出是什么）
- 发现阻塞和瓶颈，提醒相关人员
- 生成进度报告

> 在小团队/简单项目中，协调职能可由 Facilitator 兼任。

### 3.2 角色间信息传递

**基于文件的结构化传递**——每个角色的产出是标准化的文件，下游角色消费该文件作为输入。

```
需求分析师 ──→ 需求文档.md
                    ↓
产品经理   ──→ PRD.md + Story List.md
                    ↓
架构师     ──→ 架构方案.md
                    ↓
开发工程师 ──→ 代码变更 + 变更记录.md
                    ↓
测试工程师 ──→ 测试报告.md
                    ↓
代码评审员 ──→ Review 意见.md
```

**为什么选文件传递而不是 API/消息传递？**
- **可追溯**：每个环节的产出有据可查
- **可审查**：人在任何时候可以介入修改
- **可版本控制**：放进代码库，Git 追踪所有变更
- **松耦合**：角色之间不依赖对方的技术实现，只依赖文件格式
- **工具无关**：不绑定特定平台或框架

### 3.3 配置分层

```
项目层（随代码库版本控制）
├── .agents/
│   ├── config.yaml          # 项目级角色配置
│   ├── roles/               # 角色定义
│   │   ├── architect.md
│   │   ├── developer.md
│   │   └── ...
│   ├── workflows/           # 工作流定义
│   │   └── feature-flow.md
│   └── templates/           # 产出模板
│       ├── prd-template.md
│       └── arch-template.md

个人层（本地，不进入版本控制）
├── ~/.agents/
│   ├── config.yaml          # 个人偏好
│   ├── preferences/         # 编码习惯、沟通风格
│   └── evolution/           # 角色自演进数据
```

**项目层定义"团队共识"：**
- 我们的技术栈是什么
- 代码规范是什么
- 需求流转的环节有哪些
- 每个角色的职责边界

**个人层定义"个体偏好"：**
- 我喜欢什么样的沟通风格
- 我习惯用什么模型
- 我的历史偏好和上下文

**核心价值：团队共识和个体偏好解耦。** 项目配置可迁移（换人接手自然继承），个人配置有主权（不随项目传播）。

## 四、需求流转工作流

以一个完整的功能需求为例，展示角色如何被串起来：

```
阶段一：分析
  需求输入 → [需求分析师] → 需求文档
                                  ↓
阶段二：规划                    ↓
  需求文档 → [产品经理] → PRD + Story List
                                  ↓
阶段三：设计                    ↓
  PRD → [架构师] → 架构方案 + 技术选型
                                  ↓
阶段四：实现                    ↓
  架构方案 + Story → [开发工程师] → 代码 + 变更记录
                                  ↓
阶段五：验证                    ↓
  代码 → [测试工程师] → 测试报告
       → [代码评审员] → Review 意见
                                  ↓
阶段六：交付                    ↓
  全部产出 → [Facilitator] → 交付摘要，人做最终验收
```

**关键设计：**
- 每个阶段的产出是结构化文件，下一个阶段的角色读取该文件作为上下文
- 人可以在任何阶段介入修改产出，修改后的文件自动成为下游角色的新输入
- 阶段之间不是严格串行——测试工程师和代码评审员可以并行工作
- 不是所有需求都要走全流程——简单的 Bug 修复可以跳过分析和规划阶段

## 五、自演进机制

### 5.1 个体进化

每个角色在与用户协作的过程中持续学习：

| 进化维度 | 说明 | 存储位置 |
|---------|------|---------|
| 技术栈偏好 | 用户常用什么框架、库、工具 | 个人层 |
| 沟通风格 | 用户喜欢详细还是简洁、中文还是英文 | 个人层 |
| 质量标准 | 用户对测试覆盖率、文档详细度的要求 | 个人层 |
| 决策历史 | 过去的架构选择和原因 | 项目层 + 个人层 |

### 5.2 团队进化

系统记录每次协作的经验：

| 进化维度 | 说明 |
|---------|------|
| 最优阵容 | 某类任务的最佳角色组合和顺序 |
| 常见问题 | 哪些环节容易出问题，怎么预防 |
| 效率数据 | 每个角色的平均耗时和质量评分 |

### 5.3 演化的边界

- 演化结果沉淀在**个人层**，不污染项目层的团队共识配置
- 项目负责人可以决定是否将某些个人层的优秀实践"提升"到项目层
- 演化是增量式的，不覆盖原有配置，而是作为补充层叠加

## 六、模型路由

不同角色可以运行在不同的 AI 模型上：

| 角色类型 | 模型倾向 | 原因 |
|---------|---------|------|
| 需求分析、架构设计 | 强推理模型（Opus 级） | 需要深度思考和多维度权衡 |
| 代码实现 | 快速编码模型（Sonnet/GLM 级） | 吞吐量优先，上下文干净 |
| 测试生成 | 专项模型 | 覆盖率和准确性优先 |
| Facilitator | 轻量快速模型 | 响应速度优先，不执行复杂任务 |

**实现依赖：** 需要底层工具支持 subagent + 模型指定能力。当前 OpenClaw 的 `sessions_spawn` 已支持 `model` 参数，基本可行。

## 七、关键设计决策及理由

| 决策 | 选择 | 理由 |
|------|------|------|
| Agent 定位 | 辅助角色，非执行主体 | 当前 AI 可靠性不足以承担执行主体责任；人做最终判断才能保证质量 |
| 信息传递方式 | 基于文件 | 可追溯、可审查、可版本控制、工具无关 |
| 工作流驱动方式 | 需求流转（而非任务列表） | 以交付物为主线更符合实际工作节奏 |
| 入口设计 | Facilitator 零学习成本 | 解决多角色系统"不知道找谁"的核心体验问题 |
| 配置管理 | 项目层 vs 个人层分离 | 团队共识和个体偏好解耦，项目可迁移 |
| 是否强制流程 | 否，可跳步可绕过 | 简单需求不需要重型流程，灵活性优先 |

## 八、与现有方案的对比

| 维度 | BMAD | Qoder 专家团 | 本方案 |
|------|------|-------------|-------|
| Agent 定位 | 执行主体 | 执行主体 | **辅助角色，人为主** |
| 多 Agent 实现 | 单 LLM 模拟 | 真正多 Agent 并行 | 真正多 Agent（依赖底层） |
| 入口体验 | Master 角色有菜单 | 模式切换 | **Facilitator 零学习成本** |
| 信息传递 | 文件传递 | Team Lead 翻译 | **文件传递** |
| 配置分层 | 有但不够彻底 | 项目/个人分层 | **项目/个人分层 + 演化层** |
| 自演进 | 无 | 有（个体+团队） | **有，演化沉淀在个人层** |
| 模型路由 | 无 | 有 | **有（依赖底层）** |
| 学习成本 | 高 | 中 | **低（Facilitator 入口）** |
| 工具绑定 | 无（纯提示词） | 绑定 Qoder IDE | **无（文件驱动）** |

## 九、可行性评估

### 9.1 技术可行性

| 能力 | 当前状态 | 说明 |
|------|---------|------|
| 多 Agent 调度 | ✅ 可行 | OpenClaw sessions_spawn 支持 subagent + model 指定 |
| 文件传递 | ✅ 可行 | 纯文件读写，无技术障碍 |
| 配置分层 | ✅ 可行 | 项目目录 + 用户目录，标准文件管理 |
| 模型路由 | ⚠️ 基本可行 | sessions_spawn 支持 model 参数，但需要验证不同模型对相同角色定义的兼容性 |
| 自演进 | ⚠️ 需要设计 | 存储和调用方式需要进一步设计，核心是"什么信号触发学习、学习结果存什么格式" |
| Facilitator | ✅ 可行 | 本质是一个路由 prompt，技术复杂度低 |

### 9.2 主要风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 角色定义不够精确导致输出质量不稳定 | 高 | 先用少量核心角色验证，逐步扩展 |
| 文件接口规范设计不当导致角色间信息丢失 | 中 | 参照 BMAD 的模板体系，先跑通一个完整流程再优化 |
| 自演进机制复杂度高 | 中 | 先做简单版本（记录偏好），复杂版本（自动调优）放后期 |
| 团队接受度——新工作方式的采用成本 | 高 | Facilitator 设计降低入门门槛；先在非关键项目试点 |

## 十、建议的下一步

1. **选一个具体项目做原型验证**——建议选一个中等复杂度的内部项目
2. **定义第一版角色集**——先做 3-4 个核心角色（架构师、开发、测试、Facilitator）
3. **跑通一个完整的需求流转**——从一个真实需求出发，验证文件传递和工作流可行性
4. **收集团队反馈**——重点关注 Facilitator 的入口体验和角色间协作的流畅度
5. **迭代角色定义和接口规范**——基于实际使用反馈优化

## 附录：术语表

| 术语 | 定义 |
|------|------|
| **专家角色（Specialist）** | 有明确职责边界的专项 Agent，只负责特定领域的工作 |
| **Facilitator** | 零学习成本的入口角色，帮助用户理解应该找谁、按什么顺序协作 |
| **项目层配置** | 随代码库版本控制的团队共识配置，所有人共享 |
| **个人层配置** | 保留在用户本地的个体偏好配置，不进入版本控制 |
| **需求流转** | 一个需求从提出到交付的完整生命周期，每个环节由对应的角色负责 |
| **文件传递** | 角色间通过结构化文件（而非 API 或消息）传递信息的协作方式 |
| **模型路由** | 不同角色使用不同 AI 模型的能力 |
| **自演进** | 角色基于历史协作经验持续优化自身表现的能力 |
