# 规范驱动开发框架研究报告

> 日期：2026-05-09
> 研究范围：7 个主流框架 + 行业趋势

---

## 一、研究概览

研究了 7 个 GitHub 上最活跃的规范驱动开发/AI 编程框架。这些框架覆盖了从纯方法论（Superpowers）到完整工具链（SpecKit、OpenSpec）到企业级工作流系统（GSD、gstack）的完整光谱。

**整体趋势：**
1. **工作流阶段化已成共识**——所有框架都把"从想法到交付"拆成多个阶段
2. **Spec 先行是共同信条**——先写规范再写代码，没有例外
3. **"人把关 + AI 执行"是主流模式**——人负责决策和审核，AI 负责实现
4. **技能/Skill 文件化是核心实现方式**——通过 .md 文件定义 AI 的行为
5. **可观测性和状态管理越来越受重视**——GSD 的 `.planning/STATE.md`、OpenSpec 的 `state.yaml` 都是例证

### GitHub Stars 排名

### 1. Superpowers（⭐ 183K）

**仓库：** https://github.com/obra/superpowers
**作者：** Jesse Vincent (Prime Radiant)
**定位：** 一套基于可组合技能的开发方法论，适配所有主流 IDE

#### 核心理念

**技能（Skill）驱动 + 自动触发。** Superpowers 不要求用户手动选择流程——AI agent 在做任何事之前自动检查是否有适用的 skill，然后强制执行。

关键哲学：
- **TDD 强制**——先写测试，再写代码，不例外
- **系统化优于即兴**——用流程代替猜测
- **证据优于声明**——验证后才算完成
- **YAGNI 极简**——不提前做不需要的东西

#### 目录结构

```
superpowers/
├── skills/                    # 核心技能定义
│   ├── brainstorming/         # 阶段 1：设计头脑风暴
│   │   └── SKILL.md
│   ├── writing-plans/         # 阶段 2：写实现计划
│   │   └── SKILL.md
│   ├── executing-plans/       # 阶段 3a：执行计划（单 agent）
│   │   └── SKILL.md
│   ├── subagent-driven-development/  # 阶段 3b：执行计划（多 agent）
│   │   └── SKILL.md
│   ├── test-driven-development/  # 强制 TDD
│   │   └── SKILL.md
│   ├── requesting-code-review/   # 代码评审
│   │   └── SKILL.md
│   └── finishing-a-development-branch/  # 收尾
│       └── SKILL.md
├── .cursor-plugin/            # Cursor 插件
├── .claude-plugin/            # Claude Code 插件
├── .codex-plugin/             # Codex 插件
└── tests/                     # 技能测试
```

#### 工作流设计

```
Brainstorm → Plan → Execute → Review → Finish
```

**关键设计决策：**
1. **HARD-GATE 机制**——brainstorming skill 有一个硬门控：`Do NOT invoke any implementation skill, write any code... until you have presented a design and the user has approved it.`
2. **自审核循环**——写完 spec 后自动做 placeholder scan、内部一致性检查、scope 检查
3. **Subagent 模式**——每个 task 分给独立的 subagent 执行，主 agent 做两阶段 review（先检查 spec 合规性，再检查代码质量）
4. **计划粒度极细**——每个 step 2-5 分钟，包含完整代码和验证命令
5. **无缝衔接**——brainstorming 的终端状态是调用 writing-plans，writing-plans 的终端状态是调用 executing-plans

#### Skill 文件结构

每个 SKILL.md 包含：
```yaml
---
name: brainstorming
description: "..."
---
```
然后是完整的 Markdown 指令，包括：
- 过程流程图
- 具体步骤 checklist
- 检查清单
- 关键原则

#### 亮点

1. **自动触发机制**——AI 自动检测应该用哪个 skill，用户不需要手动选择
2. **极致的 TDD 强制**——代码评审中如果发现代码写在测试之前，会删掉代码
3. **Two-stage review**——先检查"是否按 spec 做"，再检查"代码质量好不好"
4. **计划即 prompt**——PLAN.md 本身就是给 subagent 的 prompt，不是文档
5. **"Anti-pattern: This Is Too Simple"**——即使是简单项目也强制过设计流程

#### 不足

- 没有"角色"概念——所有 skill 都由同一个 agent 执行
- 没有配置分层——没有项目层/个人层的区分
- 没有自演进机制

### 2. BMAD Method（⭐ 46.7K）

**仓库：** https://github.com/bmad-code-org/BMAD-METHOD
**定位：** 完整的 AI 驱动开发方法论，从需求到交付

#### 核心理念

**角色驱动 + 阶段流程。** BMAD 定义了一组有名字、有性格的 AI 角色（Mary the Analyst、Winston the Architect 等），用户通过对话与这些角色交互。

#### 目录结构

```
BMAD-METHOD/src/
├── core-skills/               # 核心技能（通用）
│   ├── bmad-brainstorming/
│   ├── bmad-advanced-elicitation/
│   ├── bmad-distillator/
│   └── ...
├── bmm-skills/                # BMAD 方法论模块
│   ├── 1-analysis/            # 阶段 1：分析
│   │   ├── bmad-agent-analyst/    # Mary - 需求分析师
│   │   ├── bmad-product-brief/
│   │   └── bmad-prfaq/
│   ├── 2-plan-workflows/      # 阶段 2：规划
│   │   ├── bmad-agent-pm/        # 产品经理
│   │   ├── bmad-create-prd/      # 创建 PRD（12 步引导流程）
│   │   ├── bmad-agent-ux-designer/
│   │   └── bmad-edit-prd/
│   ├── 3-solutioning/         # 阶段 3：方案设计
│   │   ├── bmad-agent-architect/  # Winston - 架构师
│   │   ├── bmad-create-architecture/
│   │   ├── bmad-create-epics-and-stories/
│   │   └── bmad-generate-project-context/
│   └── 4-implementation/      # 阶段 4：实现
│       ├── bmad-agent-dev/        # 开发工程师
│       ├── bmad-dev-story/
│       ├── bmad-code-review/
│       ├── bmad-sprint-planning/
│       └── bmad-retrospective/
```

#### 角色设计

每个角色有独立的 SKILL.md + customize.toml：

**Mary（需求分析师）**：
- 图标：📊
- 身份："Strategic business analyst and requirements expert"
- 沟通风格：专业、证据驱动
- 激活流程：解析配置 → 采纳人设 → 加载持久事实 → 问候 → 展示菜单
- 菜单系统：每个角色有编号菜单，用户可以选择功能

**Winston（架构师）**：
- 图标：🏗️
- 身份："Channels Martin Fowler's pragmatism and Werner Vogels's cloud-scale realism"
- 原则："Rule of Three before abstraction"、"Boring technology for stability"

#### 定制化系统

BMAD 有三层配置合并机制：
```
base (customize.toml) → team (_bmad/custom/{skill}.toml) → user (_bmad/custom/{skill}.user.toml)
```
- 标量覆盖
- 表深度合并
- 带 code/id 的数组替换匹配项并追加新项

#### 亮点

1. **有性格的角色**——不只是"架构师"，而是"Winston，务实风格，Martin Fowler 的追随者"
2. **三层配置合并**——base → team → user，精细控制
3. **菜单系统**——每个角色有编号菜单，用户像点菜一样选择功能
4. **12 步 PRD 引导流程**——从 discovery 到 polish，每步有独立的 .md 文件
5. **persistent_facts**——角色记住的持久事实，支持文件引用和 glob 模式
6. **config.yaml 集中配置**——用户名、沟通语言、文档语言、产出目录等

#### 不足

- 角色太多太重——完整安装有 20+ skills
- 学习曲线陡峭——需要理解整个方法论
- 角色之间切换成本高——每个角色有独立的激活流程

### 3. SpecKit（⭐ 93.8K，GitHub 官方）

**仓库：** https://github.com/github/spec-kit
**定位：** GitHub 官方的 Spec 驱动开发工具链

#### 核心理念

**Spec 变成可执行的。** SpecKit 的核心创新是把 specification 从"废弃的脚手架"变成"直接生成工作实现的源文件"。

#### 目录结构

```
spec-kit/
├── templates/                 # 模板
│   ├── spec-template.md       # 功能规范模板（User Story + Requirements + Success Criteria）
│   ├── plan-template.md       # 实现计划模板（技术上下文 + 任务拆分）
│   ├── tasks-template.md      # 任务模板
│   └── constitution-template.md  # 项目宪法（质量门控）
├── workflows/                 # 工作流定义
│   └── speckit/
│       └── workflow.yml       # specify → plan → tasks → implement
├── presets/                   # 预设配置
├── extensions/                # 扩展
└── integrations/              # IDE 集成
```

#### 工作流设计（workflow.yml）

```yaml
steps:
  - id: specify          # 生成 spec
  - id: review-spec      # 🚪 门控：人工审核 spec
  - id: plan             # 生成计划
  - id: review-plan      # 🚪 门控：人工审核计划
  - id: tasks            # 生成任务
  - id: implement        # 执行实现
```

**关键设计：**
- 工作流是声明式的（YAML），不是代码
- 每个步骤之间有 `gate` 类型的审核节点
- 支持 `on_reject: abort` 和重新开始

#### Spec 模板结构

```markdown
# Feature Specification
## User Scenarios & Testing (mandatory)
  ### User Story 1 - [Title] (Priority: P1)
    - Why this priority
    - Independent Test
    - Acceptance Scenarios (Given/When/Then)
## Requirements (mandatory)
  ### Functional Requirements (FR-001, FR-002...)
  ### Key Entities
## Success Criteria (mandatory)
  ### Measurable Outcomes (SC-001, SC-002...)
## Assumptions
```

#### 亮点

1. **GitHub 官方背书**——与 GitHub Copilot 深度集成
2. **声明式工作流**——YAML 定义步骤和门控，清晰可改
3. **Constitution 机制**——项目级"宪法"定义质量标准
4. **模板驱动**——规范、计划、任务都有标准模板
5. **Need Clarification 标记**——不确定的需求显式标记 `[NEEDS CLARIFICATION: ...]`
6. **独立可测试的 User Story**——每个 story 必须能独立测试、独立交付价值

#### 不足

- 更偏"规范生成工具"而非"开发方法论"
- 没有角色概念
- 没有自演进

### 4. OpenSpec（⭐ 46.4K）

**仓库：** https://github.com/Fission-AI/OpenSpec
**定位：** 开放的 Spec 工作流框架，强调"fluid not rigid"

#### 核心理念

**非线性工作流。** OpenSpec 认为真实工作不是线性的——你会实现到一半发现设计有问题，需要回头改 spec。所以它提供的是"actions"（随时可以执行的动作），而不是"phases"（必须按顺序走的阶段）。

#### OPSX 工作流

```
/opsx:propose → specs/ + design.md + tasks.md
/opsx:apply   → 自动实现
/opsx:archive → 归档到历史
```

**关键创新：**
- **Action-based，不是 Phase-based**——create、implement、update、archive 随时可用
- **依赖是 enabler，不是约束**——依赖关系告诉你"什么可能"，不强制"下一步"
- **Schema.yaml + templates/*.md 可定制**——任何人可以改 prompt 模板，即时生效

#### 目录结构

```
OpenSpec/
├── src/
│   ├── commands/          # CLI 命令
│   ├── core/              # 核心逻辑
│   ├── prompts/           # Prompt 模板
│   └── ui/                # 仪表盘 UI
├── openspec/
│   └── specs/             # 自身用 OpenSpec 管理
├── schemas/               # JSON Schema
└── docs/
```

#### 亮点

1. **非线性工作流**——最贴近真实开发节奏的设计
2. **模板可定制**——改 prompt 模板立即生效，不需要发布新版本
3. **仪表盘 UI**——可视化追踪所有 spec 和变更
4. **`config.yaml` 项目配置**——注入技术栈上下文到所有产出
5. **"fluid not rigid"哲学**——整个设计围绕"工作不是线性的"这个认知

#### 不足

- TypeScript 实现，工具链较重
- 仍在早期，很多功能在重构中
- 不如 Superpowers 的 skill 生态成熟

### 5. Get Shit Done / GSD（⭐ 61K）

**仓库：** https://github.com/gsd-build/get-shit-done
**定位：** 轻量级上下文工程系统，解决"context rot"问题

#### 核心理念

**上下文质量管理。** GSD 的核心洞见是 AI 编程质量下降的根因是"上下文腐烂"——随着对话进行，AI 的 context window 被填满，质量从 PEAK → GOOD → DEGRADING → POOR 逐步下降。

**解决方案：** 把工作拆成小粒度的 PLAN.md（每个消耗 10-30% context），每个 PLAN 由独立 agent 执行。

#### 目录结构

```
get-shit-done/
├── agents/                    # 33 个专业 agent 定义
│   ├── gsd-planner.md         # 计划器（最复杂，~500 行）
│   ├── gsd-executor.md        # 执行器
│   ├── gsd-verifier.md        # 验证器
│   ├── gsd-code-reviewer.md   # 代码评审
│   ├── gsd-debugger.md        # 调试器
│   ├── gsd-research-synthesizer.md
│   └── ...                    # 还有 26 个
├── get-shit-done/             # 工作流和引用文件
├── hooks/                     # Git hooks
├── sdk/                       # GSD SDK
└── tests/                     # 测试
```

#### 工作流设计

```
/gsd-new-project  → 初始化（问题 → 研究 → 需求 → 路线图）
/gsd-discuss-phase → 讨论（用户确认决策）
/gsd-plan-phase   → 计划（生成 PLAN.md）
/gsd-execute-phase → 执行（subagent 执行每个 PLAN）
/gsd-verify-phase → 验证（自动验证 + 人工确认）
```

#### Agent 定义结构（以 gsd-planner 为例）

极其详细，包含：
- `<role>` — 角色定义和核心职责
- `<context_fidelity>` — 用户决策保真（锁定决策必须执行，延迟想法必须排除）
- `<scope_reduction_prohibition>` — 禁止简化用户决策
- `<planner_authority_limits>` — 规划者不能判断"太难"
- `<philosophy>` — 独行侠 + Claude 的工作模式
- `<task_breakdown>` — 任务拆分规则（每个 PLAN 2-3 个 task）
- `<dependency_graph>` — 依赖图构建
- `<goal_backward>` — 目标倒推方法论
- `<checkpoints>` — 检查点类型和格式
- `<tdd_integration>` — TDD 集成
- `<execution_flow>` — 完整执行流程（20+ 个步骤）

#### 上下文质量曲线

| Context Usage | Quality | 状态 |
|---------------|---------|------|
| 0-30% | PEAK | 全面、细致 |
| 30-50% | GOOD | 自信、扎实 |
| 50-70% | DEGRADING | 开始走捷径 |
| 70%+ | POOR | 敷衍了事 |

**核心规则：每个 PLAN 必须在 ~50% context 内完成。**

#### 亮点

1. **上下文质量管理是核心创新**——其他框架都没解决这个问题
2. **Goal-Backward 方法论**——从目标倒推必须满足的 truth、必须存在的 artifact、必须连接的 wiring
3. **Wave-based 并行**——按依赖图分组，无依赖的 PLAN 可以并行执行
4. **Interface-First**——先定义接口合约，再实现
5. **极度详细的 Agent 定义**——gsd-planner.md 有 500+ 行，覆盖几乎所有边界情况
6. **33 个专业 agent**——从研究员到安全审计员，每个有独立职责
7. **决策保真机制**——锁定决策不可违反，延迟想法不可混入
8. **CONTEXT.md 全局上下文**——所有决策和术语定义集中管理

#### 不足

- 复杂度极高——学习成本可能比 BMAD 还高
- 33 个 agent 有过度设计之嫌
- 依赖 Claude Code 的 subagent 能力

### 6. gstack（⭐ 91.8K）

**仓库：** https://github.com/garrytan/gstack
**作者：** Garry Tan (YC/Initialized Capital)
**定位：** AI 工程工作流，包含浏览器自动化、代码评审、设计系统

#### 核心理念

**技能集合而非方法论。** gstack 不定义一个"从 A 到 Z 的完整流程"，而是提供一组独立的技能，用户按需调用。

#### 技能分类

| 类别 | 技能 |
|------|------|
| 规划评审 | office-hours, plan-ceo-review, plan-eng-review, plan-design-review, plan-devex-review, autoplan |
| 实现评审 | review, codex, investigate, design-review, design-shotgun, qa |
| 发布部署 | ship, land-and-deploy, canary, document-release |
| 运维记忆 | context-save, context-restore, learn, retro, health |
| 安全 | cso (OWASP Top 10 + STRIDE) |

#### 亮点

1. **CEO 级别评审**——从产品视角审视需求，找"10-star product"
2. **多维度评审**——CEO + 设计 + 工程 + DX，每个维度独立评分
3. **Autoplan**——一条命令跑完 CEO → 设计 → 工程 → DX 评审
4. **跨会话记忆**——context-save/context-restore 支持跨会话恢复
5. **Learnings 系统**——跨会话学习，用 JSONL 存储经验
6. **浏览器自动化 QA**——用真实浏览器做 QA，比静态分析更有说服力
7. **安全审计（CSO）**——OWASP Top 10 + STRIDE，专业级安全检查

#### 不足

- 更像"工具箱"而非"方法论"——没有明确的端到端流程
- 依赖 Claude Code 的 skill 系统
- 复杂度高，单个 SKILL.md 就有数千行

### 7. specs.md（⭐ 142）

**仓库：** https://github.com/fabriqaai/specs.md
**定位：** 多流程可选的 Spec 框架

#### 核心理念

**不同场景用不同流程。** specs.md 提供三种流程：

| 流程 | 定位 | Agent 数 | 检查点 |
|------|------|----------|--------|
| Simple | 只生成 spec | 1 | 3（阶段门控） |
| FIRE | 自适应执行 | 3 | 0-2（按复杂度） |
| AI-DLC | 完整方法论 + DDD | 4 | 全覆盖 |

#### FIRE 流程设计

```
/fire-orchestrator   → 入口路由
/fire-planner        → 意图捕获 + 工作项拆分
/fire-builder        → 执行 + 自动生成 walkthrough
```

**关键创新：自适应检查点**——根据任务复杂度决定是否需要人工确认（0、1 或 2 个检查点），而不是固定检查点。

#### 亮点

1. **流程可选**——不像其他框架只提供一种工作流
2. **FIRE 的自适应检查点**——简单任务自动跑，复杂任务才暂停确认
3. **Walkthrough 自动生成**——每个变更自动文档化
4. **VS Code Extension**——可视化追踪进度
5. **Brownfield 友好**——自动检测已有项目的模式和约定

## 三、对比分析

## 五、对我们框架的启示

### 5.1 高优先级借鉴（直接影响 v0.3 设计）

#### 1. 引入"门控"机制

**来源：** Superpowers (HARD-GATE) + SpecKit (review-spec gate)

**现状：** v0.3 有"检查点"概念但比较软——"建议系统提示人确认"。

**建议：** 在 workflow 文件中支持硬门控定义：

```yaml
# workflows/feature.md 中
checkpoints:
  plan:
    type: hard_gate  # 必须人工确认，不通过不能继续
    message: "请确认架构方案后再进入开发"
  review:
    type: hard_gate
    message: "请确认 Review 结果"
```

#### 2. 引入 Goal-Backward 方法论

**来源：** GSD

**现状：** v0.3 的 Plan 阶段只说"输出技术方案和可执行步骤"。

**建议：** 在 Architect 角色的"方案设计"能力中加入 Goal-Backward 方法：
1. 目标是什么（outcome，不是 task）
2. 目标成立必须满足哪些 truth
3. 每个 truth 需要哪些 artifact
4. 每个 artifact 需要哪些 wiring

这让 Plan 阶段的产出质量可验证、可追溯。

#### 3. 引入自审核循环

**来源：** Superpowers

**现状：** v0.3 没有产出质量自检机制。

**建议：** 在每个阶段产出后加一个轻量自检：
- Placeholder scan：有没有 TBD/TODO/待定？
- 一致性检查：方案前后有没有矛盾？
- Scope 检查：是不是太大了需要拆分？

#### 4. 引入 Need Clarification 标记

**来源：** SpecKit

**现状：** v0.3 没有处理不确定性的机制。

**建议：** 在 Analyst 的"澄清"能力和 Architect 的"方案设计"能力中，遇到不确定的点必须显式标记 `[待澄清: ...]`，不允许用模糊表述混过去。

### 5.2 中优先级借鉴（v0.4+ 考虑）

#### 5. 上下文质量管理

**来源：** GSD（核心创新）

**启示：** 我们框架目前没有考虑 context window 的管理。GSD 的洞见是：**每个阶段/任务的产出应该在 50% context 以内完成**。我们可以在 workflow 文件中为每个阶段设定建议的 context 预算。

#### 6. 声明式工作流定义

**来源：** SpecKit (workflow.yml) + OpenSpec (schema.yaml)

**现状：** v0.3 的 workflow 是 Markdown 文件。

**建议：** v0.4 可以考虑引入 YAML 格式的 workflow 定义，更结构化、更容易解析和验证。

#### 7. 三层配置合并

**来源：** BMAD

**启示：** v0.3 已有项目层/个人层分离，但缺少"团队层"和精细的合并规则。BMAD 的 base → team → user 三层合并值得参考。

### 5.3 长期借鉴（v0.5+）

#### 8. Action-based 工作流

**来源：** OpenSpec

**启示：** 当前的工作流还是线性的（Brainstorm → Plan → Execute → Review）。OpenSpec 的 Action-based 思路更贴合实际——未来可以支持"随时 update spec"、"随时回退到 Plan"等非线性操作。

#### 9. Interface-First 任务排序

**来源：** GSD

**启示：** Execute 阶段的任务排序应该遵循"先定义接口合约，再实现"的原则，避免"scavenger hunt"问题。

#### 10. 跨会话学习和记忆

**来源：** gstack (learnings) + GSD (learnings)

**启示：** 自演进机制可以参考 gstack 的 JSONL 存储方式，轻量且可追溯。

## 六、总结：我们框架的差异化定位

| 维度 | 主流框架 | 我们的框架 |
|------|---------|-----------|
| 核心创新点 | 各有侧重 | **工作流驱动 + 角色能力的双层解耦** |
| 工作流 vs 角色 | 要么工作流要么角色 | **两者结合** |
| 信息流转 | 文件传递（Superpowers/BMAD） | **对话上下文为主 + 文件快照为辅** |
| 用户心智模型 | "选角色"或"走流程" | **"在哪个阶段"——最直觉** |
| 上下文管理 | GSD 独有 | 需要补充 |
| 门控机制 | Superpowers/SpecKit 有 | 需要加强 |
| 自演进 | gstack/GSD 有（轻量） | 已规划（渐进式） |

**核心差异化：** 我们是唯一一个把"工作流阶段"和"角色能力"明确解耦的框架。Superpowers 纯阶段、BMAD 纯角色、GSD 重上下文——我们取三者之长，用一个干净的双层架构组合起来。
