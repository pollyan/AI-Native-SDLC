## 一、背景与问题

### 1.1 当前 AI 工具的共同瓶颈

无论是 AI 编码工具(Cursor、Claude Code)、AI 协作平台(CodeBanana)、还是提示词框架(BMAD),在复杂任务面前都面临同一个问题:

**单 Agent 承担过多职责 → 上下文膨胀 → 可靠性下降 → 人工兜底成本高。**

具体表现:
- 长任务中 Agent "失忆"--做到后面忘了前面
- 一个人盯 Agent 比自己写还累
- 角色混淆--该追问的时候在执行,该执行的时候在发散
- 知识无法沉淀--换个人来,Agent 又从零开始

### 1.2 行业趋势

AI 编程工具正在从单 Agent 模式转向多智能体协作模式。阿里 Qoder 的专家团(Experts Mode)、开源框架 BMAD 的多角色体系,都在验证同一个方向:**让多个专精的 Agent 协作,而不是让一个全能 Agent 硬扛。**

### 1.3 我们的机会

现有方案各有缺陷:
- **BMAD**:设计理念好,但学习成本高、单 LLM 模拟(非真正多 Agent)、无持久记忆
- **Qoder**:真正多 Agent,但"Agent 执行、人旁观"的定位有风险,且绑定在特定 IDE 内
- **CodeBanana**:三位一体架构有野心,但 Agent 过于复杂、无法并发

我们需要一个**吸收各方优点、规避已知缺陷**的方案。

## 二、核心理念

**一套与项目/工作区稳定绑定的多角色专家智能体系统,以需求流转为主线,让 AI 作为辅助角色参与全流程,人始终是执行主体和决策者。**

三个关键定位:
1. **多角色分工**,而不是一个 Agent 全栈
2. **辅助定位**,而不是替代人
3. **需求流转驱动**,而不是工具堆砌

### 2.1 核心场景:对话式工作流

本系统的核心使用场景是**人在 IDE 对话框中、以连贯上下文驱动 AI 专家协作**。

- 人是中心节点,不是旁观者。对话本身承担了反馈、澄清、纠偏的功能
- 每个 AI 专家角色在人的对话框中"出场",协助完成当前阶段的任务后"退场"
- 阶段性产出通过文件持久化,但**信息流转不依赖文件**--对话上下文是第一信息源,文件是确认快照
- 人随时可以在对话框中指出问题、跳过环节、修改方向,不需要通过特殊的接口或机制来"通知"系统
- **物理实现上**,所有角色都是 `.md` 文件定义的身份,在同一个 IDE 对话进程中被先后加载。不存在独立进程、独立上下文或"换人"的问题--从头到尾就是同一个对话,只是在不同阶段加载不同的 `.md` 来切换行为

## 三、架构设计(v0.3 核心变更)

### 3.1 双层架构:工作流 × 角色能力

v0.3 的核心架构是**两层结构**:

```
用户感知层(工作流)
  Brainstorm → Plan → Execute → Review
       │          │         │         │
       ▼          ▼         ▼         ▼
能力支撑层(角色)
  Analyst    Architect  Developer  Reviewer
  澄清能力    方案设计    编码能力    质量验证
  调研能力    任务拆分    调试能力    安全审查
```

**外层(用户视角)**:工作流阶段,决定"现在在做什么"。用户只感知阶段的切换--"我在 Brainstorm"、"切到 Plan"。

**内层(系统能力)**:角色 + 能力,决定"怎么做、做到什么标准"。每个阶段背后是某个专业角色在用某种能力在工作。

**关键设计:两者解耦。**
- 角色定义"专业能力池"
- 工作流定义"什么时候用哪个能力"
- 同一个角色可以贡献能力到不同阶段(architect 既做方案设计也做任务拆分)
- 同一个阶段也可以调用不同角色的能力(复杂的 Plan 可能需要 analyst + architect 协作)

### 3.2 为什么是"工作流驱动"而不是"角色驱动"

| 维度 | 角色驱动 | 工作流驱动 |
|------|---------|-----------|
| 用户心智模型 | "把架构师叫出来" | "进入 Plan 阶段" |
| 与已有工作习惯的契合度 | 低--需要理解角色体系 | **高--符合日常"先想清楚、再动手"的节奏** |
| Superpowers 验证 | 无 | **已验证--GitHub 星标数远超其他框架** |
| 专业性保障 | 角色本身提供 | **角色能力提供,不损失深度** |
| 扩展方式 | 加角色 | **加工作流或加角色,独立扩展** |

结论:**用户按工作流阶段思考和操作,系统按专业角色能力执行。** 前者降低认知成本,后者保证专业质量。

### 3.3 v0.2 角色集

v0.2 采用最小角色集策略,4 个核心角色:

| 角色 | 能力池 |
|------|--------|
| **Analyst**(需求分析师) | 澄清、调研、可行性评估 |
| **Architect**(架构师) | 方案设计、技术选型、任务拆分 |
| **Developer**(开发工程师) | 编码、调试、优化 |
| **Reviewer**(评审专家) | 代码评审、测试验证、安全审查 |

> 角色是**可插拔**的。在 `config.yaml` 中定义激活哪些角色,后续可按需扩展(产品经理、运维工程师、技术文档等)。

### 3.4 v0.2 工作流

v0.2 先做一个核心工作流:**Feature Workflow**(从需求到交付)。

```
阶段 1: Brainstorm
  角色:Analyst / 能力:澄清
  做什么:引导用户把模糊想法变成清晰需求
  产出:需求要点
  退出条件:用户确认"需求清楚了,进入下一步"

阶段 2: Plan
  角色:Architect / 能力:方案设计 + 任务拆分
  做什么:基于需求输出技术方案和可执行步骤
  产出:架构方案、Task List
  退出条件:用户确认"方案 OK,开始实现"

阶段 3: Execute
  角色:Developer / 能力:编码
  做什么:按 Task List 逐步实现
  产出:代码变更
  退出条件:所有 Task 完成,用户确认进入 Review

阶段 4: Review
  角色:Reviewer / 能力:质量验证
  做什么:代码评审 + 测试验证
  产出:Review 意见
  退出条件:所有问题解决或确认可接受
```

> 后续可扩展其他工作流:`bugfix.md`(Bug 修复流程,可能只有 Execute + Review)、`refactor.md`(重构流程)、`spike.md`(技术探索流程)。

## 四、信息流转方式

### 4.1 对话上下文 + 文件快照的双层模型

**第一层:对话上下文(实时、连续、零成本)**
- 人和当前出场的 AI 角色在同一对话窗口中交互
- 所有澄清、追问、纠偏都在对话中自然发生
- 人发现问题直接在对话框里说就行
- 对话上下文天然承担了反馈回路的功能

**第二层:文件快照(持久化、可审查、可版本控制)**
- 每个阶段的产出以文件形式持久化,作为"确认快照"
- 文件的角色不是 agent 之间的"API 接口",而是**人对 AI 产出的确认记录**
- 下一阶段的 agent 从对话上下文(主要)+ 前序文件(补充)两方面获取信息
- 文件放进代码库,Git 追踪所有变更

### 4.2 产出格式设计原则

文件是给人审核的,不是给机器解析的:

1. **清晰可审**:每个阶段的产出必须足够清晰,让人能判断"这步 OK,继续下一步"
2. **低成本可改**:人想修改时,直接在对话框里说,或者直接编辑文件
3. **信息边界明确**:每个角色从前序环节获取什么信息?从对话上下文能拿到什么,必须从文件里拿什么?这在角色定义文件中明确说明

### 5.1 文件结构

```
项目层(随代码库版本控制)
├── .agents/
│   ├── active/                 # 活跃任务(当前正在进行的)
│   │   └── {task-name}/
│   │       ├── spec.md
│   │       ├── plan.md
│   │       └── status.yaml
│   │
│   ├── archive/                # 已完成任务(归档)
│   │   └── 2026-05-09-{task-name}/
│   │
│   ├── config.yaml              # 项目级配置
│   │                            #   - 激活哪些工作流和角色
│   │                            #   - 技术栈、代码规范
│   │                            #   - 检查点设置
│   │                            #   - 流程推荐规则
│   │
│   ├── workflows/               # 工作流定义(主线)
│   │   ├── feature.md           # 功能开发工作流(Standard/Full)
│   │   ├── quick.md             # 快速工作流(Bug 修复等)
│   │   ├── bugfix.md            # Bug 修复工作流(后续)

│   │   ├── analyst.md           # 需求分析师:澄清、调研
│   │   ├── architect.md         # 架构师:方案设计、任务拆分
│   │   ├── developer.md         # 开发工程师:编码、调试
│   │   └── reviewer.md          # 评审专家:代码评审、测试
│   │
│   ├── templates/               # 产出模板(可选)
│   │   ├── requirement-notes.md
│   │   ├── architecture-plan.md
│   │   └── review-report.md
│   │

个人层(本地,不进入版本控制)
├── ~/.agents/
│   ├── config.yaml              # 个人偏好(模型选择、沟通风格)
│   ├── preferences/             # 编码习惯、沟通风格偏好
│   └── evolution/               # 角色自演进数据(v0.3+)
```

### 5.2 配置职责

**项目层定义"团队共识":**
- 我们的技术栈是什么
- 代码规范是什么
- 有哪些可用的工作流
- 每个角色的职责边界

**个人层定义"个体偏好":**
- 我喜欢什么样的沟通风格
- 我习惯用什么模型
- 我的历史偏好和上下文

**核心价值:团队共识和个体偏好解耦。** 项目配置可迁移(换人接手自然继承),个人配置有主权(不随项目传播)。

### 5.3 config.yaml 示例

```yaml
# 项目级配置
project:
  name: "my-project"
  tech_stack: ["typescript", "react", "node"]
  code_style: "eslint-config-airbnb"

workflows:
  active:
    - feature
    # - bugfix    # 后续启用
    # - refactor  # 后续启用
  default: feature  # 默认工作流

roles:
  active:
    - analyst
    - architect
    - developer
    - reviewer

checkpoints:
  plan: required      # Plan 结束后必须人工确认
  execute: auto        # Execute 结束后可自动流转
  review: required     # Review 结束后必须人工确认
```

## 六、活跃/归档分离(借鉴 OpenSpec)

### 6.1 设计思路

OpenSpec 的核心文件管理洞察:**活跃任务和已完成任务用不同文件夹管理。** 活跃的变更放在 `changes/` 下,完成后归档到 `changes/archive/`。

这解决了一个实际问题:**随着项目推进,`.agents/` 下会堆积大量历史产出,新任务很难快速找到当前相关文件。**

### 6.2 目录结构

```
.agents/
├── active/                    # 活跃任务(当前正在进行的)
│   └── {task-name}/           # 每个任务一个文件夹
│       ├── spec.md            # 需求/设计 spec
│       ├── plan.md            # 实现计划
│       ├── status.yaml        # 任务状态
│       └── notes.md           # 对话中的关键决策记录
│
├── archive/                   # 已完成任务(归档)
│   └── 2026-05-09-{task-name}/
│       ├── spec.md
│       ├── plan.md
│       └── review.md
│
├── config.yaml                # 项目配置
├── workflows/                 # 工作流定义
├── roles/                     # 角色定义
├── templates/                 # 产出模板
└── log.jsonl                  # 可观测性日志
```

### 6.3 任务状态流转

```yaml
# active/{task-name}/status.yaml
status: plan  # draft → brainstorm → plan → execute → review → done
stage: plan
date_started: 2026-05-09
date_updated: 2026-05-09
context_budget_used: 0.35
```

当一个任务完成后:
1. 状态标记为 `done`
2. 整个文件夹从 `active/` 移到 `archive/`,加上日期前缀
3. 关键经验(如果有)沉淀到 `~/.agents/preferences/`(个人层)

### 6.4 好处

- **干净的活跃视图**--`active/` 只有当前在做的任务,一目了然
- **历史可追溯**--`archive/` 保留完整历史,带日期前缀方便查找
- **并行任务**--可以同时有多个活跃任务,互不干扰
- **新任务启动快**--看 `active/` 就知道现在在干什么

## 七、轻量/重量流程选择(借鉴 BMAD)

### 7.1 设计思路

不是所有需求都需要走完整的 Brainstorm → Plan → Execute → Review 流程。修一个 typo 和设计一个新功能,复杂度差了几个数量级。

BMAD 有 `bmad-quick-dev`(轻量开发流程)和完整方法论的区别。我们也需要这种分层。

### 7.2 三种流程模式

| 模式 | 适用场景 | 阶段 | 门控 |
|------|---------|------|------|
| **Quick** | Bug 修复、小调整、简单修改 | Execute + Review | 无 |
| **Standard** | 常规功能开发、中等工作量 | Brainstorm → Plan → Execute → Review | Plan 后确认 |
| **Full** | 复杂功能、架构变更、多模块涉及 | 完整流程 + 额外验证 | 每个阶段都确认 |

### 7.3 流程选择逻辑

用户可以显式指定流程,也可以让系统根据任务特征推荐:

```yaml
# config.yaml 中配置默认推荐规则
flow_recommendation:
  quick:  # 触发条件
    - 修改文件数 <= 3
    - 不涉及新接口
    - 不涉及架构变更
  standard:  # 触发条件
    - 修改文件数 4-10
    - 或涉及新接口
  full:  # 触发条件
    - 修改文件数 > 10
    - 或涉及架构变更
    - 或涉及多个模块
```

但**最终决策权在用户**--系统只是推荐,用户可以覆盖。

### 7.4 Quick 流程细节

Quick 流程跳过 Brainstorm 和 Plan,直接进入 Execute:

```
用户说:"修一下登录页面的 typo"
系统识别:这是一个 Quick 任务
→ 直接加载 developer.md
→ 在对话框中确认修改内容
→ 执行修改
→ 简单 Review
→ 完成,归档
```

**Quick 流程仍然要写 log.jsonl**,只是记录更简略。

### 7.5 Full 流程额外步骤

Full 流程在 Standard 基础上增加:

- Brainstorm 阶段:输出正式的 spec 文档(而不是只在对话中澄清)
- Plan 阶段:Goal-Backward 分析(目标 → truth → artifact → wiring)
- Execute 阶段:每个 task 有独立的验证标准
- Review 阶段:多维度评审(功能、安全、性能、可维护性)
- 所有阶段:自审核循环(placeholder scan、一致性检查、scope 检查)

## 八、上下文预算管理(借鉴 GSD)

### 8.1 设计思路

GSD 的核心洞察:**AI 编程质量随上下文使用量递减。**

| 上下文使用量 | 质量 | AI 状态 |
|-------------|------|--------|
| 0-30% | 最高 | 全面、细致 |
| 30-50% | 良好 | 自信、扎实 |
| 50-70% | 下降 | 开始走捷径 |
| 70%+ | 较差 | 敷衍了事 |

这意味着:**每个工作阶段应该控制在一个合理的上下文预算内。** 如果一个阶段的任务太多,应该拆分。

### 8.2 实现方式

**在 workflow 文件中为每个阶段设定建议的上下文预算:**

```markdown
# workflows/feature.md

## 阶段 2: Plan
- 角色:Architect
- 上下文预算:40%
- 建议:如果方案复杂度超过预算,拆分为多个 sub-task
```

**在任务状态文件中追踪上下文消耗:**

```yaml
# active/{task-name}/status.yaml
status: plan
context_budget_used: 0.35  # 估算当前上下文使用比例
context_note: "方案设计消耗约 35%,剩余空间足够进入 Execute"
```

### 8.3 上下文预算指导原则

- **单个阶段建议消耗 < 50%** 上下文
- **如果一个阶段的 Plan 产出超过 30%**,建议拆分为多个 Execute 轮次
- **Execute 阶段的每个 sub-task 建议 10-30%** 上下文
- **如果对话接近 70%**,建议保存产出、归档当前状态、开启新对话继续

### 8.4 实际应用

这不是一个精确的度量(LLM 不暴露实际 token 使用量),而是一个**定性的指导信号**:

- AI 角色在执行过程中自我评估上下文消耗
- 如果感觉当前任务太重,主动建议拆分
- 人在对话中观察到 AI 输出质量下降时,可以主动要求拆分

这比 GSD 的实现更轻量--我们不需要精确的 token 计数,而是用经验规则 + 人的判断来管理。

## 九、需求流转工作流

### 6.1 典型流转

以"给系统加一个微信登录功能"为例:

```
阶段 1: Brainstorm
  人:"我想给系统加一个微信登录功能"
  → 加载 analyst.md
  → Analyst 用澄清能力引导对话
  → 对话中:什么场景?OAuth 还是扫码?已有用户体系怎么对接?
  → 人确认需求清楚后退出

阶段 2: Plan
  → 加载 architect.md
  → Architect 用方案设计能力输出技术方案
  → 对话中:SDK 选型、Token 管理方案、接口设计
  → 人确认方案后,Architect 用任务拆分能力输出 Task List
  → 产出:架构方案.md(确认快照)
  → 人确认进入下一步

阶段 3: Execute
  → 加载 developer.md
  → Developer 按方案和 Task List 逐步实现
  → 对话中:实现细节、边界处理、遇到的问题
  → 产出:代码 + 变更记录

阶段 4: Review
  → 加载 reviewer.md
  → Reviewer 做代码评审和测试验证
  → 对话中:Review 意见、测试覆盖、改进建议
  → 产出:Review 意见.md
```

### 6.2 关键设计

- **人可以随时介入**:任何阶段发现问题,直接在对话框里提出
- **可以跳步**:简单 Bug 修复可以跳过 Brainstorm 和 Plan,直接 Execute + Review
- **可以回退**:在 Execute 阶段发现方案有问题,切回 Plan 重新设计
- **对话是连贯的**:整个过程中人在同一个对话框中,上下文连贯

### 6.3 检查点

某些节点天然是"检查点"--建议系统在这些节点提示人确认后再继续:

| 检查点 | 说明 | 默认行为 |
|--------|------|---------|
| Plan 完成 | 架构方案确认后再进入开发 | 暂停等待确认(可配置) |
| Execute 完成 | 开发完成后确认进入 Review | 可自动流转(可配置) |
| Review 完成 | 确认所有问题已处理 | 暂停等待确认(可配置) |

> 这些检查点不是强制的,可以在 `config.yaml` 中配置。

## 十、自演进机制(v0.2 简化版)

### 7.1 v0.2 范围:记录偏好

v0.2 的自演进仅限于**个人偏好的记录和应用**:

## 十一、可观测性

多 agent 协作最大的调试坑是**出了问题不知道问题出在哪个环节**。

### 8.1 最轻量日志

v0.2 就加一个 `.agents/log.jsonl`,每次角色处理追加一条记录:

```jsonl
{"ts":"2026-05-09T14:30:00+08:00","stage":"brainstorm","role":"analyst","output":"","status":"done","duration_s":180}
{"ts":"2026-05-09T14:35:00+08:00","stage":"plan","role":"architect","output":"架构方案.md","status":"done","duration_s":300}
{"ts":"2026-05-09T15:30:00+08:00","stage":"execute","role":"developer","output":"变更记录.md","status":"done","duration_s":900}
{"ts":"2026-05-09T15:50:00+08:00","stage":"review","role":"reviewer","output":"review-意见.md","status":"issues_found","duration_s":180}
```

字段定义:
- `ts`:时间戳
- `stage`:当前工作流阶段
- `role`:当前出场的角色
- `output`:产出的文件(如有)
- `status`:`done` / `issues_found` / `blocked` / `skipped`
- `duration_s`:耗时(秒)

**不需要可视化 dashboard,但至少能 `tail -f` 看流转状态。**

## 十二、模型路由

不同角色可以运行在不同的 AI 模型上:

| 角色类型 | 模型倾向 | 原因 |
|---------|---------|------|
| Architect | 强推理模型(Opus 级) | 需要深度思考和多维度权衡 |
| Developer | 快速编码模型(Sonnet/GLM 级) | 吞吐量优先,上下文干净 |
| Reviewer | 专项模型 | 覆盖率和准确性优先 |
| Analyst | 中等推理模型 | 需要理解力和追问能力 |

**实现依赖:** 需要底层工具支持模型指定能力。

## 十三、关键设计决策及理由

| 决策 | 选择 | 理由 |
|------|------|------|
| 架构模式 | 工作流驱动 + 角色能力双层 | 用户按阶段思考（低认知成本），系统按专业角色执行（高专业质量） |
| Agent 定位 | 辅助角色，非执行主体 | 当前 AI 可靠性不足以承担执行主体责任 |
| 核心场景 | 人在 IDE 中的对话式工作流 | 对话本身就是反馈回路 |
| 信息流转 | 对话上下文为主 + 文件快照为辅 | 对话是第一信息源，文件是持久化确认 |
| 文件角色 | 人的确认快照，不是 agent 间的 API | 文件是给人审核的 |
| 角色与工作流关系 | 解耦：角色是能力容器，工作流是调度主线 | 同一角色可服务多个工作流，同一工作流可灵活替换角色 |
| 入口设计 | 工作流阶段为入口 | 用户感知“我在 Plan”，不需要理解角色体系 |
| v0.2 角色数量 | 4 个核心角色 | 先验证协作机制，再扩展 |
| 流程分层 | Quick / Standard / Full 三种模式 | 不同复杂度的任务用不同重量的流程 |
| 任务生命周期 | 活跃/归档分离 | active/ 保持干净，archive/ 保留历史 |
| 上下文管理 | 每阶段 <50% 预算指导 | 防止上下文腐烂导致质量下降 |
| 自演进范围 | v0.2 只做偏好记录 | 先证明多角色协作有效，再做智能进化 |
| 可观测性 | v0.1 就有 log.jsonl | 成本极低，调试价值极高 |
| 配置管理 | 项目层 vs 个人层分离 | 团队共识和个体偏好解耦 |
| 是否强制流程 | 否，可跳步可回退 | 灵活性优先 |

| 维度 | BMAD | Qoder 专家团 | Superpowers | 本方案 |

## 十五、可行性评估

### 12.1 技术可行性

| 能力 | 当前状态 | 说明 |
|------|---------|------|
| .md 文件模拟多身份 | ✅ 可行 | 核心机制,零技术障碍 |
| 对话式交互 | ✅ 可行 | IDE 内 AI 对话窗口天然支持 |
| 工作流阶段切换 | ✅ 可行 | 加载不同 .md 文件即可 |
| 文件持久化 | ✅ 可行 | 纯文件读写 |
| 配置分层 | ✅ 可行 | 项目目录 + 用户目录 |
| 可观测性(log.jsonl) | ✅ 可行 | 纯文本追加 |
| 偏好记录 | ✅ 可行 | 本地文件存储 |

### 12.2 主要风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 角色定义不够精确导致输出质量不稳定 | 高 | 角色定义中明确能力边界、质量标准和不要做的事 |
| 对话上下文过长导致后期"遗忘"早期信息 | 中 | 文件快照承担关键信息持久化;阶段切换时用文件补充 |
| 工作流阶段划分与实际工作节奏不匹配 | 中 | 可跳步、可回退、可配置;先跑一个真实需求验证 |
| 团队接受度 | 中 | 阶段切换直觉低门槛;先在非关键项目试点 |

## 十六、落地路径

### Phase 1:骨架搭建(Week 1)

定义角色、工作流、配置结构。

产出:
```
.agents/
├── config.yaml
├── workflows/
│   └── feature.md
├── roles/
│   ├── analyst.md
│   ├── architect.md
│   ├── developer.md
│   └── reviewer.md
├── templates/
│   ├── requirement-notes.md
│   ├── architecture-plan.md
│   └── review-report.md
└── log.jsonl
```

### Phase 2:跑通一个真实需求(Week 2)

用一个真实的小需求(比如给现有项目加一个 API endpoint),跑一遍完整流转。

验证重点:
- 阶段切换体验是否自然
- 角色能力在每个阶段是否够用
- 对话上下文是否足够支撑下游阶段工作
- 文件快照是否有价值

### Phase 3:调优(Week 3)

基于跑通的结果:
- 调整角色定义(能力边界、质量标准)
- 调整工作流阶段(是否需要加/减/合并阶段)
- 调整映射关系(某些阶段是否需要换角色)
- 收集痛点

### Phase 4:决策点(Week 4)

基于三周验证结果决定:
- 是否扩展角色集
- 是否增加新工作流(bugfix、refactor)
- 是否启动偏好记录机制
- 是否需要更结构化的产出模板

## 附录 A:术语表

| 术语 | 定义 |
|------|------|
| **活跃/归档分离** | 活跃任务存放在 active/,完成后归档到 archive/,保持工作区清洁 |
| **流程模式** | Quick(轻量)、Standard(标准)、Full(完整),适配不同复杂度 |
| **上下文预算** | 每个工作阶段建议的上下文消耗上限,防止质量下降 |
| **工作流(Workflow)** | 一个完整的工作流程,由多个阶段组成,定义"做什么、按什么顺序" |
| **阶段(Stage)** | 工作流中的一个步骤,如 Brainstorm、Plan、Execute、Review |
| **角色(Role)** | 专业能力容器,定义"谁能做什么、做到什么标准" |
| **能力(Capability)** | 角色提供的具体技能,如"方案设计"、"任务拆分" |
| **对话式工作流** | 人在 IDE 对话框中、以连贯上下文驱动 AI 协作的工作模式 |
| **文件快照** | 阶段性产出的持久化记录,用于确认和版本追踪 |
| **检查点** | 建议人确认后再继续的节点,非强制 |
| **项目层配置** | 随代码库版本控制的团队共识配置 |
| **个人层配置** | 保留在用户本地的个体偏好配置 |
| **模型路由** | 不同角色使用不同 AI 模型的能力 |
| **需求流转** | 一个需求从提出到交付的完整生命周期,每个环节由对应的角色负责 |
